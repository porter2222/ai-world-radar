Source Strategy

AI World Radar 信息源策略

P1 冷启动阶段先用稳定可回源的信息源跑通 AI 情报生产闭环，再在 P1.5 接入更复杂的社交热度源。

版本：v0.1 更新时间：2026-06-03 阶段：P1 冷启动

1. 文档目的

本文档用于定义 AI World Radar P1 冷启动阶段的信息源选择、信息源分层、候选事件入池规则、热度判断、事件合并、排序、待核验线索处理、丢弃规则与更新节奏。

本文档不进入具体爬虫实现、API 调用方式、数据库字段、Agent 提示词和技术架构设计。后续 Agent 系统设计与技术架构设计将基于本文档继续展开。

2. 核心原则

P1 冷启动阶段不追求全网覆盖，而追求稳定、可回源、可验证、可跑通闭环。

先用简单、稳定、可回源的信息源跑通 AI 情报生产闭环；X、Reddit、YouTube 等复杂高热度平台作为重要热度源保留到 P1.5 增强。

P1 的产品口径是热度优先、轻核验、重表达边界。系统不要求每条高热度事件在自动发布前都找到官方事实源；只要热度信号成立，且正文明确表达为“社区正在讨论”“外网热议”“传闻正在发酵”或“尚未看到官方确认”，就可以作为热议型事件进入发布链路。

P1 的目标不是一开始就抓到所有外网热点，而是先完成：信息采集 -> 候选事件 -> 去重聚合 -> Agent 生成 -> 自动发布 -> 管理页校正。

3. 信息源分层

事实源

用于判断事情是否真实发生，通常来自官方、一手页面或可验证原始资产。

热度源

用于判断外网是否正在关注这件事，反映舆论、社区关注度和趋势变化；它可以支撑“讨论正在发生”，但不能支撑“讨论中的说法已经被证实”。

线索源

用于帮助发现可能被漏掉的事件。线索源不能直接作为正文事实依据。

4. P1 冷启动信息源

官方与一手事实源

  1. OpenAI News：模型发布、ChatGPT、API、Codex、安全、政策和合作事件。
  2. Anthropic News：Claude、Claude Code、企业合作、安全政策和研究报告。
  3. Google DeepMind Blog：Gemini、Veo、Imagen、Alpha 系列、科学 AI 和研究突破。
  4. NVIDIA RSS：AI 基础设施、GPU、CUDA、机器人、产业合作和算力生态。
  5. GitHub Changelog：GitHub Copilot、开发者工具、平台能力和供应链安全更新。

社区与开源热度源

  1. Hugging Face Papers / Models：论文热度、新模型、新 demo 和模型社区关注。
  2. Hacker News API：工程师圈正在讨论的 AI 基建、开源、开发者工具和争议。
  3. GitHub Trending：AI 开源项目爆发。它是弱信号源，不直接作为权威事实源。

线索源

  1. TLDR AI：英文技术圈和工程圈高信噪比线索。
  2. AIBase：中文 AI 产品、国内公司、工具和中文圈动态。

5. P1.5 增强信息源

以下来源具有价值，但 P1 冷启动阶段不作为硬依赖：X、Reddit、YouTube、arXiv 全量、Metacurate、The 500 Feed、smol.ai AINews、NewClaw、Ben’s Bites、大规模竞品线索源、Microsoft / Meta AI 官方源。

这些来源进入 P1.5 的原因不是价值不足，而是接入成本、噪声、权限、合规、去重或维护成本更高。

6. 社交热度源定位

X、Reddit 和 YouTube 是外网 AI 圈热度判断的重要平台。它们非常适合反映 AI 圈正在讨论什么、哪些事件正在发酵、哪些观点形成争议、哪些工具或模型正在被传播。

P1 阶段应在产品设计上承认它们是热度层核心来源，并为后续接入预留能力；工程实现上先用简单源跑通闭环。

7. 候选事件定义

候选事件是最近一段时间内，AI 圈发生或正在形成讨论的、具备明确主体、明确变化、可追溯来源，并且对中文 AI 学习型用户有认知价值的信息单元。

候选事件必须能说成一句：谁或什么东西，发生了什么变化。

候选事件应满足：有明确主体、有明确变化或讨论触发点、有时间性、有来源、对目标用户有价值。

8. 四档分流

  • 事实型候选事件：已经有明确事实发生，可以进入事件生成链路。
  • 热议型候选事件：外网讨论热度很高、能反映趋势、争议或舆论变化的信息。
  • 待核验线索：暂时不适合公开发布，但值得继续观察的信息。
  • 丢弃内容：不具备事件价值、来源价值或用户认知价值的信息。
热议型事件必须区分“事实已经发生”和“讨论正在发生”。已确认的是讨论，未确认的是讨论中的具体说法。P1 不把官方事实源作为热议型事件自动发布的前置条件，审稿节点要判断的是表达是否和来源强度匹配。

9. 热度判断

热度判断不应只依赖绝对数字，因为不同平台热度尺度不同。

一个强信号，或两个中等信号，可以进入热议型候选事件。

系统应综合平台位置、互动增速、跨源重复、主体影响力和讨论质量。纯标题党、搬运号传播、低质量重复转发不应被视为有效热度。热度成立时，可以发布热度和讨论本身，但不能把社区说法改写成确定事实。

10. 事件合并规则

事件合并以“主体 + 触发点 + 时间窗口”为核心。

  • 同一主体的同一变化，应合并为同一个事件。
  • 同一主体出现不同触发点时，应拆分为不同事件。
  • 社区讨论依附于事实事件时，应合并到事实事件下。
  • 讨论本身成为焦点时，可以单独形成热议型事件。
  • 同一事件的后续发展优先更新原事件，只有出现实质新变化时才拆分为新事件。

11. 排序策略

候选事件排序采用综合优先级，不采用单一时间倒序或单一热度排序。

  • 热度。
  • 来源质量。
  • 新鲜度。
  • 影响范围。
  • 用户相关性。
  • 可解释性。
热度优先，但来源和影响兜底。

12. 待核验线索处理

待核验线索是事件候选池的缓冲区。它不直接展示到前台，但系统会继续观察。

  • 出现更多来源后可以升级。
  • 热度升高后可以升级。
  • 人工确认后可以升级。
  • 长期无进展则丢弃。
  • 管理页可人工升级、丢弃、补源或暂缓观察。

13. 丢弃规则

普通教程、工具合集、明显营销软文、旧闻重复发布、无来源搬运、主体不清内容、AI 相关性弱的泛科技新闻、低质量争议、纯投资或股价短讯、重复内容应直接丢弃。

例外：如果教程、工具合集或观点文章本身形成高热讨论，可以作为热议型事件入池。此时事件不是教程内容本身，而是该内容引发了外网热议。

14. 更新节奏

  • 信息采集：默认每 6 小时采集一次信息源。
  • 首页雷达：每轮采集后重新计算最近 24 小时事件池，更新 Top 12。
  • 今日简报：每天生成主版本，并允许补充更新。
  • 事件详情：只对入选事件生成。
  • 管理页：支持重新采集、重新生成、隐藏和恢复。

15. 当前阶段结论

P1 信息源策略的核心不是“源越多越好”，而是用稳定可回源的信息源跑通 AI 情报生产闭环，用热度源判断事件价值，用线索源补充盲区，并通过分流、合并、排序和人工兜底控制噪声。

当前策略能够支撑下一阶段进入 Agent 系统设计。
