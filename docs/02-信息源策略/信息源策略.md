# AI World Radar 信息源策略

版本：v0.1  
更新时间：2026-06-03  
阶段：P1 冷启动信息源策略确认版

## 1. 文档目的

本文档用于定义 AI World Radar P1 冷启动阶段的信息源选择、信息源分层、候选事件入池规则、热度判断、事件合并、排序、待核验线索处理、丢弃规则与更新节奏。

本文档不进入具体爬虫实现、API 调用方式、数据库字段、Agent 提示词和技术架构设计。后续 Agent 系统设计与技术架构设计将基于本文档继续展开。

## 2. 核心原则

P1 冷启动阶段不追求全网覆盖，而追求稳定、可回源、可验证、可跑通闭环。

核心原则是：

> 先用简单、稳定、可回源的信息源跑通 AI 情报生产闭环；X、Reddit、YouTube 等复杂高热度平台作为重要热度源保留到 P1.5 增强。

P1 的目标不是一开始就抓到所有外网热点，而是先完成：

> 信息采集 -> 候选事件 -> 去重聚合 -> Agent 生成 -> 自动发布 -> 管理页校正。

## 3. 信息源分层

AI World Radar 的信息源分为三层：事实源、热度源和线索源。

### 3.1 事实源

事实源用于判断“事情是否真实发生”。

事实源通常来自官方、一手页面或可验证原始资产，例如：

- 官方新闻页。
- 官方博客。
- 官方公告。
- GitHub 仓库。
- Hugging Face 页面。
- 论文页面。
- 产品发布页。

事实源优先用于支撑正式事件。

### 3.2 热度源

热度源用于判断“外网是否正在关注这件事”。

热度源不一定能证明事件事实，但能反映舆论、社区关注度和趋势变化。

典型热度源包括：

- Hacker News。
- GitHub Trending。
- Hugging Face trending。
- X。
- Reddit。
- YouTube。

P1 冷启动阶段优先接入简单热度源；复杂热度源进入 P1.5。

### 3.3 线索源

线索源用于帮助发现可能被漏掉的事件。

线索源可以来自竞品网站、AI 日报、聚合站、Newsletter 或中文资讯站。

线索源只能提示“可能有事”，不能直接作为正文事实依据。线索源命中的内容必须尽量回溯到一手来源。

## 4. P1 冷启动信息源清单

P1 冷启动阶段优先接入 10 个来源。

### 4.1 官方与一手事实源

1. OpenAI News  
   用于发现 OpenAI 模型发布、ChatGPT、API、Codex、安全、政策和合作事件。

2. Anthropic News  
   用于发现 Claude、Claude Code、企业合作、安全政策和研究报告。

3. Google DeepMind Blog  
   用于发现 Gemini、Veo、Imagen、Alpha 系列、科学 AI 和研究突破。

4. NVIDIA RSS  
   用于发现 AI 基础设施、GPU、CUDA、机器人、产业合作和算力生态事件。

5. GitHub Changelog  
   用于发现 GitHub Copilot、开发者工具、平台能力和供应链安全更新。

### 4.2 社区与开源热度源

6. Hugging Face Papers / Models  
   用于发现论文热度、新模型、新 demo 和模型社区关注。

7. Hacker News API  
   用于发现工程师圈正在讨论的 AI 基建、开源、开发者工具和争议。

8. GitHub Trending  
   用于发现 AI 开源项目爆发。它是弱信号源，不直接作为权威事实源。

### 4.3 线索源

9. TLDR AI  
   用于补充英文技术圈和工程圈高信噪比线索。

10. AIBase  
    用于补充中文 AI 产品、国内公司、工具和中文圈动态。

## 5. P1.5 增强信息源

以下来源具有价值，但 P1 冷启动阶段不作为硬依赖：

- X。
- Reddit。
- YouTube。
- arXiv 全量。
- Metacurate / The Daily AI Brief。
- The 500 Feed。
- smol.ai AINews。
- NewClaw。
- Ben’s Bites。
- 大规模竞品线索源。
- Microsoft / Meta AI 官方源。

这些来源进入 P1.5 的原因不是价值不足，而是接入成本、噪声、权限、合规、去重或维护成本更高。

## 6. X、Reddit、YouTube 的定位

X、Reddit 和 YouTube 是外网 AI 圈热度判断的重要平台。它们非常适合反映：

- AI 圈正在讨论什么。
- 哪些事件正在发酵。
- 哪些观点形成争议。
- 哪些工具、模型、论文或产品正在被传播。

但它们不一定适合作为 P1 冷启动的自动硬依赖。

P1 阶段应在产品设计上承认它们是热度层核心来源，并为后续接入预留能力；工程实现上先用简单源跑通闭环。

P1.5 阶段再逐步加入：

- X 高价值账号列表与关键词追踪。
- Reddit 指定社区监听。
- YouTube 已知频道 RSS 或 API。
- 更细的社交热度评分。

## 7. 候选事件定义

候选事件是最近一段时间内，AI 圈发生或正在形成讨论的、具备明确主体、明确变化、可追溯来源，并且对中文 AI 学习型用户有认知价值的信息单元。

候选事件必须能说成一句：

> 谁或什么东西，发生了什么变化。

候选事件应满足：

- 有明确主体。
- 有明确变化或讨论触发点。
- 有时间性。
- 有来源。
- 对目标用户有价值。

候选事件不等同于新闻文章。系统应从多源信息中提取事件，而不是把每篇文章都当作事件。

## 8. 四档分流

采集内容进入系统后，应先分为四档。

### 8.1 事实型候选事件

事实型候选事件是已经有明确事实发生的信息。

例如：

- 官方发布新模型。
- 产品上线新功能。
- 开源项目发布。
- 公司发布公告。
- 论文或报告发布。

事实型候选事件可以进入事件生成链路。

### 8.2 热议型候选事件

热议型候选事件是外网讨论热度很高、能反映趋势、争议或舆论变化的信息。

热议型事件不一定已经有官方确认，但“外网正在讨论”本身可以成为事件。

这类事件必须区分：

- 已确认的是讨论正在发生。
- 未确认的是讨论中的具体说法。

表达方式应谨慎。例如：

> 外网热议某公司可能收购某 AI 公司，目前尚未看到官方确认。

而不是：

> 某公司即将收购某 AI 公司。

### 8.3 待核验线索

待核验线索是暂时不适合公开发布，但值得继续观察的信息。

它不直接展示到前台，不生成首页事件卡，不进入今日简报，也不生成详情页。

待核验线索后续可以升级、丢弃或由维护者人工处理。

### 8.4 丢弃内容

丢弃内容是不具备事件价值、来源价值或用户认知价值的信息。

丢弃内容不进入 Agent 生成链路。

## 9. 热度判断

热度判断不应只依赖绝对数字，因为不同平台热度尺度不同。

P1 阶段采用原则判断：

> 一个强信号，或两个中等信号，可以进入热议型候选事件。

### 9.1 强信号

强信号包括：

- Hacker News 出现高位讨论。
- GitHub Trending 排名靠前。
- Hugging Face Papers / Models 排名靠前。
- 已知 AI 频道集中讨论同一事件。
- 指定社区出现高互动讨论。
- 重要主体或关键人物引发集中讨论。

### 9.2 中等信号

中等信号包括：

- 多个线索源重复提及同一事件。
- 同一项目同时在 GitHub、HN、HF 或 AI 日报中出现。
- 某内容互动增长明显，但暂未形成强平台信号。
- 竞品日报与社区讨论同时出现。

### 9.3 热度判断维度

系统应综合：

- 平台位置。
- 互动增速。
- 跨源重复。
- 主体影响力。
- 讨论质量。

纯标题党、搬运号传播、低质量重复转发不应被视为有效热度。

## 10. 事件合并规则

事件合并以“主体 + 触发点 + 时间窗口”为核心。

### 10.1 同一主体 + 同一触发点

同一主体的同一变化，应合并为同一个事件。

例如：

- OpenAI 官方发布新模型。
- Hacker News 讨论该模型。
- TLDR AI 报道该模型。
- YouTube 视频讲解该模型。

这些应合并为同一事件。

### 10.2 同一主体 + 不同触发点

同一主体出现不同触发点时，应拆分为不同事件。

例如：

- OpenAI 发布新模型。
- OpenAI 调整 API 价格。
- OpenAI 出现安全争议。

这些不应强行合并。

### 10.3 社区讨论依附事实事件

当社区讨论依附于事实事件时，应合并到事实事件下，并作为讨论焦点、后续关注或来源补充。

### 10.4 社区讨论本身成为事件

当讨论本身成为焦点时，可以单独形成热议型事件。

### 10.5 后续发展

同一事件的后续发展优先更新原事件，不急着拆新卡。

只有出现实质新变化时，才拆分为新事件。

## 11. 排序策略

候选事件排序采用综合优先级，不采用单一时间倒序或单一热度排序。

排序应综合：

- 热度。
- 来源质量。
- 新鲜度。
- 影响范围。
- 用户相关性。
- 可解释性。

P1 冷启动排序原则是：

> 热度优先，但来源和影响兜底。

具体含义：

- 很热的事件优先展示。
- 官方重大事件即使暂时不热，也要展示。
- 热但来源差的事件可以展示，但要作为热议型或待确认事件谨慎表达。
- 低热、低影响、低相关内容不展示。

## 12. 待核验线索处理

待核验线索是事件候选池的缓冲区。

适合进入待核验线索的内容包括：

- 只有一个竞品日报提到，但没找到一手来源。
- 只有单个平台在讨论，热度还不够高。
- 主体明确，但变化不清楚。
- 传闻有扩散迹象，但缺少可靠来源。
- 事件可能重要，但当前信息太少。
- 疑似旧闻翻新，需要确认时间。

待核验线索处理规则：

- 不直接展示到前台。
- 继续自动观察。
- 允许补充来源后升级。
- 热度升高后可以升级。
- 人工确认后可以升级。
- 长期无进展则丢弃。
- 管理页可人工升级、丢弃、补源或暂缓观察。

## 13. 丢弃规则

以下内容应直接丢弃：

- 普通教程。
- 工具合集。
- 明显营销软文。
- 旧闻重复发布。
- 无来源搬运。
- 主体不清内容。
- AI 相关性弱的泛科技新闻。
- 低质量争议。
- 纯投资或股价短讯。
- 已被合并进其他事件的重复内容。

例外规则：

> 如果教程、工具合集或观点文章本身形成高热讨论，可以作为热议型事件入池。此时事件不是教程内容本身，而是该内容引发了外网热议。

## 14. 更新节奏

P1 不做实时分钟级热榜，而采用定时采集与滚动更新。

### 14.1 信息采集

默认每 6 小时采集一次信息源。

### 14.2 首页雷达

每轮采集完成后，重新计算最近 24 小时事件池，更新首页 Top 12。

### 14.3 今日简报

今日简报每天生成主版本，并允许根据当天新事件进行补充更新。

### 14.4 事件详情

事件详情只对入选事件生成，不对所有原始采集内容生成。

### 14.5 管理页手动触发

维护者可以手动触发：

- 重新采集。
- 重新生成某条事件。
- 重新生成今日简报。
- 隐藏事件。
- 恢复事件。

## 15. 当前阶段结论

P1 信息源策略的核心不是“源越多越好”，而是：

> 用稳定可回源的信息源跑通 AI 情报生产闭环，用热度源判断事件价值，用线索源补充盲区，并通过分流、合并、排序和人工兜底控制噪声。

当前策略能够支撑下一阶段进入 Agent 系统设计。
