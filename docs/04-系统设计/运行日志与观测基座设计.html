运行日志与观测基座设计

面向终端观察、定时任务追溯、Agent 运行监控和未来后台观测的统一日志方案。

1. 背景与目标

当前主分支已经具备真实的日常生产链路：run_daily_pipeline.py 启动，DailyPipelineService 编排采集、新信号读取、候选构造、LLM 编辑筛选和事件生产，collect_source_signals.py 写入 source_signals，run_event_pipeline.py 与 LangGraph 工作流推进到 PublishedEvent。

这个阶段的主要问题是运行过程像黑盒。用户在终端盯着时，不知道卡在采集、网络、数据库、selector、writer、reviewer 还是发布；定时任务后台运行时，也缺少一份可以回放的完整运行轨迹。

中文终端日志 结构化 JSONL Agent 一等公民 30 秒心跳 敏感信息脱敏

2. 第一版边界

第一版要做

  • 中文终端日志、文本文件日志和 JSONL 结构化日志。
  • 日常 pipeline、source 采集、Agent、LLM、工具调用和发布阶段日志。
  • 失败、重试、fallback、token usage、耗时和心跳记录。
  • 敏感信息脱敏，日志失败不拖垮主流程。

第一版不做

  • 后台管理页面、Redis、消息队列和 OpenTelemetry 全量接入。
  • 日志检索服务和完整 prompt / response 默认归档。
  • source 网络超时治理、LLM provider 限流和退避策略。

3. 设计原则

中文优先

[10:42:03] [采集开始] source=hn_algolia limit=5
[10:42:07] [采集成功] source=hn_algolia 写入=3 跳过=12 耗时=4.2s

结构化优先

终端日志只是展示层。底层必须是统一的结构化事件对象，同一条事件可以同时输出到终端、文本文件、JSONL 文件，未来也可以输出到数据库、Redis 或消息队列。

默认详细，但不泄露敏感信息

默认打印阶段、数量、耗时、错误类型、错误摘要、模型名、重试次数、关键 ID 和 token usage；默认不打印 API key、数据库密码、Cookie、完整 HTML 正文、完整 LLM prompt、完整 LLM 原始响应或图片 base64。

可定位卡点

每个长步骤都要至少有 started、heartbeat、succeeded、failed。如果某一步超过 30 秒未结束，持续打印心跳。

4. 总体架构

业务代码
  -> RunLogger.emit(event)
      -> ConsoleSink      中文终端实时日志
      -> TextFileSink     runtime/daily-pipeline-latest.log
      -> JsonlFileSink    runtime/daily-pipeline-latest.jsonl
      -> DbSink           后续扩展
      -> RedisSink        后续扩展
      -> QueueSink        后续扩展

RunLogger 是业务链路唯一直接调用的日志入口，负责生成 run_id、维护追踪字段、补齐时间戳、分发事件、提供阶段上下文和统一脱敏。

Sink作用第一版是否实现
ConsoleSink终端中文实时输出是
TextFileSink人类可读 .log 文件是
JsonlFileSink结构化 .jsonl 文件是
DbSink写入运行记录表，供后台管理页读取预留
RedisSink写入实时运行状态，供前端订阅预留
QueueSink推送运行事件到消息队列预留

第一版优先接入 run_daily_pipeline.py、DailyPipelineService、collect_source_signals.py、run_event_pipeline.py、LLM Agent 和 llm_client.py。

5. 日志事件协议

字段说明
timestampISO 8601 时间，带时区。
leveldebug/info/warning/error。
run_id / trace_id一次 daily pipeline 运行和追踪 ID。
span_id / parent_span_id当前阶段和父阶段 ID。
componentdaily_pipeline/collector/selector/workflow/agent/llm/tool/db/scheduler。
stage / event / status阶段名、事件类型和业务状态。
message_zh中文可读消息。
duration_ms阶段耗时。
source_key / candidate_group_id来源和候选事件组标识。
pipeline_run_id / agent_run_id数据库运行记录或本轮临时执行 ID。
agent_name / tool_nameAgent 和工具名。
provider / modelLLM provider 和 model。
retry_count / token_usage重试次数和 token 统计。
counts写入、跳过、发布等数量统计。
error_type / error_message异常类型和脱敏后的错误摘要。
metadata少量可 JSON 序列化扩展信息。
 
{"timestamp":"2026-06-26T10:42:03+08:00","level":"info","run_id":"daily_20260626_104201","component":"collector","stage":"collect_source","event":"started","message_zh":"开始采集来源 hn_algolia","source_key":"hn_algolia","counts":{"limit":5}}

6. 输出文件

runtime/
  daily-pipeline-latest.log
  daily-pipeline-latest.jsonl
  daily-pipeline-20260626-104201.log
  daily-pipeline-20260626-104201.jsonl

每次运行创建一组带时间戳的日志文件，latest.* 覆盖为最近一次运行。所有文件默认 UTF-8，runtime 产物不提交 Git。

8. Agent 级监控

第一版至少覆盖 EditorialSelectorLLMAgent、ResearchWriterLLMAgent、ReviewPublisherLLMAgent。每个 Agent 必须记录启动、输入摘要、LLM 请求、结构校验、完成或失败。

[Agent开始] agent=ResearchWriterLLMAgent candidate=group_xxx model=gpt-5.5
[Agent输入摘要] agent=ResearchWriterLLMAgent signals=2 sources=hn_algolia,github_repo_trends chars=1800
[LLM请求开始] agent=ResearchWriterLLMAgent provider=openai model=gpt-5.5 retry=0
[LLM请求成功] agent=ResearchWriterLLMAgent prompt_tokens=12000 completion_tokens=1800 耗时=31.2s
[结构校验成功] agent=ResearchWriterLLMAgent schema=EventDossierDraft
[Agent完成] agent=ResearchWriterLLMAgent status=succeeded words=850 耗时=38.2s
[LLM请求失败] agent=ReviewPublisherLLMAgent error=APITimeoutError retry=1 will_retry=true
[结构校验失败] agent=ReviewPublisherLLMAgent error=JSONDecodeError retry=2
[Agent失败] agent=ReviewPublisherLLMAgent status=failed action=manual_review
[工具调用开始] agent=ResearchWriterLLMAgent tool=fetch_source_snapshot source=github_repo_trends
[工具调用成功] agent=ResearchWriterLLMAgent tool=fetch_source_snapshot bytes=18234 耗时=1.4s

主日志仍是统一时间线：daily-pipeline-<run_id>.log/jsonl。后续如有需要，再从 JSONL 派生 Agent 独立视图。

9. 心跳机制

第一版心跳间隔为 30 秒。心跳必须包含当前阶段、当前 source / agent / candidate 和已运行时长。

[仍在运行] stage=collect_source source=github_repo_trends elapsed=30s
[仍在运行] stage=collect_source source=github_repo_trends elapsed=60s

10. 错误与脱敏

  • URL 中的密码段替换为 ***。
  • API key、token、authorization、cookie 字段统一替换为 [REDACTED]。
  • 错误信息中出现 .env 中的敏感值时必须替换。
  • 数据库连接串只允许输出 driver、host、port、database，不允许输出用户名和密码。
  • Prompt、response、HTML 正文默认只输出长度、hash 或摘要，不输出全文。

12. 最小实施顺序

  1. 新增 worker/observability/run_logger.py，实现事件模型、脱敏和三个第一版 Sink。
  2. 为 RunLogger 写单元测试，覆盖中文终端格式、JSONL 合法性、文件写入和脱敏。
  3. run_daily_pipeline.py 创建 logger，并传给 DailyPipelineService。
  4. DailyPipelineService 在启动、采集、新信号读取、候选构造、selector、per-candidate、最终总结阶段打日志。
  5. collect_source_signals.py 和 source collector 统一返回或记录采集统计。
  6. run_event_pipeline.py 和 LangGraph 节点接入 workflow / Agent 阶段日志。
  7. LLM Agent wrapper 记录 LLM 请求、响应摘要、token usage、结构校验、重试和失败。
  8. 长阶段加 30 秒 heartbeat。
  9. 更新测试记录，记录终端输出样例和 runtime/*.log/jsonl 样例。

13. 验收标准

  • 手动运行 python scripts/run_daily_pipeline.py 时，终端实时输出中文阶段日志。
  • runtime/daily-pipeline-latest.log 存在，并包含本轮完整阶段日志。
  • runtime/daily-pipeline-latest.jsonl 存在，每行是合法 JSON。
  • 每个 source、selector、selected candidate 和 Agent 都有开始、成功或失败日志。
  • Agent 日志包含 agent_name、agent_run_id、candidate_group_id、模型名、重试次数和 token usage。
  • 失败路径记录 error_type 和脱敏后的 error_message。
  • 长阶段每 30 秒出现 heartbeat。
  • 日志系统异常不影响主流程。
