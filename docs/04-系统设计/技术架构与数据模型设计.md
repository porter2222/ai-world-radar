# AI World Radar 技术架构与数据模型设计

版本：v0.1

更新时间：2026-06-04

阶段：P1 MVP 技术架构与数据模型确认版

## 1. 文档目的

本文档用于定义 AI World Radar P1 MVP 的技术架构、模块分工、运行链路和核心数据模型。

本文档基于产品宪章、PRD、信息源策略、Agent 系统设计、GitHub 采集策略调研、QFT 晨报项目参考价值调研以及本轮数据模型讨论形成。

如果本文档与早期 Agent 系统设计文档存在口径差异，以本文档为最新 P1 技术口径。

## 2. 当前核心拍板

P1 当前技术与数据模型口径如下：

- 技术栈采用 `Next.js + Python Worker + PostgreSQL`。
- P1 暂不引入 FastAPI。
- Next.js 负责前台页面、管理页和简单 API / Server Action。
- Python Worker 负责采集、Agent 流水线、内容生成、质量门禁和定时任务。
- PostgreSQL 负责保存所有数据和运行状态。
- P1 不单独建 `RawItem` 表。
- P1 不单独建 `CandidateEvent` 表。
- P1 不做真实性核验，不做可信状态标签，不做事实确认评分。
- 排序采用“热度主导 + 重要度兜底 + 中文用户价值修正”。
- 所有前台公开展示的事件都必须有详情页。
- 今日简报只从已发布事件中精选，不是全量事件合集。
- 管理员操作通过可视化后台完成，并写入操作审计。

## 3. P1 技术架构

P1 采用轻量产品化架构：

```text
Next.js Web App
  -> 前台首页
  -> 今日简报
  -> 事件详情页
  -> 管理页
  -> 简单 API / Server Action

Python Worker
  -> Source 采集
  -> EvidenceCard 生成
  -> EventCluster 聚合
  -> ContentArtifact 生成
  -> QualityGate 检查
  -> 发布状态写入

PostgreSQL
  -> 来源配置
  -> 证据卡
  -> 事件聚合
  -> 内容版本
  -> 发布事件
  -> 简报
  -> 运行日志
  -> 管理操作
```

P1 暂不引入：

- FastAPI。
- Redis 队列。
- 微服务拆分。
- 复杂权限系统。
- embedding 事件聚类。
- X、Reddit、YouTube、Facebook 等复杂高热平台采集。
- 真实性核验系统。

## 4. 模块分工

### 4.1 Next.js

Next.js 负责用户界面和轻量后端接口：

- 首页事件流。
- 今日简报页。
- 事件详情页。
- 管理页。
- 查询已发布事件。
- 查询简报。
- 隐藏、恢复、置顶、排序等管理操作。
- 写入 `admin_actions`。
- 触发重新生成任务的状态变更。

P1 不单独做 FastAPI，因此 Next.js 会承担轻量 Web 后端职责。

### 4.2 Python Worker

Python Worker 负责后台自动生产链路：

- 按计划读取 `sources`。
- 采集来源内容。
- 生成 `evidence_cards`。
- 聚合 `event_clusters`。
- 生成 `content_artifacts`。
- 执行 Quality Gate。
- 写入 `published_events` 和 `briefs`。
- 更新 `pipeline_runs`。

Python Worker 与 Next.js 不通过 HTTP 服务通信，P1 阶段主要通过 PostgreSQL 协作。

### 4.3 PostgreSQL

PostgreSQL 是系统唯一主数据存储，保存：

- 来源配置。
- Agent 生产结果。
- 发布状态。
- 简报数据。
- 运行审计。
- 后台操作记录。

## 5. P1 运行链路

P1 自动生产链路如下：

```text
Source
-> PipelineRun
-> EvidenceCard
-> EventCluster
-> EventClusterCard
-> ContentArtifact
-> QualityGateResult
-> PublishedEvent
-> Brief / BriefItem
```

管理链路如下：

```text
Admin Page
-> Next.js Action
-> Update PublishedEvent / ContentArtifact / Brief
-> Write AdminAction
```

重新生成链路如下：

```text
Admin Page 点击重新生成
-> Next.js 写入任务状态或操作记录
-> Python Worker 扫描待处理任务
-> 生成新的 ContentArtifact
-> Quality Gate
-> 替换 PublishedEvent 当前引用
```

## 6. 数据表总览

P1 核心表如下：

```text
sources
pipeline_runs
evidence_cards
event_clusters
event_cluster_cards
content_artifacts
quality_gate_results
published_events
briefs
brief_items
admin_actions
```

P1 暂不建立：

```text
raw_items
candidate_events
raw_snapshots
fastapi_jobs
```

其中：

- `sources` 管来源。
- `pipeline_runs` 管每次自动运行。
- `evidence_cards` 管来源信号和 AI 理解结果。
- `event_clusters` 管内部事件。
- `event_cluster_cards` 管事件与证据卡的关联。
- `content_artifacts` 管 AI 生成内容版本。
- `quality_gate_results` 管内容质检结果。
- `published_events` 管前台发布事件。
- `briefs` 管每日简报。
- `brief_items` 管简报条目。
- `admin_actions` 管后台操作审计。

## 7. sources

`sources` 记录系统长期采集哪些来源，以及这些来源在发现、热度和重要度上的权重。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 数据库主键 |
| `source_id` | 来源业务标识，例如 `openai_news`、`github_trending` |
| `name` | 来源名称 |
| `source_type` | 来源类型：`official` / `platform` / `newsletter` / `competitor` / `community` |
| `fetch_method` | 采集方式：`rss` / `api` / `html` / `github` / `hn_api` |
| `url` | 来源入口地址 |
| `enabled` | 是否启用 |
| `discovery_weight` | 发现权重 |
| `heat_signal_weight` | 热度信号权重 |
| `impact_signal_weight` | 重要度信号权重 |
| `language` | 主要语言 |
| `category_hint` | 默认分类倾向 |
| `fetch_interval` | 抓取频率 |
| `capabilities` | 来源能力说明，例如是否有发布时间、热度指标、作者、图片 |
| `last_fetched_at` | 上次抓取时间 |
| `last_status` | 最近状态：`success` / `failed` / `paused` / `degraded` |
| `failure_reason` | 最近失败原因 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

P1 不在 `sources` 中保存真实性、可信度或事实锚定权重。

## 18. 关键设计取舍

### 18.1 不建 RawItem

P1 将原始来源条目和 AI 理解结果合并到 `evidence_cards` 中。这样可以减少表数量，适合先落地。

### 18.2 不建 CandidateEvent

P1 将候选判断字段放入 `evidence_cards`，然后直接进入 `event_clusters`。如果后续需要候选池、人工候选审核或 rules/LLM compare，再拆出 `candidate_events`。

### 18.3 不做真实性系统

P1 不做事实核验、官方确认、多源真实性判断和可信状态标签。Quality Gate 只做生成质量和来源追溯检查。

### 18.4 不做 FastAPI

P1 使用 Next.js 承担轻量 Web API，Python Worker 负责后台生产。后续如果管理接口和任务控制复杂化，再引入 FastAPI。
