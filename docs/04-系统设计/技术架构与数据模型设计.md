# AI World Radar 技术架构与数据模型设计

版本：v1.0

更新时间：2026-06-12

阶段：终局技术蓝图与 P1 单机落地口径

适用范围：AI World Radar 事件档案生产与发布系统

## 1. 文档目的

本文档定义 AI World Radar 的新版技术架构、模块边界、运行链路、Python Worker 内部结构、Pydantic / SQLAlchemy 分层原则、核心数据模型和阶段化工程扩展路径。

本文档基于新版 PRD v2.0 与新版 Agent 系统设计 v1.0。若本文档与旧版技术架构、旧版后端 P1 实现计划、旧版 Agent stub 命名存在差异，以本文档和新版 Agent 系统设计为准。

本文档不直接给出代码实现，不展开 UI 视觉设计，也不替代后续实现计划。后续进入开发前，还需要基于本文档编写具体 implementation plan。

## 2. 核心结论

P1 技术架构采用：

```text
Next.js + Python Worker + PostgreSQL
```

P1 必须使用：

```text
Pydantic
SQLAlchemy
Alembic
LangGraph
LLMClient
pytest
```

P1 暂不引入：

```text
FastAPI
Redis
Celery / RQ / Dramatiq 队列
对象存储
向量数据库
复杂调度系统
复杂监控大盘
多用户权限系统
```

核心理由：

> P1 的第一目标是跑通“外部信号 -> Agent 生产事件档案 -> 审稿 -> 发布 -> 前台/后台可查看”的闭环。不要让第一版被三服务架构、队列、缓存和复杂基础设施拖慢。

同时，P1 必须从第一天保留以下扩展边界：

- `run_id`
- `agent_run_id`
- 状态字段
- 幂等键
- `source_hash`
- `content_hash`
- `dossier.version`
- `review_result`
- `manual_review` 状态
- 运行记录
- 后台操作记录

## 3. 总体架构

P1 总体结构：

```text
Next.js Web App
  -> 首页 / 事件雷达页
  -> 事件详情页
  -> 后台管理页
  -> 调用 FastAPI 产品接口

Python Worker
  -> 信息源采集
  -> 正文缓存
  -> LangGraph workflow
  -> 编辑部式 Agent
  -> Skill / Tool 调用
  -> Pydantic 结构化校验
  -> 审稿修订循环
  -> 发布服务写库
  -> 运行报告

FastAPI Product API
  -> 已发布事件列表 / 详情
  -> pipeline run / agent run 审计查询
  -> manual_review 队列查询
  -> 只读 HTTP JSON 接口

PostgreSQL
  -> 业务数据
  -> 事件状态
  -> Agent 运行记录
  -> 后台操作记录
```

Next.js 和 Python Worker 在 P1 阶段仍不直接通过 HTTP 通信。Python Worker 负责后台生产链路，FastAPI Product API 负责把 PostgreSQL 中的已发布内容和审计记录以只读 JSON 接口暴露给 Next.js。

2026-06-22 用户已确认：从 P1-6 起引入 FastAPI 薄适配层。FastAPI 当前只做产品查询接口，不触发采集、不触发 Agent、不发布、不重跑。

## 4. 模块职责

### 4.1 Next.js Web App

Next.js 负责用户可见产品界面。

P1 职责：

- 首页 / 事件雷达页。
- 事件详情页。
- 后台管理页。
- 调用 FastAPI 查询 `published_events`。
- 调用 FastAPI 查询事件来源。
- 调用 FastAPI 查询 `pipeline_runs` 和 `agent_runs`。
- 编辑事件内容。
- 隐藏、恢复、发布事件。
- 写入 `admin_actions`。
- 写入重跑请求状态。

P1 不负责：

- 信息源采集。
- Agent 调用。
- LangGraph workflow。
- LLM prompt 执行。
- 长任务执行。

### 4.2 Python Worker

Python Worker 负责后台自动生产链路。

P1 职责：

- 读取信息源配置。
- 调用信息源 adapter。
- 采集和标准化 source signals。
- 抓取或缓存正文。
- 执行 LangGraph workflow。
- 调用值班编辑 Agent、研究写作 Agent、审稿发布 Agent。
- 执行 Pydantic schema 校验。
- 执行最多 2 次修订循环。
- 调用发布服务写入 `published_events`。
- 写入 `pipeline_runs`、`agent_runs`、`review_results`。
- 生成本地运行报告。

P1 不负责：

- 作为常驻 HTTP 服务对外提供 API。
- 直接渲染页面。
- 用户会话和权限体系。

### 4.3 PostgreSQL

PostgreSQL 是系统事实源。

保存：

- 信息源配置。
- 外部信号。
- 候选事件。
- 事件档案草稿。
- 审稿结果。
- 已发布事件。
- pipeline 运行记录。
- Agent 运行记录。
- 后台操作记录。

原则：

- 前台只读 `published_events` 和必要关联数据。
- 草稿、审稿、失败、重跑状态进入后台。
- Agent 不直接改核心状态，所有发布状态由工程服务写入。

## 5. P1 运行链路

自动生产链路：

```text
Source Adapter
-> source_signals
-> LangGraph Workflow
   -> editorial_triage
   -> merge_and_rank_events
   -> build_research_package
   -> draft_event_dossier
   -> review_event_dossier
   -> revise_if_needed
-> Publish Service
-> published_events
-> pipeline_runs / agent_runs
```

前台读取链路：

```text
Browser
-> Next.js Page / Server Action
-> PostgreSQL
-> published_events
-> 首页 / 详情页
```

后台管理链路：

```text
Admin Page
-> Next.js Server Action
-> PostgreSQL
-> event_candidates / event_dossiers / published_events
-> admin_actions
```

重跑链路：

```text
Admin Page 点击重跑
-> Next.js 写入 admin_actions 或 regenerate_requested 状态
-> Python Worker 扫描待处理任务
-> 重新生成 event_dossier
-> review_result
-> publish / manual_review
```

P1 可以先通过命令手动运行：

```powershell
cd apps/worker
python scripts/run_event_pipeline.py --source hn --days 7 --limit 100
```

## 7. Pydantic 与 SQLAlchemy 分层

P1 必须使用 Pydantic。P1-6 起已确认引入 FastAPI，但 FastAPI 只能复用 Pydantic 响应 schema 和 service 层契约，不能把业务规则散落在 HTTP endpoint 中。

Pydantic 负责：

- Agent 输入 schema。
- Agent 输出 schema。
- Workflow state schema。
- Service DTO schema。
- 数据库写入前校验 schema。
- 测试 fixture schema。

SQLAlchemy 负责：

- ORM 模型。
- 数据库字段映射。
- 关系约束。
- 查询和写入。

Alembic 负责：

- schema migration。
- 本地数据库迁移。
- 后续跨机器开发时重建数据库。

分层原则：

```text
Pydantic schema 不直接等于数据库表。
SQLAlchemy model 不直接等于 Agent 输出。
Service 层负责把 Pydantic 输出转换成数据库写入。
```

这样 P1-6 引入 FastAPI 时，可以直接复用 Pydantic schema 暴露 HTTP API，而不需要重写 Agent 数据结构。

## 11. 幂等与去重

P1 必须做最小幂等设计。

幂等点：

- `sources.source_key` 唯一。
- `source_signals.source_hash` 防止重复采集。
- `event_candidates.candidate_key` 防止重复候选事件。
- `event_candidate_signals` 防止重复关联。
- `event_dossiers(candidate_id, version)` 防止版本冲突。
- `published_events.candidate_id` 可根据策略限制一个候选事件只有一个当前发布版本。

P1 原则：

> 重复运行不能重复发布同一个事件。

## 12. 正文缓存与对象存储边界

P1 不引入对象存储。

P1 使用本地文件缓存：

```text
runtime/source-pages/
runtime/artifacts/
runtime/pipeline-reports/
```

PostgreSQL 保存：

- `content_hash`
- `content_cache_path`
- `source_hash`
- 摘要
- 抓取状态

后续上线或数据变多时，可迁移到对象存储：

```text
AWS S3
阿里云 OSS
腾讯云 COS
Cloudflare R2
MinIO
```

迁移原则：

> 数据库存索引、状态和 hash；对象存储存 HTML、Markdown、图片、截图、PDF、长文本 artifact 等大文件。

## 13. FastAPI 边界

P1-6 起引入 FastAPI，但只作为产品接口层的薄适配服务。

引入原因：

- 用户已确认当前阶段直接使用 FastAPI。
- 前端需要稳定消费 `GET /events`、`GET /events/{slug}` 和后台审计只读接口。
- Python 侧已经拥有 SQLAlchemy model、Pydantic schema 和生产链路服务，继续在 Python 内沉淀查询契约可以减少 Next.js 与 Worker 的重复建模。
- 后续如果要把 Agent run 触发、状态查询、取消、重试等能力服务化，可以在同一 FastAPI 服务边界内扩展。

P1-6 FastAPI 必须遵守：

- 只读已有 PostgreSQL 数据。
- 不触发采集。
- 不触发 Agent / LangGraph workflow。
- 不提供发布、隐藏、驳回、重跑等写操作。
- 业务查询规则放在 `ProductQueryService`，endpoint 只做参数解析和响应返回。
- 管理端只读接口当前仅用于本地或内网开发；公开部署前必须补鉴权。

未来 FastAPI Agent Service 的扩展触发条件：

- 前端需要实时调用 Python Agent 服务。
- 后台需要立即触发 Agent run，而不是等待 Worker 扫描。
- 需要任务状态查询、取消任务、重试任务等标准 API。
- Python 侧需要对外提供独立服务。
- 项目从个人项目升级为正式部署型产品。

未来可演进为：

```text
Next.js
-> FastAPI Product API / Agent Service
-> Queue
-> Python Worker
-> PostgreSQL
```

## 14. LangGraph 与 Agent 集成

P1 LangGraph workflow 建议放在：

```text
apps/worker/worker/workflows/event_pipeline.py
```

节点：

```text
collect_signals
normalize_signals
editorial_triage
merge_and_rank_events
build_research_package
draft_event_dossier
review_event_dossier
revise_if_needed
publish_or_manual_review
record_run
```

Workflow State 使用 Pydantic 定义，至少包含：

- `run_id`
- `source_scope`
- `signal_ids`
- `candidate_ids`
- `dossier_ids`
- `current_node`
- `status`
- `retry_count`
- `revision_count`
- `last_error`
- `review_decision`

Agent 输出必须经过 Pydantic 校验。解析失败最多 repair 2 次；仍失败则标记 failed 或 manual_review。

## 15. 服务层设计

Service 层负责工程控制权。

建议服务：

| Service | 职责 |
| --- | --- |
| `SignalService` | 写入和去重 source_signals |
| `CandidateService` | 创建、更新、归并候选事件 |
| `DossierService` | 保存事件档案版本 |
| `ReviewService` | 保存审稿结果和修订状态 |
| `PublishService` | 将通过审稿的 dossier 发布为 published_event |
| `RunLogService` | 写入 pipeline_runs 和 agent_runs |
| `AdminActionService` | 记录后台操作 |

核心原则：

> Agent 只产出结构化建议，Service 层负责数据库写入和状态变更。
