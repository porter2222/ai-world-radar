# 运行日志与观测基座设计

更新时间：2026-06-26

## 1. 背景与目标

当前主分支已经具备一条真实的日常生产链路：`run_daily_pipeline.py` 负责启动，`DailyPipelineService` 负责编排采集、新信号读取、候选构造、LLM 编辑筛选和事件生产，`collect_source_signals.py` 与各类 collector 负责写入 `source_signals`，`run_event_pipeline.py` 与 LangGraph 工作流负责把候选事件推进到 `PublishedEvent`。

这个阶段的主要问题不是“功能完全不能跑”，而是“跑起来像黑盒”：用户在终端盯着时，不知道卡在采集、网络、数据库、selector、writer、reviewer 还是发布；定时任务后台运行时，也缺少一份可以回放的完整运行轨迹。

本设计的目标是建立一套强力但不过度复杂的运行日志与观测基座。第一版先解决三件事：

- 手动运行时，终端实时打印中文、详细、可定位卡点的进度日志。
- 定时运行时，同步写入 `runtime` 下的人类可读日志和结构化 JSONL 日志。
- 每个 Agent、LLM 调用、工具调用、重试、结构化校验和失败原因都能被单独追踪。

这不是简单往代码里散落 `print`。日志基座要从第一版开始具备统一事件协议、统一入口、多个输出 Sink 和后续扩展口径，让未来的后台管理页、Redis 实时状态、消息队列、Agent 运行监控和问答助手都能沿用同一套事件流。

## 2. 第一版边界

第一版要做：

- 中文终端日志。
- `.log` 人类可读文件日志。
- `.jsonl` 结构化事件日志。
- 日常 pipeline 阶段级日志。
- source 采集级日志。
- Agent 级日志。
- LLM 调用日志。
- 工具调用日志。
- 失败、重试、fallback、token usage 和耗时记录。
- 长步骤 30 秒心跳。
- 敏感信息脱敏。

第一版不做：

- 后台管理页面。
- Redis。
- 消息队列。
- OpenTelemetry 全量接入。
- 日志检索服务。
- 完整 prompt / response 默认归档。
- source 网络超时治理。
- LLM provider 限流、熔断和退避策略。

这些后续都可以接在同一套日志事件协议上演进，不需要第一版一起上。

## 3. 设计原则

### 3.1 中文优先

终端输出默认面向中文用户，阶段名和消息文本必须中文可读。字段名可以保留英文，方便开发者复制搜索。

```text
[10:42:03] [采集开始] source=hn_algolia limit=5
[10:42:07] [采集成功] source=hn_algolia 写入=3 跳过=12 耗时=4.2s
```

### 3.2 结构化优先

终端日志只是展示层。底层必须是统一的结构化事件对象，同一条事件可以同时输出到终端、文本文件、JSONL 文件，未来也可以输出到数据库、Redis 或消息队列。

### 3.3 默认详细，但不泄露敏感信息

默认打印阶段、数量、耗时、错误类型、错误摘要、模型名、重试次数、关键 ID 和 token usage。默认不打印：

- API key、数据库密码、Cookie、Authorization header。
- 完整 HTML 正文。
- 完整 LLM prompt。
- 完整 LLM 原始响应。
- 超长正文、图片 base64 或大对象。

如果后续确实需要排查完整 payload，应通过显式 debug artifact 实现，而且必须脱敏。

### 3.4 可定位卡点

每个长步骤都要至少有：

- `started`
- `heartbeat`
- `succeeded`
- `failed`

如果某一步超过 30 秒未结束，持续打印心跳，让用户知道进程仍在运行，并能看到最后停在哪个阶段、哪个 source、哪个 Agent 或哪个 candidate。

### 3.5 日志系统不能拖垮主流程

日志写入失败不能导致事件生产链路失败。文件写入失败时最多降级到终端输出；未来 DB / Redis / Queue Sink 失败也不能影响主流程。

## 4. 总体架构

第一版采用 `RunLogger + Sink` 架构：

```text
业务代码
  -> RunLogger.emit(event)
      -> ConsoleSink      中文终端实时日志
      -> TextFileSink     runtime/daily-pipeline-latest.log
      -> JsonlFileSink    runtime/daily-pipeline-latest.jsonl
      -> DbSink           后续扩展
      -> RedisSink        后续扩展
      -> QueueSink        后续扩展
```

### 4.1 RunLogger

`RunLogger` 是业务链路唯一直接调用的日志入口，负责：

- 生成和保存 `run_id`。
- 记录 `trace_id`、`span_id`、`parent_span_id`。
- 统一补齐时间戳、日志等级、组件名和阶段名。
- 将事件分发给多个 Sink。
- 提供阶段上下文工具，例如 `with logger.stage("collect_source", source_key="hn_algolia")`。
- 捕获异常并输出 `failed` 事件。
- 提供脱敏能力，避免 Sink 各自重复处理敏感信息。

### 4.2 Sink

Sink 是日志输出目的地。第一版实现三个，后续预留三个。

| Sink | 作用 | 第一版是否实现 |
| --- | --- | --- |
| `ConsoleSink` | 终端中文实时输出 | 是 |
| `TextFileSink` | 人类可读 `.log` 文件 | 是 |
| `JsonlFileSink` | 结构化 `.jsonl` 文件 | 是 |
| `DbSink` | 写入运行记录表，供后台管理页读取 | 预留 |
| `RedisSink` | 写入实时运行状态，供前端订阅 | 预留 |
| `QueueSink` | 推送运行事件到消息队列 | 预留 |

### 4.3 与现有系统的接入点

第一版优先接入以下位置：

- `apps/worker/scripts/run_daily_pipeline.py`
- `apps/worker/worker/services/daily_pipeline_service.py`
- `apps/worker/scripts/collect_source_signals.py`
- `apps/worker/scripts/run_event_pipeline.py`
- `apps/worker/worker/agents/*`
- `apps/worker/worker/llm_client.py`

接入方式是把 logger 作为可选依赖传入，而不是让底层模块自己到处创建 logger。这样测试时可以注入内存 Sink，真实运行时可以注入终端和文件 Sink。

## 5. 日志事件协议

统一事件字段如下。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `timestamp` | string | ISO 8601 时间，带时区 |
| `level` | string | `debug/info/warning/error` |
| `run_id` | string | 一次 daily pipeline 运行 ID |
| `trace_id` | string | 一次运行内的追踪 ID，默认等于 `run_id` |
| `span_id` | string | 当前阶段 ID |
| `parent_span_id` | string/null | 父阶段 ID |
| `component` | string | `daily_pipeline/collector/selector/workflow/agent/llm/tool/db/scheduler` |
| `stage` | string | 阶段名，例如 `collect_source` |
| `event` | string | `started/heartbeat/succeeded/failed/skipped` |
| `status` | string/null | 业务状态，例如 `succeeded/partial_failed/no_new_signals` |
| `message_zh` | string | 中文可读消息 |
| `duration_ms` | int/null | 阶段耗时 |
| `source_key` | string/null | 来源标识 |
| `candidate_group_id` | string/null | 候选事件组 ID |
| `pipeline_run_id` | string/null | 数据库 pipeline run ID |
| `agent_run_id` | string/null | 数据库 agent run ID 或本轮临时 Agent 执行 ID |
| `agent_name` | string/null | Agent 名称 |
| `tool_name` | string/null | Agent 调用的工具名 |
| `provider` | string/null | LLM provider |
| `model` | string/null | LLM model |
| `retry_count` | int/null | 当前重试次数 |
| `token_usage` | object/null | `prompt_tokens/completion_tokens/total_tokens` |
| `counts` | object | 数量统计，例如写入、跳过、发布数 |
| `error_type` | string/null | 异常类型 |
| `error_message` | string/null | 脱敏后的错误摘要 |
| `metadata` | object | 少量扩展信息，必须可 JSON 序列化 |

JSONL 示例：

```json
{"timestamp":"2026-06-26T10:42:03+08:00","level":"info","run_id":"daily_20260626_104201","component":"collector","stage":"collect_source","event":"started","message_zh":"开始采集来源 hn_algolia","source_key":"hn_algolia","counts":{"limit":5}}
```

## 6. 输出文件设计

第一版写入 `runtime`：

```text
runtime/
  daily-pipeline-latest.log
  daily-pipeline-latest.jsonl
  daily-pipeline-20260626-104201.log
  daily-pipeline-20260626-104201.jsonl
```

规则：

- 每次运行创建一组带时间戳的日志文件。
- `latest.*` 覆盖为最近一次运行，便于快速查看。
- 定时任务 stdout / stderr 可以继续写入 `runtime/daily-pipeline-scheduler.log`，但业务细节由 `daily-pipeline-*.log/jsonl` 承担。
- 所有文件默认 UTF-8。
- `runtime` 产物不提交 Git。

## 7. 阶段日志清单

### 7.1 启动阶段

必须输出：

```text
[运行启动] run_id=daily_20260626_104201
[运行配置] source_group=daily_all lookback_hours=8 agent_mode=llm max_selected=5
[模型配置] provider=openai model=gpt-5.5
[数据库] 连接成功
[日志文件] runtime/daily-pipeline-20260626-104201.log
```

### 7.2 采集窗口与来源展开

必须输出：

```text
[采集窗口] start=2026-06-26T02:42:01+08:00 end=2026-06-26T10:42:01+08:00 lookback_hours=8
[来源展开] daily_all => hn_algolia, github_releases, github_repo_trends, openai_news
```

### 7.3 每个来源采集

每个 source 必须输出：

```text
[采集开始] source=hn_algolia limit=5
[采集成功] source=hn_algolia fetched=15 写入=3 跳过=12 耗时=4.2s
```

失败时：

```text
[采集失败] source=pytorch_blog 错误=HTTPStatusError message=403 Forbidden 耗时=11.0s continue=true
```

需要尽量记录：

- 请求 URL 或 profile key，但不打印 token。
- fetched / written / duplicate / skipped_stale / skipped_missing_published_at / skipped_future。
- source 是否失败后继续，不终止整轮流程。

### 7.4 采集汇总

必须输出：

```text
[采集汇总] 成功来源=10 失败来源=2 本轮新写入=8
[跳过统计] stale=37 missing_published_at=5 future=0 duplicate=12
```

### 7.5 新信号读取

必须输出：

```text
[新信号读取] since=2026-06-26T10:42:01+08:00 count=8
[新信号样例] 1. OpenAI releases ...
[新信号样例] 2. GitHub trending repo ...
```

样例最多 5 条，避免刷屏。

### 7.6 候选事件构造

必须输出：

```text
[候选构造开始] raw_signals=8
[硬过滤] empty_title=0 no_url=0 already_processed=0 stale=0
[候选合并] groups=6 merged_by_url=1 merged_by_title=1
[候选样例] group=... title=... sources=2
```

如果当前代码暂时无法拿到细粒度过滤统计，第一版可以先输出 `groups` 和样例，后续再补 service 级统计。

### 7.7 LLM 编辑筛选

必须输出：

```text
[编辑筛选开始] groups=6 provider=openai model=gpt-5.5 batch_size=30
[编辑筛选成功] selected=4 rejected=2 manual_review=0 耗时=28.1s
[入选] group=... score=91 reason=外网讨论热度高且与 AI 开发者相关
[拒绝] group=... reason=信息重复且影响范围较小
```

失败时：

```text
[编辑筛选失败] error=JSONDecodeError retry=2 fallback=false
```

默认不打印完整 prompt 和 LLM 原始响应。

### 7.8 单个事件生产

每个 selected candidate 必须输出：

```text
[事件生产开始] 1/4 candidate=... title=...
[资料写作开始] agent=ResearchWriterLLMAgent
[资料写作成功] words=850 retries=0 耗时=38.2s
[审稿发布开始] agent=ReviewPublisherLLMAgent
[审稿通过] decision=publish slug=...
[事件发布成功] published_event_id=... slug=...
```

失败时：

```text
[事件生产失败] candidate=... stage=review error=JSONDecodeError action=manual_review
```

### 7.9 最终总结

必须输出：

```text
[运行完成] status=succeeded duration=3m19s
[最终统计] raw_signals=8 candidate_groups=6 selected=4 published=4 failed=0
[日志文件] runtime/daily-pipeline-20260626-104201.log
```

## 8. Agent 级监控

Agent 日志不是后续可选项，而是本基座的核心能力。第一版至少覆盖现有三类 Agent：

- `EditorialSelectorLLMAgent`
- `ResearchWriterLLMAgent`
- `ReviewPublisherLLMAgent`

每个 Agent 必须输出：

```text
[Agent开始] agent=ResearchWriterLLMAgent candidate=group_xxx model=gpt-5.5
[Agent输入摘要] agent=ResearchWriterLLMAgent signals=2 sources=hn_algolia,github_repo_trends chars=1800
[LLM请求开始] agent=ResearchWriterLLMAgent provider=openai model=gpt-5.5 retry=0
[LLM请求成功] agent=ResearchWriterLLMAgent prompt_tokens=12000 completion_tokens=1800 耗时=31.2s
[结构校验成功] agent=ResearchWriterLLMAgent schema=EventDossierDraft
[Agent完成] agent=ResearchWriterLLMAgent status=succeeded words=850 耗时=38.2s
```

失败和重试必须输出：

```text
[LLM请求失败] agent=ReviewPublisherLLMAgent error=APITimeoutError retry=1 will_retry=true
[结构校验失败] agent=ReviewPublisherLLMAgent error=JSONDecodeError retry=2
[Agent失败] agent=ReviewPublisherLLMAgent status=failed action=manual_review
```

如果 Agent 调用工具，必须输出工具级日志：

```text
[工具调用开始] agent=ResearchWriterLLMAgent tool=fetch_source_snapshot source=github_repo_trends
[工具调用成功] agent=ResearchWriterLLMAgent tool=fetch_source_snapshot bytes=18234 耗时=1.4s
```

Agent 级日志需要记录：

| 类别 | 必填信息 |
| --- | --- |
| 身份 | `agent_name`、`agent_run_id`、`candidate_group_id`、`pipeline_run_id` |
| 模型 | `provider`、`model`、`temperature` 如果可得 |
| 输入摘要 | signal 数量、source 列表、输入字符数、是否含正文快照 |
| 工具调用 | `tool_name`、开始 / 结束 / 失败、耗时、结果摘要 |
| LLM 调用 | 开始 / 结束 / 失败、重试次数、token usage、耗时 |
| 输出校验 | schema 名称、校验是否成功、失败字段摘要 |
| 结果 | selected / rejected / manual_review、publish / revise / reject、slug、字数等 |
| 降级 | 是否 fallback、fallback 原因 |

默认仍不打印完整 prompt、完整 LLM response 和完整正文。需要调试时，后续可以增加脱敏后的 debug artifact 文件，但不作为默认终端输出。

## 9. Agent 日志组织方式

第一版不建议把每个 Agent 物理拆成独立主日志文件，因为这样会让跨 Agent 排查变困难。推荐方式是：

- 主日志仍是统一时间线：`daily-pipeline-<run_id>.log/jsonl`。
- 每条 Agent 日志都带 `agent_name`、`agent_run_id`、`candidate_group_id`。
- 后续如有需要，再从 JSONL 派生 Agent 视图。

```text
runtime/agents/<run_id>/EditorialSelectorLLMAgent.log
runtime/agents/<run_id>/ResearchWriterLLMAgent.log
runtime/agents/<run_id>/ReviewPublisherLLMAgent.log
```

这样既能按时间线看完整 pipeline，也能按 Agent 单独看执行细节。

## 10. 心跳机制

第一版心跳间隔为 30 秒。

当某个阶段超过 30 秒仍未结束，输出：

```text
[仍在运行] stage=collect_source source=github_repo_trends elapsed=30s
[仍在运行] stage=collect_source source=github_repo_trends elapsed=60s
```

心跳必须包含：

- 当前阶段。
- 当前 source / agent / candidate，如果有。
- 已运行时长。

心跳不负责终止任务。真正的超时控制后续由 source 请求超时、LLM 超时或任务执行上限负责。

## 11. 错误与脱敏

错误日志分两层：

- 终端：中文摘要，适合人看。
- JSONL：结构化错误字段，适合后续检索。

脱敏规则：

- URL 中的密码段替换为 `***`。
- API key、token、authorization、cookie 字段统一替换为 `[REDACTED]`。
- 错误信息中出现 `.env` 中的敏感值时必须替换。
- 数据库连接串只允许输出 driver、host、port、database，不允许输出用户名和密码。
- Prompt、response、HTML 正文默认只输出长度、hash 或摘要，不输出全文。

## 12. 后续扩展口径

### 12.1 数据库运行记录

后续可以新增运行日志表，或扩展现有 `pipeline_runs/agent_runs`：

```text
run_log_events
  id
  run_id
  timestamp
  level
  component
  stage
  event
  message_zh
  payload_json
```

后台管理页可直接查询该表。

Agent 级 DB 视图可以优先复用现有 `agent_runs`，再补日志事件表：

```text
agent_run_events
  id
  run_id
  pipeline_run_id
  agent_run_id
  agent_name
  timestamp
  event
  message_zh
  payload_json
```

后台页未来可以按 Agent 展示成功率、平均耗时、失败原因、token 成本和 fallback 次数。

### 12.2 Redis 实时状态

后续 Redis Sink 可以维护：

```text
daily_pipeline:latest_status
daily_pipeline:{run_id}:events
daily_pipeline:{run_id}:current_stage
```

用于前端后台页实时刷新。

### 12.3 消息队列

后续 Queue Sink 可以把日志事件发布到：

```text
pipeline.events
agent.events
collector.events
```

用于异步消费者、通知系统或审计系统。

### 12.4 问答助手

问答助手未来可以读取 JSONL / DB 日志回答：

- “今天 13 点那次为什么失败？”
- “哪个 source 最近最不稳定？”
- “LLM selector 为什么没选这个事件？”
- “ResearchWriterLLMAgent 最近失败最多的原因是什么？”

## 13. 最小实施顺序

后端代理后续实施时，应按下面顺序做，每一步都要有测试：

1. 新增 `worker/observability/run_logger.py`，实现事件模型、脱敏、Console/Text/JSONL Sink。
2. 为 RunLogger 写单元测试，覆盖中文终端格式、JSONL 合法性、文件写入和脱敏。
3. `run_daily_pipeline.py` 创建 logger，并传给 `DailyPipelineService`。
4. `DailyPipelineService` 在启动、采集前后、新信号读取、候选构造、selector、per-candidate、最终总结阶段打日志。
5. `collect_source_signals.py` 和 source collector 统一返回或记录采集统计。
6. `run_event_pipeline.py` 和 LangGraph 节点接入 workflow / Agent 阶段日志。
7. LLM Agent wrapper 记录 LLM 请求、响应摘要、token usage、结构校验、重试和失败。
8. 长阶段加 30 秒 heartbeat。
9. 更新测试记录，记录终端输出样例和 `runtime/*.log/jsonl` 样例。

## 14. 验收标准

第一版实现完成后必须满足：

- 手动运行 `python scripts/run_daily_pipeline.py` 时，终端实时输出中文阶段日志。
- `runtime/daily-pipeline-latest.log` 存在，并包含本轮完整阶段日志。
- `runtime/daily-pipeline-latest.jsonl` 存在，每行是合法 JSON。
- 每个 source 至少有 `started` 和 `succeeded/failed`。
- selector 至少有 `started` 和 `succeeded/failed`。
- 每个 selected candidate 至少有生产开始、writer、reviewer、发布或失败日志。
- 每个 Agent 至少有开始、LLM 请求、结构校验、完成或失败日志。
- Agent 日志包含 `agent_name`、`agent_run_id`、`candidate_group_id`、模型名、重试次数和 token usage，如果当前 provider 返回 token usage。
- 失败路径会记录 `error_type` 和脱敏后的 `error_message`。
- 长阶段每 30 秒出现 heartbeat。
- 日志系统异常不影响主流程。
