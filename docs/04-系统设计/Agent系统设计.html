System Design

AI World Radar Agent 系统设计

终局架构蓝图与阶段化落地口径：用 LangGraph 编排编辑部式 Multi-Agent 工作流，将全球 AI 圈碎片化信号生产成中文事件档案。

1. 文档目的

本文档定义 AI World Radar 的新版 Agent 系统设计。它描述终局形态下系统如何使用编辑部式 Multi-Agent 工作流，把全球 AI 圈的碎片化公开信号加工成中文用户能理解、可追踪、可修正的事件档案。

本文档同时明确 P1 的落地边界：终局蓝图可以完整成熟，但第一阶段实现必须克制，先完成单机可运行的事件生产闭环，再逐步引入队列、缓存、对象存储、向量检索、成本监控等增强能力。

2. 核心结论

AI World Radar 的 Agent 系统不是自由上网写新闻的万能 Agent，也不是普通爬虫后面接一个摘要 prompt。它是基于 LangGraph 的编辑部式 Multi-Agent 情报生产系统。
终局架构：
LangGraph 工作流编排
+ 编辑部式主 Agent
+ 能力型子 Agent / Skill
+ 工程 Tool 白名单
+ PostgreSQL 状态与审计记录
+ 后台监控与人工修正

P1 实现：
单机 Python Worker
+ PostgreSQL
+ 本地文件缓存
+ 同步 pipeline
+ 最小 Agent 闭环
+ 最小运行记录

3. 业务生产流程

P1 的核心产物不是简报，也不是新闻列表，而是事件档案。

外部信号
-> 标准化信号
-> 判断哪些值得关注
-> 把重复信号合并成同一事件
-> 给事件排序
-> 整理事件资料
-> 写成中文事件档案
-> 审稿检查
-> 发布到前台
-> 后台可监控、可修改、可隐藏

外部信号可以来自 Hacker News、GitHub、Hugging Face、公司官网、AI 新闻站、中文聚合站，以及后续扩展的 X、Reddit、YouTube 等平台。信号刚进入系统时不能直接展示给用户，系统要把它们加工成用户能看懂的事件档案。

4. 职责边界

工程代码负责确定性动作：采集、解析、正文缓存、URL 归一化、hash、数据库读写、状态流转、发布写库、重试控制、运行日志和幂等控制。

Agent 负责语义判断和内容生产：AI 相关性判断、事件归并、展示价值判断、中文用户价值判断、资料整理、事件卡和详情正文生成、审稿检查和修订。

Agent 可以提出判断、草稿和建议，但不能直接修改核心状态，不能直接发布，不能绕过工程服务。

5. 总体架构

1. Source Layer 信息源层
2. Workflow Layer 工作流编排层
3. Agent Layer 编辑部 Agent 层
4. Tool / Skill Layer 能力层
5. Product Service Layer 产品服务层
Source Adapter
-> Source Signal
-> LangGraph Workflow
   -> On-duty Editor Agent
   -> Research Writer Agent
   -> Review Publisher Agent
-> Publish Service
-> PostgreSQL
-> Frontend / Admin Console

Source Layer 由工程代码控制采集。Workflow Layer 使用 LangGraph 控制状态机。Agent Layer 模拟真实编辑部岗位。Tool / Skill Layer 区分语义能力和确定性工具。Product Service Layer 负责数据库写入、发布、隐藏、重跑和后台操作记录。

6. 编辑部式主 Agent

值班编辑 Agent

负责回答“这件事值不值得做成一个事件”。职责包括 AI 相关性判断、信号筛选、事件归并、事件类型识别、热度/重要度/中文用户价值判断、优先级和写作角度。

研究写作 Agent

负责回答“怎么把这件事讲清楚”。职责包括整理来源材料、生成 research package、首页事件卡、详情正文、为什么值得关注、后续关注点，并根据审稿意见重写。

审稿发布 Agent

负责回答“这篇事件档案能不能发”。职责包括结构完整性、来源支撑、过度推断、搬运风险、标题党、空泛内容检查，并输出 publish、revise、manual_review 或 reject 建议。

7. 能力型子 Agent / Skill

能力型子 Agent / Skill 是主 Agent 可注册、可复用、可替换的智能能力。P1 不要求全部独立实现，可以先把部分 Skill 实现为结构化 LLM 调用，后续再升级为更完整的子 Agent。

Skill归属主 Agent作用
AI 相关性判断 Skill值班编辑判断信号是否属于 AI 圈
事件归并 Skill值班编辑判断多个信号是否是同一事件
选题角度 Skill值班编辑判断解释角度
来源摘要 Skill研究写作将长来源压缩成材料
事件卡生成 Skill研究写作生成首页卡片字段
详情正文写作 Skill研究写作生成自然中文详情正文
搬运风险检查 Skill审稿发布检查是否过度接近原文
过度推断检查 Skill审稿发布检查来源是否支撑结论

8. 工程 Tool

工程 Tool 负责确定性动作，必须白名单管理。示例包括读取来源正文、查询相似候选事件、查询历史已发布事件、计算基础热度分、URL 归一化、生成来源幂等键、读取候选事件支撑信号、保存中间产物和记录 Agent 运行。

工具默认只读。发布、隐藏、删除、状态流转不作为 Agent 可直接调用的工具。

9. LangGraph 工作流设计

collect_signals
-> normalize_signals
-> editorial_triage
-> merge_and_rank_events
-> build_research_package
-> draft_event_dossier
-> review_event_dossier
-> revise_if_needed
-> publish_or_manual_review
-> record_run

修订循环必须有限制：最多修订 2 次，超过 2 次仍不通过，进入 manual_review。

Workflow State 至少包含 run_id、source_batch_id、candidate_id、dossier_id、current_node、status、retry_count、revision_count、last_error、agent_outputs 和 review_decision。

10. 数据模型蓝图

数据库可以按新版生产线推倒重来。核心生产链路为：

Signal
-> EventCandidate
-> EventDossier
-> ReviewResult
-> PublishedEvent
表用途
sources信息源配置表
source_signals外部信号标准化结果
event_candidates多个 signal 归并后的内部候选事件
event_candidate_signals候选事件和信号关联
event_dossiers事件档案草稿，支持版本
review_results审稿发布 Agent 的检查结果
published_events前台稳定读取的已发布事件快照
pipeline_runs完整生产任务运行记录
agent_runs主 Agent、Skill 或 LLM 节点运行记录
tool_calls终局独立工具调用记录，P1 可先放入 trace_json
admin_actions后台人工操作记录

11. 状态设计

event_candidate:
new / triaged / merged / drafting / reviewing / ready_to_publish /
published / manual_review / rejected / failed / hidden

event_dossier:
draft / reviewing / needs_revision / approved / manual_review /
rejected / published_snapshot

pipeline_run:
running / succeeded / partial_failed / failed / cancelled

12. 发布与人工干预边界

发布必须由工程发布服务执行。Agent 可以输出发布建议、修订建议、风险说明和人工处理建议，但不能直接写 published_events、隐藏内容、删除候选事件或绕过 review_result。

自动发布要求 dossier 结构完整、审稿结果为 publish、来源链接存在、未超过风险阈值且幂等检查通过。风险内容进入 manual_review。

13. 运行记录与可观测性

P1 必须记录 pipeline run、Agent run、节点状态、输入摘要、输出结构化结果、错误信息、retry_count、revision_count、发布记录和人工操作记录。

终局可增强 tool_calls、token 和成本估算、模型耗时统计、节点成功率、来源成功率、失败原因聚合和告警。

14. 工程增强能力的阶段边界

P1 必须预留 run_id、agent_run_id、状态字段、幂等键、source_hash、content_hash、dossier.version、review_result、manual_review 状态、运行记录和后台操作记录。

P1 暂不做 Redis、队列、对象存储、向量数据库、复杂调度、复杂监控大盘、多用户权限、多模型成本路由和全平台社媒采集。

15. LLMClient 边界

LLMClient 是模型调用底座，不是业务 Agent。它只负责 provider、model、api_key/base_url、chat、stream_chat 和 OpenAI-compatible 调用。业务 prompt、Agent 状态、工具绑定、结构化 schema、repair 策略、发布状态和数据库写入都不属于 LLMClient。

16. 质量门禁

P1 的质量门禁不是完整真实性核验。P1 只检查内容是否为空、结构是否完整、标题和正文是否一致、是否缺少来源链接、是否搬运原文、是否夸张、是否把不确定内容写成确定事实，以及详情正文是否能让中文用户读懂。

17. 阶段化落地路线

P1：最小硬核闭环

用单机 Python Worker 跑通从外部信号到已发布事件档案的闭环。包含最小数据表、三个主 Agent 的最小实现、最多 2 次修订循环和运行记录。

P1.5：内容组织增强

基于已发布事件生成今日简报，并增强信息源和事件归并能力。

P2：交互与研究增强

事件追问助手、事件追踪时间线、更多来源补充、专题聚合、用户收藏和更成熟后台。

P3：产品化与规模化

个性化订阅、多用户体系、队列化生产、成本监控、对象存储、复杂监控大盘和深度事实核查。

18. 风险与应对

  • 过度工程化：终局蓝图完整，实现按阶段分批落地。
  • 假 Agent 化：主 Agent 对应真实业务岗位，确定性动作保持为 Tool。
  • 成本和延迟：限制节点数量、修订次数，预留缓存和模型分层空间。
  • 事件归并错误：宁可重复、不乱合并，后台允许人工修正。
  • 自动发布风险：风险内容进入 manual_review。
  • 数据模型复杂：P1 落核心表和必要字段，复杂记录后续拆出。
