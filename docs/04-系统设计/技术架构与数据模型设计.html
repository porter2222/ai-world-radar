P1 MVP Technical Design

AI World Radar 技术架构与数据模型设计

定义 P1 MVP 的技术栈、模块分工、运行链路和核心数据表，采用轻量产品化架构，优先保证可落地、可追溯、可管理。

版本：v0.1 更新时间：2026-06-04 阶段：P1 MVP

1. 文档目的

本文档用于定义 AI World Radar P1 MVP 的技术架构、模块分工、运行链路和核心数据模型。

如果本文档与早期 Agent 系统设计文档存在口径差异，以本文档为最新 P1 技术口径。

2. 当前核心拍板

  • 技术栈采用 Next.js + Python Worker + PostgreSQL。
  • P1 暂不引入 FastAPI。
  • P1 不单独建 RawItem 表。
  • P1 不单独建 CandidateEvent 表。
  • P1 不做真实性核验、可信状态标签和事实确认评分。
  • 排序采用热度主导 + 重要度兜底 + 中文用户价值修正。
  • 所有前台公开展示的事件都必须有详情页。
  • 今日简报只从已发布事件中精选。

3. P1 技术架构

Next.js Web App
  -> 前台首页 / 今日简报 / 事件详情页 / 管理页
  -> 简单 API / Server Action

Python Worker
  -> Source 采集 / EvidenceCard 生成 / EventCluster 聚合
  -> ContentArtifact 生成 / QualityGate 检查 / 发布状态写入

PostgreSQL
  -> 来源配置 / 证据卡 / 事件聚合 / 内容版本
  -> 发布事件 / 简报 / 运行日志 / 管理操作

P1 暂不引入 FastAPI、Redis 队列、微服务拆分、复杂权限系统、embedding 聚类和真实性核验系统。

4. 模块分工

4.1 Next.js

Next.js 负责用户界面和轻量后端接口，包括首页、今日简报、事件详情页、管理页、查询已发布事件、查询简报，以及隐藏、恢复、置顶、排序等管理操作。

4.2 Python Worker

Python Worker 负责后台自动生产链路，包括采集来源、生成 EvidenceCard、聚合 EventCluster、生成 ContentArtifact、执行 Quality Gate、写入 PublishedEvent 和 Brief。

4.3 PostgreSQL

PostgreSQL 是系统唯一主数据存储，保存来源配置、Agent 生产结果、发布状态、简报数据、运行审计和后台操作记录。

5. P1 运行链路

Source
-> PipelineRun
-> EvidenceCard
-> EventCluster
-> EventClusterCard
-> ContentArtifact
-> QualityGateResult
-> PublishedEvent
-> Brief / BriefItem

管理链路为：Admin Page -> Next.js Action -> 更新 PublishedEvent / ContentArtifact / Brief -> 写入 AdminAction。

7. sources

记录系统长期采集哪些来源，以及这些来源在发现、热度和重要度上的权重。

字段含义
source_id来源业务标识，例如 openai_news、github_trending
name来源名称
source_typeofficial / platform / newsletter / competitor / community
fetch_methodrss / api / html / github / hn_api
url来源入口地址
enabled是否启用
discovery_weight发现权重
heat_signal_weight热度信号权重
impact_signal_weight重要度信号权重
capabilities来源能力说明，例如是否有发布时间、热度指标、作者、图片
last_statussuccess / failed / paused / degraded

P1 不在 sources 中保存真实性、可信度或事实锚定权重。

8. pipeline_runs

记录系统每次自动运行的整体情况。

字段含义
pipeline_run_id唯一 ID
run_typescheduled / manual / retry
triggered_bysystem / admin
statusrunning / success / partial_failed / failed
started_at / finished_at开始与结束时间
source_count本轮处理来源数
evidence_card_count生成证据卡数量
event_cluster_count生成事件聚合数量
content_artifact_count生成内容产物数量
published_event_count最终发布事件数量
failed_step失败步骤
error_summary错误摘要

9. evidence_cards

保存来源信号和 AI 理解结果。P1 不单独建 RawItem，因此该表同时保存原始条目摘要和结构化理解。

字段含义
evidence_card_id唯一 ID
pipeline_run_id由哪次运行生成
source_id来自哪个来源
original_title / original_url原始标题与链接
canonical_url规范化链接
item_typearticle / post / repo / model / video / newsletter_item
published_at / discovered_at来源发布时间与系统发现时间
raw_summary / raw_excerpt来源摘要与短摘
raw_heat_metrics原始热度指标
raw_metadata来源专属字段
claim_summary一句话概括内容主张
subjects涉及主体
event_trigger / event_type触发点与事件类型
heat_clues / impact_clues热度与重要度线索
suggested_routehigh_heat_candidate / high_impact_candidate / normal_candidate / discarded
merge_key_hint聚合线索
processing_statuspending / processed / clustered / discarded / failed

该表不保存真实性评分、官方确认状态或可信度标签。

18. 关键设计取舍

  • P1 不建 RawItem，降低数据模型复杂度。
  • P1 不建 CandidateEvent，候选判断字段放入 evidence_cards。
  • P1 不做真实性系统，Quality Gate 只做生成质量和来源追溯检查。
  • P1 不做 FastAPI，先使用 Next.js + Python Worker + PostgreSQL。
