Technical Architecture

AI World Radar 技术架构与数据模型设计

终局技术蓝图与 P1 单机落地口径：用 Next.js、Python Worker、PostgreSQL、Pydantic、SQLAlchemy、Alembic 与 LangGraph 支撑编辑部式事件档案生产链路。

1. 文档目的

本文档定义新版技术架构、模块边界、运行链路、Python Worker 内部结构、Pydantic / SQLAlchemy 分层原则、核心数据模型和阶段化工程扩展路径。

本文档基于新版 PRD v2.0 与新版 Agent 系统设计 v1.0。若本文档与旧版技术架构、旧版后端 P1 实现计划、旧版 Agent stub 命名存在差异，以本文档和新版 Agent 系统设计为准。

2. 核心结论

P1 技术架构：\nNext.js + Python Worker + PostgreSQL\n\nP1 必须使用：\nPydantic / SQLAlchemy / Alembic / LangGraph / LLMClient / pytest\n\nP1 暂不引入：\nFastAPI / Redis / 队列 / 对象存储 / 向量数据库 / 复杂监控
P1 的第一目标是跑通“外部信号 -> Agent 生产事件档案 -> 审稿 -> 发布 -> 前台/后台可查看”的闭环。不要让第一版被三服务架构、队列、缓存和复杂基础设施拖慢。

3. 总体架构

Next.js Web App\n  -> 首页 / 事件雷达页\n  -> 事件详情页\n  -> 后台管理页\n  -> 轻量查询 API / Server Action\n\nPython Worker\n  -> 信息源采集\n  -> 正文缓存\n  -> LangGraph workflow\n  -> 编辑部式 Agent\n  -> Skill / Tool 调用\n  -> Pydantic 结构化校验\n  -> 审稿修订循环\n  -> 发布服务写库\n  -> 运行报告\n\nPostgreSQL\n  -> 业务数据\n  -> 事件状态\n  -> Agent 运行记录\n  -> 后台操作记录

Next.js 和 Python Worker 在 P1 阶段不通过 HTTP 服务通信，而是通过 PostgreSQL 协作。P1 不引入 FastAPI，未来复杂任务控制再抽出 FastAPI Agent Service。

4. 模块职责

Next.js Web App

负责首页、事件详情页、后台管理页、轻量查询 API / Server Action、后台操作写库和重跑请求状态。

Python Worker

负责读取来源、采集和标准化信号、正文缓存、LangGraph workflow、三类主 Agent、Pydantic 校验、审稿修订循环、发布服务写库和本地运行报告。

PostgreSQL

作为系统事实源，保存信息源、信号、候选事件、事件档案、审稿结果、已发布事件、运行记录和后台操作记录。

7. Pydantic 与 SQLAlchemy 分层

Pydantic 负责 Agent 输入输出、Workflow State、Service DTO、数据库写入前校验和测试 fixture。SQLAlchemy 负责 ORM 模型、字段映射、关系约束、查询和写入。Alembic 负责 migration。

Pydantic schema 不直接等于数据库表；SQLAlchemy model 不直接等于 Agent 输出；Service 层负责把 Pydantic 输出转换成数据库写入。

11. 幂等与去重

P1 必须保证重复运行不能重复发布同一个事件。建议约束包括 source_key 唯一、source_hash 防重复采集、candidate_key 防重复候选事件、candidate-signal 关联唯一、dossier 版本唯一。

12. 正文缓存与对象存储边界

P1 不引入对象存储，先使用本地文件缓存。PostgreSQL 保存 content_hash、content_cache_path、source_hash、摘要和抓取状态。

runtime/source-pages/\nruntime/artifacts/\nruntime/pipeline-reports/

后续可迁移到 AWS S3、阿里云 OSS、腾讯云 COS、Cloudflare R2 或 MinIO。

13. FastAPI 边界

P1 不引入 FastAPI。原因是 P1 不需要 Python 对外提供 HTTP 服务，Next.js 足以承担前台和后台轻量 API，Python Worker 通过 PostgreSQL 协作即可。

P1 必须认真设计 Pydantic schema，以便未来 FastAPI 复用。未来需要实时 Agent 服务、任务状态查询、取消任务和重试任务时，再引入 FastAPI。

14. LangGraph 与 Agent 集成

collect_signals\nnormalize_signals\neditorial_triage\nmerge_and_rank_events\nbuild_research_package\ndraft_event_dossier\nreview_event_dossier\nrevise_if_needed\npublish_or_manual_review\nrecord_run

Workflow State 使用 Pydantic 定义，Agent 输出必须经过 Pydantic 校验。解析失败最多 repair 2 次，仍失败则标记 failed 或 manual_review。

16. 测试策略

P1 必须有 pytest，覆盖 Pydantic schema、source adapter fixture、content cache、Agent 输出解析、workflow state、service 幂等、pipeline 集成和数据库 migration。网络测试必须用 network marker 标记。

17. 阶段化工程路线

P1

Next.js 基础页面、Python Worker 单机命令、PostgreSQL 新核心表、Pydantic、SQLAlchemy、Alembic、LangGraph event pipeline、三类主 Agent 最小实现、本地文件缓存和运行记录。

P1.5

今日简报、更多信息源、embedding 辅助归并、独立 tool_calls 表、更强后台批量操作。

P2

FastAPI Agent Service、任务队列、实时任务状态、事件追问助手、对象存储和更强监控。

P3

个性化订阅、多用户权限、成本监控、复杂告警、深度事实核查和产品级部署。

18. 与旧版工程骨架的关系

旧版 apps/worker 工程骨架可参考 HN collector、原文缓存、pytest、pipeline report 和 LLMClient 接入方式，但旧表模型、旧 Agent stub 命名、Brief 优先链路和 EvidenceCard / EventCluster / ContentArtifact 命名需要重构。

19. 下一步

本文档确认后，进入 implementation plan 前，应先拆出 P1 技术落地任务：新数据库 migration、Pydantic schema、SQLAlchemy model、LangGraph workflow state、三类主 Agent schema、旧 worker 迁移或重建策略、测试计划与验收记录模板。
