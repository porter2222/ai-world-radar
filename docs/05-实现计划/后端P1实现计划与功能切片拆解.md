# 后端 P1 实现计划与功能切片拆解

版本：v0.1

更新时间：2026-06-08

阶段：后端 P1 第一功能切片确认版

## 1. 文档目的

本文档用于定义 AI World Radar 后端 P1 的第一条可交付功能切片：HN AI 事件生产闭环。

它不是后端全部功能的完整开发清单，也不是前端 UI 设计文档。它要解决的是：

- 后端 P1 先做什么。
- 第一条纵向功能切片如何跑通。
- 哪些部分由工程代码负责，哪些部分由 Agent 负责。
- 本地如何运行、如何验收、如何给后续开发代理交接。

本文档确认后，后续后端代理、Agent 代理、验收代理应以本文档为第一实现依据。

## 2. 当前阶段与开发原则

当前项目已经完成产品宪章、PRD、信息源策略、Agent 系统设计 v0.2、技术架构与数据模型设计、采集策略调研和 QFT 晨报项目参考价值调研。

后端 P1 的开发原则是：

> 模块是实现单元，功能切片是交付单元。

因此 P1 不按技术层长期堆模块，也不先把所有基础设施一次性搭完，而是从一个能真实跑通业务价值的纵向闭环开始。

第一条闭环选择 Hacker News，是因为：

- HN 有公开可用的 Algolia API。
- HN 有 points、comments、created_at 等热度字段。
- HN 适合发现工程师圈正在讨论的 AI 基建、开发者工具、模型和争议。
- HN 接入成本低，适合冷启动验证 Agent 情报生产链路。

## 3. 第一功能切片定义

后端 P1 第一功能切片定义为：

> HN AI 事件生产闭环。

目标链路如下：

```text
HN Algolia API
-> HN story 采集
-> original_url 原文补全
-> EvidenceCard
-> EventCluster
-> Ranking
-> DetailWriter
-> BriefWriter
-> QualityGate
-> PublishService
-> PostgreSQL
-> PipelineRun 报告
```

这条切片完成后，系统应能通过本地 Python 文件触发一次完整运行，并在 PostgreSQL 中看到可发布事件和今日简报。

## 4. 本切片不做什么

第一切片聚焦冷启动闭环，不做以下内容：

- 不做通用爬虫系统。
- 不做完整 Source Adapter 框架。
- 不做 X / Reddit / YouTube / Facebook 采集。
- 不做 GitHub Trending / Hugging Face Trending。
- 不做 FastAPI。
- 不做 Next.js 前端页面。
- 不做定时任务和队列。
- 不做登录、用户系统、订阅、收藏。
- 不做真实性核验。
- 不做 embedding 聚类。
- 不做自由上网型 Agent。
- 不做人工前置审核后台。

这些内容可以进入后续 P1 切片、P1.5 或 P2。

## 5. 完成定义

本切片完成的最低标准是：

```text
能用一个本地 Python 文件跑完整条链路，
并且 PostgreSQL 中有 EvidenceCard、EventCluster、PublishedEvent、Brief 和 BriefItem。
```

更具体地说：

- 能连接本机 PostgreSQL。
- Alembic 能创建 P1 核心表。
- 能请求 HN Algolia API。
- 能抓取最近 7 天内最多 100 条 HN story。
- 能根据 story 的 original_url 抓取完整可读正文并保存到本地缓存。
- 能生成 EvidenceCard。
- 能生成 EventCluster。
- 能排序并选择最多 10 条 PublishedEvent。
- 能为每条 PublishedEvent 生成详情内容。
- 能从 PublishedEvent 中生成 1 份 Brief 和最多 5 条 BriefItem。
- 能执行 Quality Gate，失败内容不发布。
- 能输出终端报告和 txt 运行报告。
- 重复运行不制造重复 PublishedEvent。

## 6. 运行入口

P1 第一版不先做 CLI 框架，直接运行 Python 文件。

推荐命令：

```bash
cd apps/worker
python scripts/run_hn_pipeline.py --days 7 --limit 100
```

参数含义：

- `--days 7`：抓取最近 7 天窗口。
- `--limit 100`：最多保留 100 条 HN story。
- `--force`：强制重生成已处理过的内容。

默认运行不带 `--force` 时，应复用已存在的 EvidenceCard、EventCluster 和 PublishedEvent，避免开发调试时重复消耗 LLM 调用。

## 7. Worker 目录结构

P1 第一版建议目录如下：

```text
apps/worker/
  scripts/
    run_hn_pipeline.py

  worker/
    config.py
    llm_client.py

    db/
      session.py
      models.py
      migrations/

    collectors/
      hn_algolia.py

    agents/
      evidence_agent.py
      event_cluster_agent.py
      ranking_agent.py
      detail_writer_agent.py
      brief_writer_agent.py
      quality_gate_agent.py

    pipelines/
      hn_event_pipeline.py

    reports/
      pipeline_report_writer.py
```

职责划分：

- `scripts/run_hn_pipeline.py`：本地启动入口。
- `collectors/hn_algolia.py`：请求 HN Algolia API，做基础去重和排序。
- `pipelines/hn_event_pipeline.py`：串起完整 HN 事件生产链路。
- `agents/`：放各个 Agent 节点。
- `db/`：放 SQLAlchemy、Alembic 和数据库连接。
- `reports/`：生成 txt 运行报告。
- `llm_client.py`：迁入用户已有 LLMClient，只提供底层模型调用能力。

## 8. HN Algolia 冷启动采集策略

P1 第一版只使用 HN Algolia API 做冷启动采集。

采集口径：

```text
时间窗口：最近 7 天
最多保留：100 条 story
采集对象：story
排序预处理：hn_heat_score = points + num_comments * 2
```

第一版使用固定 query profile，不让 Agent 自由发明关键词。

建议初始关键词：

```text
OpenAI
ChatGPT
Anthropic
Claude
Google DeepMind
Gemini
NVIDIA
Hugging Face
LLM
AI agent
AI coding
Cursor
GitHub Copilot
MCP
machine learning
```

执行策略：

```text
每个关键词请求一次 HN Algolia API
-> 按 objectID / url 去重
-> 使用 created_at_i 限制 7 天窗口
-> 使用 hn_heat_score 做工程预排序
-> 截取前 100 条
```

注意：

- HN 是热议源，不是事实源。
- HN points 和 comments 只表示社区热度。
- HN 评论只能作为讨论证据，不能写成事实。

## 9. original_url 原文补全与本地缓存

HN Algolia 提供热度和原文链接，但通常不提供原文正文。为了避免 DetailWriterAgent 根据标题脑补，P1 应补全 story 的 original_url。

补全策略：

```text
HN story
-> original_url
-> 抓取可读正文
-> 保存完整正文到 runtime/source-pages/
-> EvidenceCard 保存正文摘要、缓存路径和 hash
```

边界：

- 只抓 HN story 自带的 original_url。
- 不递归抓取页面内其他链接。
- 不登录。
- 不绕过反爬。
- 不执行复杂浏览器自动化。
- 不把全文展示给用户。
- 不把全文提交到 Git。

推荐缓存目录：

```text
runtime/source-pages/
```

示例文件名：

```text
runtime/source-pages/2026-06-08/hn-123456-page.txt
```

数据库中不直接塞大段全文，只保存必要指针：

```text
original_url
page_title
page_excerpt
page_text_hash
page_cache_path
fetched_at
fetch_status
```

Agent 使用时可以根据正文长度控制上下文：

- 正文不长：可直接进入 Agent 上下文。
- 正文很长：先截取或后续再升级为分块摘要。

## 10. EvidenceCard 生成策略

P1 不单独建 RawItem 表，也不单独建 CandidateEvent 表。

一条 HN story 对应一张 EvidenceCard：

```text
HN Story -> EvidenceCard
```

EvidenceCard 第一版最小字段集：

```text
source_id
source_item_id / hn_id
original_title
original_url
author
published_at
points
num_comments
story_text
page_title
page_excerpt
page_text_hash
page_cache_path
claim_summary
normalized_title
subjects
event_type
category
heat_clues
candidate_score
processing_status
```

EvidenceAgent 负责从 HN story、原文补全结果和基础热度字段中提取：

- 是否 AI 相关。
- 主体。
- 触发点。
- 事件类型。
- 展示分类。
- 热度线索。
- 中文用户价值理由。
- 候选分数。

## 11. EventCluster 聚合策略

EventCluster 用于把多个 EvidenceCard 聚合成内部事件。

P1 不做 embedding 聚类。第一版采用：

```text
规则去重 + LLM 模糊合并
```

聚合流程：

```text
EvidenceCard 列表
-> 按 hn_id / canonical_url 去重
-> 按 subjects + event_trigger + 时间窗口做候选分组
-> 疑似同一事件时交给 EventClusterAgent 判断
-> 合并失败的 EvidenceCard 默认单独成事件
```

合并原则：

- 同一主体 + 同一触发点：合并。
- 同一主体 + 不同触发点：拆分。
- 社区讨论依附事实事件：合并到事实事件。
- 社区讨论本身成为焦点：单独形成热议型事件。

## 12. RankingAgent 排序策略

排序不完全交给 LLM 自由判断。

P1 采用：

```text
工程代码负责可计算分数
Agent 负责语义判断和理由
最终排序由工程代码控制
```

分数字段：

```text
heat_score：工程代码计算
impact_score：RankingAgent 判断
audience_value_score：RankingAgent 判断
freshness_score：工程代码计算
ranking_score：工程代码计算
ranking_reason：RankingAgent 生成
```

第一版公式：

```text
ranking_score =
  0.45 * heat_score
  + 0.25 * impact_score
  + 0.20 * audience_value_score
  + 0.10 * freshness_score
```

P1 暂不做：

- 跨源共振分。
- 增长速度分。
- 平台 baseline。
- 复杂爆发检测。

RankingAgent 输出示例：

```json
{
  "impact_score": 0.72,
  "audience_value_score": 0.86,
  "publish_decision": "publish",
  "brief_candidate": true,
  "ranking_reason": "该事件在 HN 有较高讨论度，并且与 AI 开发者工具使用方式有关，适合中文 AI 学习型用户了解。"
}
```

## 13. DetailWriterAgent 详情生成策略

每个 PublishedEvent 都必须有详情内容。

DetailWriterAgent 输出：

```text
事件标题
卡片摘要
详情正文
为什么值得关注
后续关注点
来源链接
```

字数建议：

```text
事件标题：20-35 个中文字符
卡片摘要：80-120 个中文字符
详情正文：500-800 个中文字符
为什么值得关注：100-200 个中文字符
后续关注点：2-3 条
来源链接：1-3 个
```

要求：

- 不直接搬运原文。
- 不把 HN 热议写成确定事实。
- 不编造来源、人物言论、日期或数字。
- 不用大量结构化字段堆砌给用户看。
- 中文表达要自然。

P1 可预留 `cover_image_url`，但允许为空。

## 14. BriefWriterAgent 简报生成策略

P1 每次运行生成 1 份 Brief。

Brief 只从已发布的 PublishedEvent 中选择 Top 3-5 条，不从未发布事件中重新发明内容。

输出结构：

```text
brief_title
brief_summary
brief_items[]
```

每个 BriefItem 包含：

```text
published_event_id
item_title
item_summary
item_reason
sort_order
```

字数建议：

```text
brief_summary：150-250 字
每条 brief_item：120-180 字
整份简报：600-1000 字
```

规则：

- 不强行凑满 5 条。
- 不写成纯链接列表。
- 每条 BriefItem 必须关联 PublishedEvent。
- 用户只看 Brief 也应能理解本轮重点。

## 15. QualityGateAgent 质量门禁

P1 不做真实性核验。Quality Gate 只检查生成质量和发布安全。

检查项：

- 是否有来源链接。
- 标题和正文是否一致。
- 是否把 HN 热议写成确定事实。
- 是否出现无来源数字、人物言论或结论。
- 是否疑似搬运原文。
- 正文是否空泛。
- 字数是否明显不合格。
- 是否缺少发布所需字段。

处理规则：

```text
QualityGate = publish
-> 发布

QualityGate = regenerate
-> 重新生成 1 次，再检查

QualityGate = manual_review / discard
-> P1 先不发布，只记录原因
```

如果内容第一次不通过，允许重生成一次。仍不通过则不发布，并写入 quality_gate_results 和运行报告。

## 16. PublishService 发布入库

P1 的发布定义不是发公众号、发推、发邮件，而是：

> 通过 Quality Gate 的事件被写入 PostgreSQL 的 published_events 表，状态为 published，可供后续 Next.js 前端查询展示。

PublishService 负责：

- 写入 published_events。
- 写入 content_artifacts。
- 写入 briefs。
- 写入 brief_items。
- 写入 quality_gate_results。
- 更新 pipeline_runs。

发布数量：

```text
最多 10 条 PublishedEvent
最多 5 条 BriefItem
```

如果高质量事件不足，可以少发，不强行凑数。

## 17. 幂等、复跑与 --force 策略

P1 必须支持重复运行时不制造重复数据。

EvidenceCard 复用规则：

```text
如果 hn_id + prompt_version 已经生成过 EvidenceCard
-> 默认跳过，直接复用
```

PublishedEvent 复用规则：

```text
如果 event_cluster 已经有 published_event
-> 默认不重复发布
```

内容版本规则：

```text
如果内容重新生成并通过 Quality Gate
-> 写入新的 content_artifact 版本
-> published_event 指向最新版本
```

强制重跑：

```bash
python scripts/run_hn_pipeline.py --days 7 --limit 100 --force
```

`--force` 允许重生成内容，但仍需遵守 Quality Gate。

## 18. PipelineRun 与 txt 运行报告

每次运行都应创建 PipelineRun，并输出本地 txt 报告。

报告输出位置：

```text
runtime/pipeline-reports/
```

示例：

```text
runtime/pipeline-reports/2026-06-08-1430-hn-event-pipeline.txt
```

报告内容：

- 运行时间。
- 运行参数。
- 抓取数量。
- EvidenceCard 数量。
- EventCluster 数量。
- PublishedEvent 数量。
- BriefItem 数量。
- Quality Gate 失败列表。
- LLM 调用概况。
- 错误堆栈摘要。

`runtime/` 是运行产物目录，不应把真实运行报告提交到 Git。后续可通过 `.gitignore` 忽略该目录，仅保留必要说明文件或 `.gitkeep`。

## 19. LLMClient 与环境变量

用户已有 Python LLMClient，P1 迁入 worker。

位置：

```text
apps/worker/worker/llm_client.py
```

LLMClient 只负责：

- provider 切换。
- model 切换。
- api_key / base_url。
- chat。
- stream_chat。

不负责：

- 业务 prompt。
- Agent schema。
- 工具调用。
- 数据库写入。
- 业务判断。

默认供应商：

```text
deepseek
```

`.env.example` 示例：

```env
LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat

DEEPSEEK_API_KEY=
DASHSCOPE_API_KEY=
OPENAI_API_KEY=
```

真实 API Key 只能放在本地 `.env` 或系统环境变量中，不得写入 Git。

## 20. Agent 结构化输出与失败修复

P1 所有 Agent 节点必须输出结构化 JSON。

工程代码使用 Pydantic 校验：

- JSON 格式。
- 必填字段。
- 枚举值。
- 分数范围。
- 字符串长度。

失败处理：

```text
第 1 次：正常生成
失败后第 2 次：repair prompt 修复
再失败第 3 次：repair prompt 再修复
仍失败：标记 failed，记录报告
```

即结构化输出失败后最多修复 2 次。

EvidenceAgent 第一版采用单条处理：

```text
1 条 HN story -> 1 次 EvidenceAgent 调用 -> 1 张 EvidenceCard
```

P1 不做批量 Agent 调用，后续如性能不足再优化 batch。

## 22. 验收标准

后端 P1 第一切片验收标准：

1. 能创建并连接 PostgreSQL。
2. Alembic 能完成核心表迁移。
3. 能运行 `python scripts/run_hn_pipeline.py --days 7 --limit 100`。
4. 能从 HN Algolia 获取数据。
5. 能抓取 story original_url 的完整可读正文并写入本地缓存。
6. 能生成 EvidenceCard。
7. 能生成 EventCluster。
8. 能生成最多 10 条 PublishedEvent。
9. 能生成 1 份 Brief 和最多 5 条 BriefItem。
10. 能生成 `runtime/pipeline-reports/*.txt`。
11. 重复运行不会制造重复 PublishedEvent。
12. Agent JSON 失败会最多修复 2 次，仍失败会记录原因。
13. Quality Gate 不通过的内容不会发布。

验收 SQL 示例：

```sql
select count(*) from evidence_cards;
select count(*) from event_clusters;
select count(*) from published_events;
select count(*) from briefs;
select count(*) from brief_items;
select * from pipeline_runs order by started_at desc limit 1;
```

- 实现结构化输出解析与 repair prompt。

验收代理：

- 根据本文档写验收清单。
- 跑本地命令。
- 查询 PostgreSQL。
- 检查 runtime 报告。

前端代理：

- 暂不进入本切片。
- 等后端能写入 published_events 和 briefs 后，再设计页面查询与展示。

运维代理：

- 暂不做部署。
- 后续补充本地环境准备、PostgreSQL 初始化、`.env.example` 和运行说明。

## 24. 后续切片与 P1.5 扩展

HN 切片跑通后，后续可按以下顺序推进：

1. 官方事实源切片：OpenAI、Anthropic、Google DeepMind、NVIDIA、GitHub Changelog。
2. GitHub / Hugging Face 切片：GitHub Releases / watchlist、HF Models / Papers。
3. 前端公开展示切片：首页、事件详情页、今日简报页。
4. 管理页切片：隐藏、恢复、重生成、调整优先级。
5. P1.5 热度增强：GitHub Trending、Reddit、YouTube、HF Trending。
6. P2：AI 追问助手、X 官方 API、更多 enrichment。

当前最重要的事情不是覆盖所有来源，而是先证明：

> AI World Radar 能从真实外网信号中生产出可追溯、可阅读、可入库的中文 AI 事件情报。
