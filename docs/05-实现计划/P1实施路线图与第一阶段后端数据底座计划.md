# P1 实施路线图与第一阶段后端数据底座计划 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将 AI World Radar P1 从旧的 HN 事件闭环骨架，重构为新版“事件档案生产与发布”架构的后端数据底座，为后续 LangGraph 多 Agent 工作流、前端首页、详情页和后台管理提供稳定契约。

**Architecture:** 本计划先给出 P1 总路线图，再只执行第一阶段。第一阶段只做 Python Worker 的数据模型、Pydantic schema、Alembic migration、服务层和可见测试证据，不接入真实 Agent，不开发前端，不引入 FastAPI。旧 HN collector、正文缓存和运行报告可以作为参考能力保留，旧 `EvidenceCard / EventCluster / ContentArtifact / Brief` 主链路不再作为新实现依据。

**Tech Stack:** Python 3.13, Pydantic 2.13.4, SQLAlchemy 2.0.41, Alembic 1.16.1, PostgreSQL, pytest 8.4.0。LangGraph 1.2.4 在第二阶段接入，本阶段只预留 schema 和服务边界。

---

## 1. 为什么先写这份计划

当前项目已经完成产品需求文档 v2.0、Agent 系统设计 v1.0、技术架构与数据模型设计 v1.0，以及旧后端 P1 工程骨架。

旧后端代码仍围绕以下旧概念运行：

```text
EvidenceCard
EventCluster
ContentArtifact
QualityGateResult
Brief / BriefItem
```

新版核心链路已经改为：

```text
SourceSignal
-> EventCandidate
-> EventDossier
-> ReviewResult
-> PublishedEvent
```

所以，下一步不能继续堆旧 HN 闭环，也不应该立刻写真实 Agent。正确顺序是先把新版数据底座立住，否则后面的 LangGraph、多 Agent、后台监控和前端页面都会接在错误对象上。

## 2. P1 总路线图

P1 不适合用一份计划一次性写完。它至少包含后端数据底座、LangGraph 工作流、三类 Agent、采集源、发布服务、Next.js 前台、后台管理和验收材料。为了避免计划过大，本阶段拆成以下路线：

| 阶段 | 名称 | 目标 | 交付物 |
| --- | --- | --- | --- |
| P1-1 | 后端数据底座与 Worker 重构 | 建立新版核心表、schema、服务层、测试证据 | 本计划执行范围 |
| P1-2 | LangGraph 工作流与三 Agent 最小闭环 | 用确定性 stub 跑通编辑部式工作流 | 下一份计划 |
| P1-3 | HN / GitHub 冷启动采集接入新版链路 | 把真实来源写入 `source_signals` | 后端采集计划 |
| P1-4 | 真实 LLM Agent 节点替换 | 值班编辑、研究写作、审稿发布逐个替换 | Agent 计划 |
| P1-5 | Next.js 首页、详情页、后台管理 | 展示已发布事件和后台运行记录 | 前端计划 |
| P1-6 | P1 验收与部署准备 | 形成可演示、可测试、可迁移的产品闭环 | 验收与运行计划 |

本计划只执行 P1-1。

## 3. P1-1 范围

必须完成：

- 新增 `pydantic` 依赖。
- 建立新版 Pydantic schema。
- 重构 SQLAlchemy ORM 为新版核心表。
- 用新的 Alembic migration 替换旧表结构。
- 建立服务层，保证后续 Agent 只能输出结构化建议，数据库写入由 service 执行。
- 建立运行记录和 Agent 运行记录的最小写入能力。
- 建立发布快照写入能力，验证同一个候选事件不会重复发布。
- 更新测试记录，让人能看到测了什么、用什么测、结果是什么。

本阶段不做：

- 不写真实 LLM prompt。
- 不接入 LangGraph。
- 不写三类 Agent 的真实逻辑。
- 不开发 Next.js 前端。
- 不开发后台 UI。
- 不引入 FastAPI。
- 不引入 Redis、队列、对象存储、向量库。
- 不做完整事实核验。
- 不继续维护旧 `BriefWriterAgent` 主链路。

from __future__ import annotations

from typing import Literal

from pydantic import BaseModel, ConfigDict, Field


class WorkerSchema(BaseModel):
    model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)


EventCandidateStatus = Literal[
    "new", "triaged", "merged", "drafting", "reviewing", "ready_to_publish",
    "published", "manual_review", "rejected", "failed", "hidden",
]
EventDossierStatus = Literal[
    "draft", "reviewing", "needs_revision", "approved",
    "manual_review", "rejected", "published_snapshot",
]
ReviewDecision = Literal["publish", "revise", "manual_review", "reject"]
RiskLevel = Literal["low", "medium", "high"]
PublishedEventStatus = Literal["published", "hidden", "archived"]
PipelineRunStatus = Literal["running", "succeeded", "partial_failed", "failed", "cancelled"]
AgentRunStatus = Literal["running", "succeeded", "failed"]


def score_field(description: str):
    return Field(default=0, ge=0, le=100, description=description)
