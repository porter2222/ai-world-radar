# P1-8 官方与开发者平台公开源扩展计划 Implementation Plan

> **For agentic workers:** 本计划必须按 TDD 执行。每个代码 task 先写失败测试，再实现，再运行测试通过，最后更新文档并单独提交。

**Goal:** 在 P1-7 `official_feeds` 基础上扩展更多低成本公开平台源，让 OpenAI、GitHub、Hugging Face、Google、AWS、PyTorch、Ollama 等平台的公开 feed 能稳定进入 `source_signals`。

**Architecture:** 本阶段继续沿用新版主链路 `SourceSignal -> EventCandidate -> EventDossier -> ReviewResult -> PublishedEvent`。新增或调整平台源只负责采集和映射，不直接调用 Agent、不直接发布、不新增数据库表。

**Tech Stack:** Python 3.13, httpx 0.28.1, BeautifulSoup 4.13.4, SQLAlchemy 2.0.41, pytest 8.4.0, PostgreSQL / SQLite smoke。

---

## 1. 当前阶段

P1-7 已完成 `github_repo_trends`、`official_feeds` / `official_news`，但官方 RSS live smoke 只真实覆盖了 `nvidia_news`。用户已经确认继续扩展更多平台，但当前阶段仍不做账号池、强反爬和复杂社媒大爬虫。

本阶段目标是扩展“公开、无需登录、低反爬风险”的平台源，优先选择 RSS / Atom / XML feed。HTML 列表页只保留为兜底或后续任务。

## 2. 阶段边界

### 负责

- 扩展 `OFFICIAL_SOURCE_PROFILES` 中的公开平台 feed。
- 优先把已有 HTML profile 中已发现 RSS 的源切换为 RSS。
- 为新增 profile 补测试，确保 source key、mode、entry_url 稳定。
- 使用 fixture mode 验证多 profile 组合采集仍只写 `sources` / `source_signals`。
- 运行 SQLite smoke；如网络允许，运行 PostgreSQL live source smoke。
- 更新验收记录、项目状态、文档索引、实现计划 README 和 HTML 阅读版。

### 不负责

- 不接 X / Reddit / YouTube / Facebook / TikTok。
- 不做账号池、代理池、验证码绕过、浏览器自动化反爬。
- 不做大规模网页深爬。
- 不做论文源、arXiv、Semantic Scholar。
- 不新增数据库表。
- 不改变默认 `AGENT_MODE=stub`。

## 5. 验收标准

- 新增 profile 可通过 `--official-profile` 调用。
- `openai_news`、`github_changelog` 优先使用 RSS，不再依赖 HTML 列表解析。
- fixture mode 多 profile 采集仍只写 `sources` / `source_signals`。
- 不创建 `pipeline_runs`、`event_candidates`、`event_dossiers`、`review_results` 或 `published_events`。
- 测试记录写清楚真实命令、真实输出、失败修复和未覆盖范围。

## 6. 风险与后续

- 官方 feed URL 可能变更，后续需要定期 source 运行观察。
- RSS 内容质量不等，后续仍需要 pipeline / Agent 做去重和筛选。
- 本阶段不解决调度、重试、告警和限流预算；这些属于后续 source operation 任务。
