1. 当前阶段

P1-4 不重写事件生产主链路，只替换 Agent 语义判断和内容生成能力。当前唯一主链路仍是：

SourceSignal
-> EventCandidate
-> EventDossier
-> ReviewResult
-> PublishedEvent

2. 阶段边界

负责

  • 真实 LLM Agent 结构化 JSON 调用基座。
  • 值班编辑、研究写作、审稿发布三个 LLM Agent。
  • --agent-mode stub|llm 脚本入口。
  • agent_runs provider、model、prompt_version、retry_count 和 trace 记录。
  • fake LLM 回归测试和可选真实 provider smoke。

不负责

  • 不恢复 EvidenceCard / EventCluster / Brief 旧链路。
  • 不让 Agent 直接写数据库、直接发布、直接隐藏或删除数据。
  • 不做自由上网、RAG、向量数据库、Redis、队列、对象存储或后台 UI。
  • 不把 API key 或 provider 私密配置写入代码或文档。

3. 设计口径

模块职责约束
LLMJsonAgent调用 LLMClient、提取 JSON、Pydantic 校验、repair retry最多 2 次 repair，失败抛出明确错误
Editor LLM Agent输出 EventCandidateDraft不发布、不写库、不删除信号
Writer LLM Agent输出 EventDossierDraft中文表达，来源支撑，卡片摘要受限
Reviewer LLM Agent输出 ReviewResultDraft只建议 publish / revise / manual_review / reject
Agent Factory创建 stub 或 LLM agent set默认 stub，真实调用显式开启

5. 验收标准

  • stub 模式仍是默认模式，且不需要 API key。
  • LLM 模式可以通过 --agent-mode llm 或 AGENT_MODE=llm 开启。
  • 三类 LLM Agent 均通过 fake LLM 测试输出既有 Pydantic schema。
  • LangGraph workflow 在注入 fake LLM agents 时能首跑生成 PublishedEvent。
  • agent_runs 能记录 provider、model、prompt_version、retry_count、output_json 和 trace_json。
  • 真实 provider smoke 已执行并记录真实结果；失败时记录原始错误。
