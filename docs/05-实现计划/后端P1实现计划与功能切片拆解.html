Backend P1 Plan

后端 P1 实现计划与功能切片拆解

本文档定义 AI World Radar 后端 P1 的第一条可交付功能切片：HN AI 事件生产闭环。它用于指导后续后端代理、Agent 代理和验收代理，把真实外网信号生产成可入库、可追溯、可阅读的中文 AI 事件情报。

版本：v0.1 更新时间：2026-06-08 阶段：后端 P1 第一功能切片确认版

1. 文档目的

本文档用于定义 AI World Radar 后端 P1 的第一条可交付功能切片：HN AI 事件生产闭环。

它不是后端全部功能的完整开发清单，也不是前端 UI 设计文档。它要解决的是后端 P1 先做什么、第一条纵向功能切片如何跑通、哪些部分由工程代码负责、哪些部分由 Agent 负责，以及本地如何运行、如何验收、如何给后续开发代理交接。

本文档确认后，后续后端代理、Agent 代理、验收代理应以本文档为第一实现依据。

2. 当前阶段与开发原则

当前项目已经完成产品宪章、PRD、信息源策略、Agent 系统设计 v0.2、技术架构与数据模型设计、采集策略调研和 QFT 晨报项目参考价值调研。

模块是实现单元，功能切片是交付单元。

P1 不按技术层长期堆模块，也不先把所有基础设施一次性搭完，而是从一个能真实跑通业务价值的纵向闭环开始。

  • HN 有公开可用的 Algolia API。
  • HN 有 points、comments、created_at 等热度字段。
  • HN 适合发现工程师圈正在讨论的 AI 基建、开发者工具、模型和争议。
  • HN 接入成本低，适合冷启动验证 Agent 情报生产链路。

3. 第一功能切片定义

HN AI 事件生产闭环。
HN Algolia API
-> HN story 采集
-> original_url 原文补全
-> EvidenceCard
-> EventCluster
-> Ranking
-> DetailWriter
-> BriefWriter
-> QualityGate
-> PublishService
-> PostgreSQL
-> PipelineRun 报告

这条切片完成后，系统应能通过本地 Python 文件触发一次完整运行，并在 PostgreSQL 中看到可发布事件和今日简报。

4. 本切片不做什么

  • 不做通用爬虫系统，不做完整 Source Adapter 框架。
  • 不做 X / Reddit / YouTube / Facebook 采集。
  • 不做 GitHub Trending / Hugging Face Trending。
  • 不做 FastAPI、Next.js 前端页面、定时任务和队列。
  • 不做登录、用户系统、订阅、收藏。
  • 不做真实性核验、embedding 聚类、自由上网型 Agent。
  • 不做人工前置审核后台。

这些内容可以进入后续 P1 切片、P1.5 或 P2。

5. 完成定义

能用一个本地 Python 文件跑完整条链路，并且 PostgreSQL 中有 EvidenceCard、EventCluster、PublishedEvent、Brief 和 BriefItem。
  • 能连接本机 PostgreSQL，Alembic 能创建 P1 核心表。
  • 能请求 HN Algolia API，抓取最近 7 天内最多 100 条 HN story。
  • 能根据 story 的 original_url 抓取完整可读正文并保存到本地缓存。
  • 能生成 EvidenceCard、EventCluster、最多 10 条 PublishedEvent。
  • 能为每条 PublishedEvent 生成详情内容。
  • 能从 PublishedEvent 中生成 1 份 Brief 和最多 5 条 BriefItem。
  • 能执行 Quality Gate，失败内容不发布。
  • 能输出终端报告和 txt 运行报告。
  • 重复运行不制造重复 PublishedEvent。

6. 运行入口

P1 第一版不先做 CLI 框架，直接运行 Python 文件。

cd apps/worker
python scripts/run_hn_pipeline.py --days 7 --limit 100
  • --days 7：抓取最近 7 天窗口。
  • --limit 100：最多保留 100 条 HN story。
  • --force：强制重生成已处理过的内容。

默认运行不带 --force 时，应复用已存在的 EvidenceCard、EventCluster 和 PublishedEvent，避免开发调试时重复消耗 LLM 调用。

8. HN Algolia 冷启动采集策略

时间窗口：最近 7 天
最多保留：100 条 story
采集对象：story
排序预处理：hn_heat_score = points + num_comments * 2

第一版使用固定 query profile，不让 Agent 自由发明关键词。

OpenAI
ChatGPT
Anthropic
Claude
Google DeepMind
Gemini
NVIDIA
Hugging Face
LLM
AI agent
AI coding
Cursor
GitHub Copilot
MCP
machine learning
每个关键词请求一次 HN Algolia API
-> 按 objectID / url 去重
-> 使用 created_at_i 限制 7 天窗口
-> 使用 hn_heat_score 做工程预排序
-> 截取前 100 条

HN 是热议源，不是事实源。points 和 comments 只表示社区热度，HN 评论只能作为讨论证据，不能写成事实。

9. original_url 原文补全与本地缓存

HN Algolia 提供热度和原文链接，但通常不提供原文正文。为了避免 DetailWriterAgent 根据标题脑补，P1 应补全 story 的 original_url。

HN story
-> original_url
-> 抓取可读正文
-> 保存完整正文到 runtime/source-pages/
-> EvidenceCard 保存正文摘要、缓存路径和 hash
  • 只抓 HN story 自带的 original_url，不递归抓取页面内其他链接。
  • 不登录，不绕过反爬，不执行复杂浏览器自动化。
  • 不把全文展示给用户，不把全文提交到 Git。
runtime/source-pages/2026-06-08/hn-123456-page.txt

数据库中不直接塞大段全文，只保存 original_url、page_title、page_excerpt、page_text_hash、page_cache_path、fetched_at 和 fetch_status。

10. EvidenceCard 生成策略

P1 不单独建 RawItem 表，也不单独建 CandidateEvent 表。一条 HN story 对应一张 EvidenceCard。

HN Story -> EvidenceCard
source_id
source_item_id / hn_id
original_title
original_url
author
published_at
points
num_comments
story_text
page_title
page_excerpt
page_text_hash
page_cache_path
claim_summary
normalized_title
subjects
event_type
category
heat_clues
candidate_score
processing_status

EvidenceAgent 负责判断是否 AI 相关，并提取主体、触发点、事件类型、展示分类、热度线索、中文用户价值理由和候选分数。

11. EventCluster 聚合策略

EventCluster 用于把多个 EvidenceCard 聚合成内部事件。P1 不做 embedding 聚类，第一版采用规则去重 + LLM 模糊合并。

EvidenceCard 列表
-> 按 hn_id / canonical_url 去重
-> 按 subjects + event_trigger + 时间窗口做候选分组
-> 疑似同一事件时交给 EventClusterAgent 判断
-> 合并失败的 EvidenceCard 默认单独成事件
  • 同一主体 + 同一触发点：合并。
  • 同一主体 + 不同触发点：拆分。
  • 社区讨论依附事实事件：合并到事实事件。
  • 社区讨论本身成为焦点：单独形成热议型事件。

12. RankingAgent 排序策略

排序不完全交给 LLM 自由判断。工程代码负责可计算分数，Agent 负责语义判断和理由，最终排序由工程代码控制。

heat_score：工程代码计算
impact_score：RankingAgent 判断
audience_value_score：RankingAgent 判断
freshness_score：工程代码计算
ranking_score：工程代码计算
ranking_reason：RankingAgent 生成
ranking_score =
  0.45 * heat_score
  + 0.25 * impact_score
  + 0.20 * audience_value_score
  + 0.10 * freshness_score

P1 暂不做跨源共振分、增长速度分、平台 baseline 和复杂爆发检测。

13. DetailWriterAgent 详情生成策略

每个 PublishedEvent 都必须有详情内容。DetailWriterAgent 输出事件标题、卡片摘要、详情正文、为什么值得关注、后续关注点和来源链接。

事件标题：20-35 个中文字符
卡片摘要：80-120 个中文字符
详情正文：500-800 个中文字符
为什么值得关注：100-200 个中文字符
后续关注点：2-3 条
来源链接：1-3 个
  • 不直接搬运原文。
  • 不把 HN 热议写成确定事实。
  • 不编造来源、人物言论、日期或数字。
  • 不用大量结构化字段堆砌给用户看。
  • 中文表达要自然。

P1 可预留 cover_image_url，但允许为空。

14. BriefWriterAgent 简报生成策略

P1 每次运行生成 1 份 Brief。Brief 只从已发布的 PublishedEvent 中选择 Top 3-5 条，不从未发布事件中重新发明内容。

brief_title
brief_summary
brief_items[]
published_event_id
item_title
item_summary
item_reason
sort_order
brief_summary：150-250 字
每条 brief_item：120-180 字
整份简报：600-1000 字

简报不强行凑满 5 条，不写成纯链接列表，每条 BriefItem 必须关联 PublishedEvent。用户只看 Brief 也应能理解本轮重点。

15. QualityGateAgent 质量门禁

P1 不做真实性核验。Quality Gate 只检查生成质量和发布安全。

  • 是否有来源链接。
  • 标题和正文是否一致。
  • 是否把 HN 热议写成确定事实。
  • 是否出现无来源数字、人物言论或结论。
  • 是否疑似搬运原文。
  • 正文是否空泛，字数是否明显不合格。
  • 是否缺少发布所需字段。
QualityGate = publish
-> 发布

QualityGate = regenerate
-> 重新生成 1 次，再检查

QualityGate = manual_review / discard
-> P1 先不发布，只记录原因

16. PublishService 发布入库

发布 = 通过 Quality Gate 的事件被写入 PostgreSQL 的 published_events 表，状态为 published，可供后续 Next.js 前端查询展示。

PublishService 负责写入 published_events、content_artifacts、briefs、brief_items、quality_gate_results，并更新 pipeline_runs。

最多 10 条 PublishedEvent
最多 5 条 BriefItem

如果高质量事件不足，可以少发，不强行凑数。

17. 幂等、复跑与 --force 策略

P1 必须支持重复运行时不制造重复数据。

如果 hn_id + prompt_version 已经生成过 EvidenceCard
-> 默认跳过，直接复用
如果 event_cluster 已经有 published_event
-> 默认不重复发布
如果内容重新生成并通过 Quality Gate
-> 写入新的 content_artifact 版本
-> published_event 指向最新版本
python scripts/run_hn_pipeline.py --days 7 --limit 100 --force

18. PipelineRun 与 txt 运行报告

每次运行都应创建 PipelineRun，并输出本地 txt 报告。

runtime/pipeline-reports/2026-06-08-1430-hn-event-pipeline.txt

报告包含运行时间、运行参数、抓取数量、EvidenceCard 数量、EventCluster 数量、PublishedEvent 数量、BriefItem 数量、Quality Gate 失败列表、LLM 调用概况和错误堆栈摘要。

runtime/ 是运行产物目录，不应把真实运行报告提交到 Git。后续可通过 .gitignore 忽略该目录，仅保留必要说明文件或 .gitkeep。

19. LLMClient 与环境变量

用户已有 Python LLMClient，P1 迁入 apps/worker/worker/llm_client.py。

LLMClient 只负责 provider 切换、model 切换、api_key / base_url、chat 和 stream_chat，不负责业务 prompt、Agent schema、工具调用、数据库写入和业务判断。

LLM_PROVIDER=deepseek
LLM_MODEL=deepseek-chat

DEEPSEEK_API_KEY=
DASHSCOPE_API_KEY=
OPENAI_API_KEY=

真实 API Key 只能放在本地 .env 或系统环境变量中，不得写入 Git。

20. Agent 结构化输出与失败修复

P1 所有 Agent 节点必须输出结构化 JSON，工程代码使用 Pydantic 校验 JSON 格式、必填字段、枚举值、分数范围和字符串长度。

第 1 次：正常生成
失败后第 2 次：repair prompt 修复
再失败第 3 次：repair prompt 再修复
仍失败：标记 failed，记录报告

结构化输出失败后最多修复 2 次。EvidenceAgent 第一版采用单条处理，不做批量 Agent 调用。

22. 验收标准

  1. 能创建并连接 PostgreSQL。
  2. Alembic 能完成核心表迁移。
  3. 能运行 python scripts/run_hn_pipeline.py --days 7 --limit 100。
  4. 能从 HN Algolia 获取数据。
  5. 能抓取 story original_url 的完整可读正文并写入本地缓存。
  6. 能生成 EvidenceCard、EventCluster、PublishedEvent、Brief、BriefItem。
  7. 能生成 runtime/pipeline-reports/*.txt。
  8. 重复运行不会制造重复 PublishedEvent。
  9. Agent JSON 失败会最多修复 2 次，仍失败会记录原因。
  10. Quality Gate 不通过的内容不会发布。
select count(*) from evidence_cards;
select count(*) from event_clusters;
select count(*) from published_events;
select count(*) from briefs;
select count(*) from brief_items;
select * from pipeline_runs order by started_at desc limit 1;

23. 后续开发代理分工建议

  • 后端代理：搭建 apps/worker、Python venv、依赖、SQLAlchemy、Alembic、HN collector、原文补全、数据库写入和报告生成。
  • Agent 代理：设计各 Agent prompt、Pydantic schema、结构化输出解析、repair prompt、Ranking、Detail、Brief 和 Quality Gate 节点。
  • 验收代理：跑本地命令、查询 PostgreSQL、检查 runtime 报告和重复运行幂等性。
  • 前端代理：暂不进入本切片，等待后端能写入 published_events 和 briefs。
  • 运维代理：后续补充本地环境准备、PostgreSQL 初始化、.env.example 和运行说明。

24. 后续切片与 P1.5 扩展

  1. 官方事实源切片：OpenAI、Anthropic、Google DeepMind、NVIDIA、GitHub Changelog。
  2. GitHub / Hugging Face 切片：GitHub Releases / watchlist、HF Models / Papers。
  3. 前端公开展示切片：首页、事件详情页、今日简报页。
  4. 管理页切片：隐藏、恢复、重生成、调整优先级。
  5. P1.5 热度增强：GitHub Trending、Reddit、YouTube、HF Trending。
  6. P2：AI 追问助手、X 官方 API、更多 enrichment。
当前最重要的事情不是覆盖所有来源，而是先证明 AI World Radar 能从真实外网信号中生产出可追溯、可阅读、可入库的中文 AI 事件情报。
