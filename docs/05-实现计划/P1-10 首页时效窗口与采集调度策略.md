# P1-10 首页时效窗口与采集调度策略

更新时间：2026-06-23

## 1. 策略结论

P1-10 的目标不是新增一套复杂日报系统，而是先修正首页展示口径：

> `PublishedEvent` 是长期事件档案库；首页事件流只展示最近发生、仍值得用户现在看的事件。

当前拍板策略：

- 采集频率：每天 3 次，建议为 09:00、14:00、21:00，时区按 Asia/Shanghai。
- 首页默认窗口：最近 48 小时内发布的 `PublishedEvent`。
- 首页展示数量：默认最多 20 条。
- 排序规则：沿用现有 `homepage_rank -> ranking_score -> published_at -> created_at`。
- 低量兜底：如果 48 小时内事件少于 8 条，首页可以向前放宽到最近 7 天补足内容，但最多仍按当前分页 limit 返回。
- 老事件不删除：仍保存在 `published_events` 中，详情页按 slug 可访问；历史归档、搜索和每日版面后置。

页面文案不强调“48 小时”，仍以“最新动态 / AI 雷达”表达，避免用户感知到机械窗口。

## 2. 项目现状分析

本策略基于当前代码现状设计，不新增数据库表，不改前端首版交互。

### 2.1 当前列表接口行为

当前 `GET /events` 由 FastAPI 层调用：

```text
apps/worker/worker/api/app.py
-> ProductQueryService.list_published_events(limit, offset, category)
```

当前 `ProductQueryService.list_published_events()` 的核心逻辑是：

```text
select(PublishedEvent)
where PublishedEvent.status == "published"
order by homepage_rank, ranking_score, published_at, created_at
limit / offset
```

这意味着它会读取数据库中所有已发布历史事件，只要状态是 `published`，就可能进入首页分页。

### 2.2 当前模型是否支持时效窗口

`PublishedEvent` 已具备本策略需要的字段：

- `published_at`
- `status`
- `homepage_rank`
- `ranking_score`
- `created_at`
- `slug`

因此首页 48 小时窗口可以直接基于 `published_at` 过滤，不需要新增 migration。

### 2.3 当前前端是否需要大改

当前 `apps/web` 首页读取：

```text
GET /events?limit=20&offset=0
```

前端只消费接口返回结果，不自己决定“新旧”。所以只要后端默认 `GET /events` 改为近期窗口，前端不需要结构性改动。

### 2.4 当前采集与生产能力

当前 P1-9 已支持：

```text
collect_source_signals.py --source-group daily_all
run_event_pipeline.py --select-top-candidates N
```

并且运行态默认 `AGENT_MODE=llm`，显式 `--agent-mode stub` 才进入离线稳定模式。

因此每天 3 次采集可以直接复用现有脚本链路，P1-10 不需要先引入队列、调度器或新服务。

## 3. 可行性分析

### 3.1 低风险点

- 不改数据库结构。
- 不改变事件详情接口，老事件 slug 仍可访问。
- 不改变采集链路。
- 不改变 Agent 工作流主链路。
- 不改变首页字段契约，只改变默认返回集合。
- 可通过少量 service/API 回归测试覆盖。

### 3.2 主要风险

| 风险 | 说明 | 处理策略 |
| --- | --- | --- |
| 首页内容过少 | 某些日期 48 小时内事件不足 | 加低量兜底，少于 8 条时扩到 7 天 |
| 旧热点被过早隐藏 | 跨天发酵事件可能仍值得看 | 48 小时比 24 小时更适合 P1 冷启动 |
| 分页语义变复杂 | 兜底窗口会影响后续页 | P1 只保证首页首屏和第一页体验，归档页后置 |
| 测试依赖当前时间 | `datetime.now()` 会导致测试不稳定 | service 层注入 `now` 或 helper 统一当前时间 |
| 前端误以为是全量历史 | 当前接口名仍为 `/events` | 文档明确 `/events` 是首页事件流，不是归档接口 |

### 3.3 为什么不是现在做 DailyEdition

`DailyEdition` 更产品化，但第一版会引入：

- 新表
- 新发布快照关系
- 新后台管理口径
- 新首页接口
- 新调度状态

当前更稳的路径是先用 `published_at` 做首页时效窗口，等真实运行一段时间后，再把“最新事件流”升级为“每日/每期版面”。

## 4. 产品策略

首页默认是“最近 AI 圈正在发生什么”，不是历史库。

P1 首页规则：

```text
default_window_hours = 48
min_recent_items = 8
backfill_window_days = 7
default_limit = 20
```

推荐行为：

1. 查询最近 48 小时内 `status=published` 的事件。
2. 如果 48 小时内数量达到 8 条，直接按现有排序返回。
3. 如果不足 8 条，则把窗口放宽到最近 7 天，再按同一排序返回。
4. 详情接口 `GET /events/{slug}` 不受窗口影响。
5. 后台接口 `/admin/*` 不受窗口影响。

## 5. 采集调度策略

P1 调度频率：

```text
09:00 daily_all
14:00 daily_all
21:00 daily_all
```

每次运行建议分两段：

```text
collect_source_signals.py --source-group daily_all
run_event_pipeline.py --select-top-candidates 5
```

运行态默认真实 LLM，不需要额外传 `--agent-mode llm`。离线验证或 fixture smoke 才显式传：

```text
--agent-mode stub
```

严格真实验收必须禁用 fallback，避免 stub 成功冒充真实 LLM 成功。

修改文件：

- `apps/worker/tests/test_product_query_service.py`

测试重点：

- 48 小时内事件会出现在 `list_published_events()`。
- 超过 48 小时的事件默认不出现在首页列表。
- 48 小时内事件不足 8 条时，可从 7 天内补足。
- `get_event_by_slug()` 仍可读取超过 48 小时的老事件。
- `category` 过滤与时效窗口同时生效。

### Task 2：实现 service 层时效窗口

修改文件：

- `apps/worker/worker/services/product_query_service.py`

建议实现：

- 新增私有 helper：计算 `recent_cutoff` 和 `backfill_cutoff`。
- `list_published_events()` 增加内部参数：
  - `recent_hours: int = 48`
  - `min_recent_items: int = 8`
  - `backfill_days: int = 7`
  - `now: datetime | None = None`
- 先 count 48 小时内数量。
- 如果数量少于 `min_recent_items`，使用 7 天窗口。
- 排序规则保持现有顺序。

### Task 3：补 FastAPI 接口回归

修改文件：

- `apps/worker/tests/test_product_api.py`

测试重点：

- `GET /events` 默认不返回明显过期事件。
- `GET /events/{slug}` 对老事件仍返回 200。
- 响应字段不新增内部评分字段。

P1 不建议在 public API 暴露 `recent_hours` 查询参数。前端默认消费首页事件流即可，归档查询后置。

## 8. 后续演进

P1-10 只做近期首页窗口。后续可以继续演进：

```text
P1.1 DailyEdition 表
```
