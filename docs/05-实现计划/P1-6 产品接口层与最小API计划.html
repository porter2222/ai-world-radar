P1-6 产品接口层与最小 API 计划

版本：v0.2 ｜ 更新日期：2026-06-22 ｜ 阶段：用户已确认采用 FastAPI

目标 为前台首页、事件详情页和后台审计页提供最小可用的只读产品接口层。

已确认 先实现 ProductQueryService 和响应 schema，再用 FastAPI 做很薄的 HTTP 适配层。

1. 当前上下文

后端 P1 已完成 P1-1 数据底座、P1-2 LangGraph 最小闭环、P1-3 HN / GitHub 采集接入、P1-4 真实 LLM Agent 和 P1-5 发布质量分支覆盖。

当前唯一新版主链路仍是：

SourceSignal
-> EventCandidate
-> EventDossier
-> ReviewResult
-> PublishedEvent

P1-6 不增强 Agent，不开发前端页面，只把已发布事件和后台审计数据通过 FastAPI 只读接口暴露出来。

2. 关键决策

用户已确认采用 ProductQueryService + FastAPI 薄适配层。业务查询规则先沉到 Python 服务层，FastAPI endpoint 只负责参数解析、调用 query service 和返回 JSON。

方案内容优点缺点结论
A只做 ProductQueryService最稳，依赖最少前端不能直接调用不采用，仅作为内部第一步
BProductQueryService + FastAPI 薄适配层前端可直接消费，查询逻辑仍可复用新增依赖，需要处理服务启动和联调已确认采用
CNext.js route handler 直接查库少一个 Python API 服务查询契约容易分叉不推荐

3. 阶段边界

负责

  • 已发布事件列表和详情查询。
  • pipeline run / agent run 审计查询。
  • manual_review 队列查询。
  • 查询响应 schema、测试和验收记录。

不负责

  • 不开发前端页面或后台管理 UI。
  • 不触发采集、Agent、LangGraph workflow 或发布。
  • 不提供写接口、权限系统或线上部署。
  • 不恢复 EvidenceCard / EventCluster / Brief 旧链路。

5. 响应契约

事件列表返回 id、slug、标题、卡片摘要、详情摘要、分类、热度标签、封面图、首页排序位和发布时间。前台列表不返回 ranking_score、heat_score、importance_score 等复杂分数；这些分数只用于内部排序和后台审计。

事件详情额外返回 detail_body、why_it_matters、follow_up_points 和 source_refs，并且必须通过 published_events.dossier_id 联查发布时对应的 event_dossiers 版本。

GET /admin/review-queue 只返回当前仍处于 manual_review 的事件；同一 candidate 存在多版 dossier / review 时，只返回最新 dossier 和最新 review。

Agent run 审计默认不暴露完整 llm_raw_text，只暴露耗时、重试次数、模型信息、token usage、状态和错误摘要。

7. 验收标准

  • 默认不会触发采集、Agent 或发布。
  • 事件列表只返回 published 状态。
  • 事件详情按 slug 查询，缺失时行为明确。
  • 后台审计能查看 pipeline run 和 agent run 摘要。
  • manual_review 队列可查询。
  • worker 全量 pytest 通过，PostgreSQL 可读取已发布事件。

8. 执行结论

用户已确认 P1-6 直接采用 FastAPI。下一步从 Task 1 开始，先按测试先行方式实现 ProductQueryService 和响应 schema，再进入 Task 2 实现 FastAPI 薄适配层。

FastAPI 在 P1-6 只提供只读产品接口，不触发采集、Agent、发布或重跑。管理端接口当前只服务本地开发和后台页面联调，公开部署前必须补鉴权。
