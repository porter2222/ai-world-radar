QFT 晨报项目参考价值调研

分析本地项目 D:\qft-agent-project\qft-morning-brief 对 AI World Radar 采集系统、证据系统、Agent 生产链路和质量门禁的参考价值。

本地项目调研 证据治理 Agent 自动生产 Quality Gate

1. 调研目的

AI World Radar 的目标不是普通 AI 新闻聚合，而是把多源碎片信息加工成可回源、可解释、可自动发布的 AI 事件情报。QFT 晨报项目虽然业务领域不同，但已经实现了从数据包、候选发现、证据构造、结构化生成、质量门禁到审计落库的完整闭环。

业务源数据
  -> 查询工具 / Source Packet
  -> 候选问题发现
  -> 候选准入与排序
  -> Issue Evidence Pack
  -> LLM 结构化生成
  -> Quality Gate
  -> PG 审计落库
  -> 报告展示 / 追问
本项目的价值不是“怎么采集外网新闻”，而是“怎么让 Agent 生成的内容仍然有证据、有审计、有门禁”。

2. 项目概况

项目路径：D:\qft-agent-project\qft-morning-brief

项目定位：AI 驱动的全房通经营晨报系统，为房屋租赁企业老板每日生成结构化经营简报。

技术栈Next.js、React、TypeScript、Zod、Drizzle ORM。
数据源MySQL 业务库作为只读源，PostgreSQL 作为晨报系统审计库。
AI RuntimeOpenAI Responses API 与 Agents SDK 双路径。

项目协作文档中的架构描述是：MySQL（业务数据）-> 查询工具（预取）-> LLM structured output -> PG 落库（审计）。

3. 总体结论

高价值参考点

  • source_packets 原始证据包。
  • tool_call_records 工具调用审计。
  • generation_runs Agent 运行审计。
  • evidenceRef 证据引用结构。
  • CandidateIssue 候选项 schema。
  • sourceAdmission 证据准入机制。
  • 规则发现 + LLM 发现的 compare 模式。
  • 候选晋级、排序 gate 和 Quality Gate。
  • deterministic fallback 与基于 QC 通过报告的追问链路。

不应照搬

不要迁移房屋租赁 SQL、全房通业务指标、四大抓手、老板晨报话术、公司隔离模型、财务对账规则和现有展示层。AI World Radar 应借鉴工程模式，而不是业务语义。

4. 与 AI World Radar 的关系

AI World Radar 的 P1 主链路已经定义为 Source Registry、Source Collector、Raw Item Store、Evidence Card、Candidate Router、Event Cluster、Event Ranking、Content Generation 和 Quality Gate。QFT 项目可映射为：

fetchMorningBriefPacketBundle
  -> source_packets
  -> runDiscoveryAgent
  -> runInvestigationPreflight
  -> runSelectionAgent
  -> buildIssueEvidencePack
  -> runMorningAgentWithTelemetry
  -> runQualityGate
  -> persistMorningBriefSuccess / Failure

两者核心相似点是：都需要从原始材料中提取值得展示的事项，并且用证据结构约束 Agent 输出。核心差异是：QFT 面向结构化业务数据，AI World Radar 面向公开网页、RSS、API、社群热度和新闻线索。

5. 关键源码参考

主编排链路

lib/server/services/morning-brief.ts 中的 runSingleCustomerMorningBrief 负责生成 runId、采集 packet、发现候选、选择候选、构造证据包、生成报告、执行质检并持久化结果。AI World Radar 应设计类似的 runRadarProduction 主链路。

Source Packet 与审计

morning-brief-packets.ts、morning-brief-persistence.ts 和 db/schema.ts 展示了 source packet、tool call 和 generation run 的审计方式。AI World Radar P1 应保存 raw snapshot、adapter version、fetch status、error type 和 source health。

EvidenceRef

lib/contracts/report.schema.ts 中的 evidenceRefSchema 是最值得借鉴的结构。AI World Radar 应扩展为包含 sourceId、sourceLevel、rawSnapshotId、rawItemId、url、quote、capturedAt 和 confidence 的证据引用。

CandidateEvent

candidate-issue.schema.ts 可改造为 CandidateEvent：包含事件类型、主体、触发点、时间窗口、证据引用、来源准入、热度信号、数据状态和是否可晋级。

Source Admission

evidence-admission.ts 将证据分成 hard、supporting、blocked。AI World Radar 应改造成事实源、热度源、线索源的准入规则，避免把聚合站线索写成确定事实。

Discovery / Selection

discovery-agent.ts、selection-agent.ts、promote-issues.ts 体现了“候选发现”和“候选选择”分层。P1 可先用规则 + 轻量 LLM，P1.5 再引入 rules / LLM compare。

EventEvidenceBundle

issue-evidence-pack.ts 说明写作 Agent 不应直接吃所有 raw body，而应读取压缩后的证据包。AI World Radar 应构造 EventEvidenceBundle，区分 primary evidence、supporting evidence 和 heat evidence。

Quality Gate

quality-gate.ts 会验证 schema、禁用词、证据引用、工具名、字段路径和 raw value 是否匹配真实响应。AI World Radar P1 必须有确定性 Quality Gate，至少检查证据存在、来源等级匹配、热议不写成事实、日期一致、非直接搬运和 raw snapshot 存在。

Runtime 与追问

agent-runtime.ts、openai-runtime.ts 和 morning-agent.ts 展示了 Responses API、Agents SDK、fallback 和 telemetry。followup.ts 与 app/api/chat/route.ts 可作为 P2 事件追问参考，但不应进入 P1。

7. 推荐架构

P1 不需要完整复制 QFT 的多阶段 Agent。建议最小闭环如下：

Source Registry
  -> Collector Run
  -> Raw Snapshot
  -> Raw Item
  -> Evidence Card
  -> Candidate Event
  -> Event Cluster
  -> Event Ranking
  -> Content Artifact
  -> Quality Gate
  -> Publish / Hold

建议最小数据表：sources、source_adapters、collector_runs、raw_snapshots、raw_items、evidence_cards、candidate_events、event_clusters、content_artifacts、quality_gate_results、generation_runs。

写作 Agent 输入必须是 Event Cluster 和 EventEvidenceBundle，不应直接拿整页 HTML、竞品长文章正文或未标注来源等级的 raw text。

8. 分阶段建议

P1

借鉴 runId、Raw Snapshot、EvidenceRef、CandidateEvent、SourceAdmission、deterministic fallback、generation run 审计和最小 Quality Gate。暂不借鉴复杂 compare、rubric/reflexion、追问助手和多轮工具 Agent。

P1.5

引入 rules / LLM compare、候选池、候选资格卡、Event Ranking gate、source health、stage packets、回放测试和失败兜底。适合加入 GitHub Trending、HF Trending、HN 热度、YouTube 已知频道或 Reddit 指定社区。

P2

参考 QFT 的 SSE 追问链路，构建事件级 AI follow-up。追问只能围绕当前事件，只基于 EventEvidenceBundle 和允许工具回答，并且回答必须带来源。

10. 风险和限制

风险说明处理建议
领域错配QFT 是结构化业务数据系统，AI World Radar 是公开互联网信息系统。只借鉴证据和审计模式，不照搬业务规则。
复杂度风险QFT 已有较多 gate、rubric、reflexion 和回归测试。P1 只保留 EvidenceRef、SourceAdmission、Quality Gate 和 run audit。
LLM 过度参与如果让 LLM 替代采集、去重和来源准入，会降低可信度。关键判断工程化，LLM 只做语义层辅助。
追问提前化QFT 已有追问，但 AI World Radar P1 不包含 AI 追问助手。追问放到 P2，等事件详情和证据层稳定后再做。

11. 最终建议

AI World Radar 后续开发可以把 QFT 晨报项目当成三个层面的参考：证据层、候选层、质量层。最推荐优先落地的是 Raw Snapshot -> Raw Item -> Evidence Card -> Candidate Event -> EventEvidenceBundle -> Content Artifact -> Quality Gate Result。

一句话总结：QFT 项目对 AI World Radar 的最大价值，不是“怎么采集数据”，而是“怎么让 Agent 生成的内容仍然有证据、有审计、有门禁”。这正是 AI World Radar 从新闻聚合站变成 AI 情报雷达的关键。
