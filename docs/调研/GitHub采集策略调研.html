Research Report

GitHub 采集策略调研

本报告通过开源仓库源码研究 AI World Radar 后续采集系统的可行路线：P1 以 RSS/API/稳定页面为主，P1.5 再接入 Reddit、YouTube、GitHub Trending 等热度信号，X 与 Facebook 的非官方方案默认高风险。

输出：Markdown + HTML + 仓库清单 范围：源码策略调研 不实际爬取社媒数据

1. 调研目的

本次调研不是立刻引入某个 scraper，也不是实际抓取 X、Reddit、YouTube 或 Facebook，而是从 GitHub 开源项目源码中提炼采集策略、限流和缓存设计、失败恢复、合规风险，以及哪些方案适合 AI World Radar 的 P1、P1.5、P2。

重点覆盖两类源：高热平台，包括 X / Twitter、Reddit、YouTube、Facebook；P1 当前信息源，包括 OpenAI News、Anthropic News、DeepMind Blog、NVIDIA Blog/RSS、GitHub Changelog、GitHub Trending、Hacker News、Hugging Face、TLDR AI、AIBase 和 AI 新闻/RSS/Newsletter 聚合源。

2. 与 AI World Radar 的关系

AI World Radar 的核心内容单元是事件卡，而不是新闻链接。因此采集层必须服务于可回源、可聚合、可解释的内部链路。

Source Registry
  -> Raw Snapshot
  -> Raw Item
  -> Evidence Card
  -> Event Cluster
  -> Published Event

采集层要保存原始响应、URL、headers、状态码、ETag、Last-Modified、抓取时间、adapter version 和 content hash。LLM 可以做语义判断，但不能替代事实证据。

3. 总体结论

P1 主路径：自建轻量 Source Registry + Adapter Manifest + Raw Snapshot Store + Normalizer。优先官方 RSS、官方 API、稳定公开页面。RSSHub、Miniflux、Huginn、changedetection.io 更适合作为设计参考，而不是整体套用。
P1.5 增强：Reddit 官方 API、YouTube Data API、GitHub Trending、Hugging Face Papers/Trending、Browser worker 和 LLM fallback 可作为热度或补充证据，但必须可降级。
不进 P1：X 非官方 GraphQL/guest token/cookie/账号池，Facebook mobile/mbasic HTML + cookie/login，YouTube 评论广域抓取，MediaCrawler 类中文社媒登录态采集，代理池和绕登录方案。

4. 高热平台采集策略

4.1 X / Twitter

源码显示两条路线：官方 API 路线以 `tweepy`、`twarc` 为代表，依赖 bearer/OAuth token，支持 pagination、rate limit 和归档；非官方路线以 `twscrape`、`snscrape`、`nitter`、`ntscraper` 为代表，常见 guest token、私有 GraphQL、cookie、账号池、session pool、proxy 或 Nitter 实例池。

可借鉴的是 endpoint budget、lock_until、next_available_at、source health、dedup、hydration、compliance jobs。不可借鉴到 P1 的是账号池、cookie 登录、绕过限制或公开 Nitter 实例轮询。

4.2 Reddit

Reddit 最稳路线是官方 API。`praw` 和 `prawcore` 处理 listing cursor、`after` 分页、`x-ratelimit-*` headers、429 和 comments tree；`asyncpraw` 可作为异步栈备选；`redditwarp` 的 token bucket 可借鉴；`redlib`、Pushshift/old Reddit 类方案不建议作为主数据源。

适合 P1.5 的热度信号包括 subreddit、标题、URL、score、comments_count、upvote ratio、发布时间和 top comments。评论只能作为热议证据，不能直接当事实。

4.3 YouTube

官方路线是 YouTube Data API，可通过 `google-api-python-client` 和 `youtube/api-samples` 获取 search、videos statistics、channels、commentThreads、captions 等，缺点是 API key/OAuth 和 quota。

非官方增强路线包括 `yt-dlp`、`YouTube.js`、`youtube-transcript-api`、`youtube-comment-downloader`、`pytube`。这些能覆盖 view_count、like_count、comment_count、upload_date、subtitles、chapters、comments continuation 等，但更适合 P1.5 精选视频补证，不适合 P1 广域硬依赖。

4.4 Facebook

官方路线是 Graph API：`facebook-sdk`、`pyfacebook`、`facebook-python-business-sdk` 都需要 token、App、权限和合规授权。网页路线如 `facebook-scraper` 依赖 mobile/mbasic HTML、cookie、credentials、checkpoint/2FA，风险高且稳定性差。

Facebook 不建议进入 P1；P2 若确有需求，应只考虑授权 Graph API 或人工验证型小规模实验。

5. P1 稳定信息源采集策略

5.1 官方博客 / RSS

`feedparser` 适合作 P1 feed parser；`RSSHub` 的 route registry 值得参考；`Miniflux` 证明 ETag、Last-Modified、entry hash、worker pool 是成熟 RSS 系统的基本功；`Huginn` 证明 source health、memory checkpoint 和 event flow 对多源采集有价值。

5.2 GitHub

GitHub Changelog、Releases、Issues、PRs 应优先走官方 API/RSS。`agents-radar/src/github.ts` 展示了 token、分页、since 时间窗口和 digest 输出。GitHub Trending 没有官方 API，`github-trending-api`、`go-trending`、`github-trending-repos` 都是 HTML 解析，应每日快照、低频运行并配 selector test。

5.3 Hacker News

HN 是 P1 友好源。可用官方 Firebase API，也可用 Algolia HN Search API 做关键词时间窗搜索。`node-hnapi` 提供 memory/Redis cache 和 news/newest/ask/show/jobs/item 封装。

5.4 Hugging Face

`huggingface_hub` 官方 API 支持 models、datasets、spaces 的 `downloads`、`likes`、`last_modified`、`trending_score` 等字段。Papers/Trending 若 API 覆盖不足，可参考 RSSHub/page adapter，但必须保留 raw snapshot。

5.5 Newsletter / AI 新闻源

TLDR AI、AIBase、LearnPrompt、TrendRadar 等适合作候选线索。聚合源只能作为辅助，Evidence Card 应尽量补回官方原始链接。

5.6 中文社媒采集参考：MediaCrawler

NanmiCoder/MediaCrawler 是 Python 多平台中文社媒采集项目，GitHub API 显示截至 2026-06-03 约 50,622 stars，最近 push 为 2026-05-29。它支持小红书、抖音、快手、B 站、微博、贴吧、知乎；当前不支持 X/Twitter、Facebook、Reddit。

关键源码包括 main.py、config/base_config.py、base/base_crawler.py、media_platform/*/core.py、media_platform/*/client.py、media_platform/*/login.py、proxy/proxy_ip_pool.py。源码结构可借鉴 adapter factory、crawler/login/client/store 分层、search/detail/creator 模式和风险配置显式化。

但它依赖 Playwright、登录态缓存、二维码/手机号/cookie 登录、平台签名、可选代理池和风控处理；许可证为非商业学习使用，明确不应用于大规模爬虫或商业用途。因此它只能作为 P2 中文社媒热度实验的源码策略参考，不应进入 P1，也不应作为 AI World Radar 的生产依赖。

  mapper: github_trending_v1

  selector_test: true
  circuit_breaker: true

  legal: medium
  stability: medium

8.3 Raw Item / Evidence Card

Raw Item 记录抓取事实：source、URL、标题、发布时间、抓取时间、headers/hash、snapshot、adapter version、parse status。Evidence Card 记录证据判断：claim、source_level、quote/excerpt、实体、时间、热度信号、confidence、raw_item_ids 和推荐动作。

8.4 Source Health

每个 source 维护 last_success_at、consecutive_failures、success_rate、parse_success_rate、latency、freshness_lag、block_rate、last_error_class、health_status。red 后暂停 browser/LLM 高成本 fallback，只保留低频健康探测。

9. P1 / P1.5 / P2 分阶段建议

P1

RSS/API/page fetcher、Raw Snapshot、Raw Item、Evidence Card、source health、dedupe。信息源为 OpenAI、Anthropic、DeepMind、NVIDIA、GitHub、HN、HF、TLDR AI、AIBase。暂不接入 MediaCrawler 或类似中文社媒登录态采集。

P1.5

Reddit 官方 API、YouTube Data API、GitHub Trending、HF Papers/Trending、Browser worker、LLM fallback、candidate event pool 和 source health dashboard。MediaCrawler 只作为 adapter 与风险字段设计参考，不作为运行时依赖。

P2

X 官方 API 或授权数据源、Facebook Graph API 授权场景、中文社媒热度实验室、AI follow-up assistant、个性化、冲突检测、可视化 source/workflow 管理。若参考 MediaCrawler，必须授权、小规模、隔离环境。

11. 后续待验证问题

  1. OpenAI、Anthropic、DeepMind、NVIDIA 当前是否都有稳定 RSS；没有 RSS 的页面 selector 如何设计。
  2. AIBase 是否有稳定 RSS 或公开列表 API；若只有 HTML，selector test 如何覆盖。
  3. TLDR AI 是否可通过公开 RSS、newsletter archive 或邮件入口稳定采集。
  4. Hugging Face Daily Papers 是否完全可由官方 API 覆盖，还是需要 RSSHub/page adapter。
  5. GitHub Trending 是否进入 P1 第二批还是 P1.5；如果进入 P1，是否接受每日一次快照。
  6. Raw Snapshot 存储用数据库、对象存储还是本地文件；保留周期如何定。
  7. 未来 X/Reddit/YouTube 热度信号如何与官方源事实事件聚合，避免热度压过可信度。
  8. 若未来引入中文社媒热度信号，哪些平台有官方/授权路径；MediaCrawler 只能作为架构参考，不能作为默认实现。
