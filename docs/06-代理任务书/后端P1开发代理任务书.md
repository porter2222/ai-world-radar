# 后端 P1 开发代理任务书

版本：v0.1

更新时间：2026-06-09

适用对象：后端开发代理

## 1. 任务书目的

本文档是给后端开发代理使用的可执行任务书。

后端开发代理拿到本文档后，应能在不重新讨论产品方向的前提下，完成后端 P1 第一轮工程骨架开发，并清楚知道：

- 需要先读哪些文档。
- 允许修改哪些目录。
- 禁止做哪些事情。
- 每个阶段交付什么。
- 每个阶段如何验证。
- 测试证据如何展示给用户。
- 最终如何提交和交接。

## 2. 后端代理角色定位

后端代理的角色是：

> 负责 AI World Radar 后端 P1 工程底座和 HN 事件生产闭环的工程实现者。

后端代理负责把管道搭出来，但不负责设计最终 Agent 智能逻辑。

后端代理负责：

- `apps/worker` 工程结构。
- Python 依赖与环境示例。
- PostgreSQL / SQLAlchemy / Alembic。
- HN Algolia collector。
- HN story 去重、热度预排序。
- `original_url` 原文抓取与本地缓存。
- PipelineRun 记录。
- Agent stub。
- pipeline 串联。
- txt 运行报告。
- 本地运行入口。
- 后端第一轮测试。

后端代理不负责：

- 真实 Agent prompt 调优。
- EvidenceAgent / RankingAgent / DetailWriterAgent 等最终智能逻辑。
- 前端页面。
- FastAPI。
- 部署。
- 产品定位修改。
- 真实性核验系统。

## 4. 本次目标

本次目标不是完成整个 AI World Radar 后端，而是完成第一轮工程骨架：

```text
HN Algolia API
-> HN story 采集
-> original_url 原文缓存
-> Agent stubs
-> Pipeline 串联
-> PostgreSQL 写入
-> PipelineRun / txt 运行报告
```

第一轮可以使用确定性 Agent stub，不调用真实 LLM。

## 5. 职责边界

后端代理只做工程底座和管道，不做真实 Agent 智能逻辑。

后端代理应提供 Agent 节点接口和 stub，使 pipeline 能跑通。后续 Agent 代理会把 stub 替换为真实 prompt、schema 和 LLM 调用。

## 6. 允许修改目录

后端代理允许修改：

```text
apps/worker/**
runtime/.gitkeep
runtime/README.md
.gitignore
.env.example
pyproject.toml
requirements.txt
docs/07-验收与运行/**
```

如果项目最终选择 `pyproject.toml`，可以不创建 `requirements.txt`；如果选择 `requirements.txt`，可以暂不创建 `pyproject.toml`。第一轮优先简单可运行。

## 7. 禁止事项

后端代理默认不允许修改：

```text
apps/web/**
docs/01-产品定义/**
docs/02-信息源策略/**
docs/04-系统设计/**
docs/05-实现计划/**
```

除非用户明确授权。

后端代理还必须遵守：

- 不得写入真实 API Key。
- 不得提交 `.env`。
- 不得提交 runtime 真实运行产物。
- 不得引入 FastAPI。
- 不得做前端页面。
- 不得改产品定位。
- 不得做 X / Reddit / YouTube / Facebook 采集。
- 不得做自由上网型 Agent。

## 8. 第一轮交付范围

第一轮后端代理应交付：

```text
apps/worker 目录建立
Python 依赖文件建立
.env.example 建立
SQLAlchemy / Alembic 初始化
P1 核心表 migration 建立
HN Algolia collector 可运行
original_url 正文抓取与 runtime/source-pages 缓存可运行
pipeline_reports txt 报告可生成
scripts/run_hn_pipeline.py 能跑
Agent 节点通过确定性 stub 占位
pytest 测试通过
```

第一轮不要求真实 LLM Agent 完成。

## 9. Agent Stub 口径

后端代理应实现确定性 stub，不调用 LLM。

建议 stub：

```text
EvidenceAgentStub
-> 根据 HN title / page_title 返回简单 EvidenceCard JSON

EventClusterAgentStub
-> 默认 1 个 EvidenceCard 生成 1 个 EventCluster

RankingAgentStub
-> 使用 hn_heat_score 生成简单 ranking_score

DetailWriterAgentStub
-> 用 title + excerpt 拼一个临时详情

BriefWriterAgentStub
-> 从 top PublishedEvent 拼一个临时 brief

QualityGateAgentStub
-> 检查字段齐全就 publish
```

stub 必须输出结构化数据，方便后续 Agent 代理替换。

## 10. 测试要求

后端代理必须写测试。测试不是可选项。

第一轮测试范围：

```text
HN Algolia 返回结果解析测试
HN story 去重测试
hn_heat_score 计算测试
original_url 缓存路径生成测试
pipeline report txt 生成测试
Agent stub 输出结构测试
```

推荐测试目录：

```text
apps/worker/
  tests/
    fixtures/
      hn_algolia_response.json
      sample_page.html

    test_hn_algolia_collector.py
    test_hn_scoring.py
    test_dedupe.py
    test_page_cache.py
    test_pipeline_report_writer.py
    test_agent_stubs.py
```

推荐测试命令：

```bash
cd apps/worker
pytest
```

## 11. 真实网络测试策略

真实网络测试不禁止，但必须和稳定单元测试分开。

要求：

- 普通单元测试应优先使用 fixture / mock，保证稳定。
- 真实网络测试必须使用 `pytest` marker 标记为 `network`。
- 真实网络测试可以请求 HN Algolia API。
- 真实网页抓取测试可以少量存在，但必须控制数量和超时。
- 真实 LLM 调用不属于后端代理第一轮测试范围。

推荐命令：

```bash
pytest
pytest -m network
```

后端代理最终报告中必须说明是否运行过 network 测试。

## 12. 测试证据与可见性要求

后端代理不能只说“测试通过了”。

后端代理必须让用户看到：

- 测了什么。
- 用什么测试的。
- 测试数据是什么。
- 执行了什么命令。
- 测试结果是什么。
- 哪些没有测试。
- 为什么没有测试。

每个阶段报告和最终报告都必须包含测试证据表。

示例：

| 测试项 | 测试文件/命令 | 测试数据 | 验证什么 | 结果 |
|---|---|---|---|---|
| HN 解析测试 | `pytest tests/test_hn_algolia_collector.py` | `fixtures/hn_algolia_response.json` | 能否解析 HN 返回字段 | 通过 |
| 热度分测试 | `pytest tests/test_hn_scoring.py` | points/comments 样例 | `points + comments * 2` 是否正确 | 通过 |
| 去重测试 | `pytest tests/test_dedupe.py` | 重复 objectID/url 样例 | 是否去掉重复 story | 通过 |
| 原文缓存测试 | `pytest tests/test_page_cache.py` | `fixtures/sample_page.html` | 是否生成缓存路径和 hash | 通过 |
| 运行报告测试 | `pytest tests/test_pipeline_report_writer.py` | fake pipeline result | 是否生成 txt 报告内容 | 通过 |

最终报告还必须包含：

```text
测试命令
测试数量
通过数量
失败数量
跳过数量
失败原因
未覆盖范围
```

## 14. 阶段化开发步骤

后端代理必须按阶段推进，不要一次性写一大坨。

### 阶段 1：工程脚手架

目标：

- 建立 `apps/worker`。
- 建立 Python 依赖文件。
- 建立 `.env.example`。
- 建立基础配置读取。
- 建立 runtime 目录说明或 `.gitkeep`。

验收：

- 能进入 `apps/worker`。
- 能安装依赖。
- `.env.example` 不包含真实 key。

### 阶段 2：数据库与迁移

目标：

- SQLAlchemy session。
- P1 核心 models。
- Alembic 初始化。
- 第一版 migration。

验收：

- Alembic migration 能执行。
- 核心表能创建。

### 阶段 3：HN collector

目标：

- HN Algolia 请求封装。
- query profile。
- 7 天时间窗。
- limit 100。
- scoring。
- dedupe。

验收：

- fixture 测试通过。
- 可选 network 测试可跑。

### 阶段 4：原文抓取与缓存

目标：

- 抓取 original_url 可读正文。
- 写入 `runtime/source-pages/`。
- 生成 hash、excerpt、cache path。
- 控制超时和失败记录。

验收：

- fixture HTML 测试通过。
- 失败 URL 不会导致整个 pipeline 崩溃。

### 阶段 5：Agent stubs 与 pipeline 串联

目标：

- 实现确定性 Agent stubs。
- 串起 HN pipeline。
- 写入数据库。
- 实现幂等。

验收：

- 本地命令可运行。
- 重复运行不制造重复 PublishedEvent。

### 阶段 6：运行报告与最终验收

目标：

- 生成终端报告。
- 生成 `runtime/pipeline-reports/*.txt`。
- 更新 `docs/07-验收与运行/后端P1测试记录.md`。
- 运行测试。

验收：

- `pytest` 通过。
- 本地 pipeline 命令运行成功或给出明确失败原因。
- 最终报告列出测试证据。

## 15. 每阶段检查点报告格式

每个阶段结束时，后端代理应输出：

```text
阶段名称：
本阶段完成内容：
修改文件：
执行命令：
测试证据：
当前风险：
下一阶段计划：
```

禁止只输出“完成了”。

## 16. 最终验收标准

最终验收至少包括：

1. `apps/worker` 工程结构存在。
2. 依赖文件存在。
3. `.env.example` 存在且无真实 key。
4. Alembic migration 存在。
5. HN collector 有测试。
6. scoring / dedupe 有测试。
7. original_url 缓存逻辑有测试。
8. Agent stubs 有测试。
9. pipeline report 有测试。
10. `scripts/run_hn_pipeline.py` 存在。
11. `runtime/pipeline-reports/` 策略存在。
12. `docs/07-验收与运行/后端P1测试记录.md` 存在。
13. 最终回答列出测试证据。

## 17. 提交与推送策略

后端代理采用：

```text
阶段检查点报告 + 最终一次 commit/push
```

不要求每个小阶段都提交。

最终提交前必须：

- 跑测试。
- 报告测试结果。
- 检查 `git status`。
- 确认没有提交 `.env`、runtime 真实产物或 API Key。

提交者：

```text
porter <2037388336@qq.com>
```

建议提交信息：

```text
feat(worker): 搭建后端P1 HN事件生产骨架
```

如果中途遇到阻塞，不要提交半成品，应输出阻塞原因、已完成内容和当前状态。

## 18. 失败和阻塞处理

后端代理遇到问题时，应先自己排查并记录证据。

必须明确区分：

- 环境问题。
- 依赖安装问题。
- 数据库连接问题。
- HN 网络问题。
- 网页抓取失败。
- 测试失败。
- 需求不清。

报告格式：

```text
阻塞点：
已尝试：
命令输出摘要：
影响范围：
需要用户决定的问题：
```

不要用“应该可以”“可能没问题”代替验证。
