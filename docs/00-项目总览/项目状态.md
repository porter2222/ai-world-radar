## 5. 核心产品结论

AI World Radar 的一句话定位是：

> AI World Radar 是一个面向中文用户的全球 AI 圈事件雷达，它从外网和中文公开信息源中发现 AI 圈热点与重要事件，并将碎片化信息整理成中文用户能快速理解、可追踪、可修正的事件档案。

P1 不以新闻链接、文章摘要或日报为核心，而是把全球 AI 圈的碎片化信号整理成可理解、可追踪、可修正的事件档案。晨报不取消，但后置到 P1.5，基于已发布事件生成。

## 6. 已确认的关键决策

- 核心用户：中文 AI 学习者和初级 AI / Agent 开发者，兼顾 AI 行业关注者。
- 核心内容单位：事件档案，首页以事件卡承载入口。
- 信息策略：多源发现 + 事件聚合 + 中文理解生成。
- 排序策略：热度主导 + 重要度兜底 + 中文用户价值修正。
- 首页形态：事件雷达 + 平衡版事件卡。
- P1 形态：匿名公开站。
- P1 包含：首页事件雷达、平衡版事件卡、事件详情页、信息来源、后台管理页、Agent 运行记录。
- P1.5 包含：基于已发布事件生成的今日简报。
- P1 不包含：登录、收藏、订阅、提醒、个性化推荐、AI 追问助手、社区评论、复杂知识图谱。
- 事件范围：宽 AI 圈事件。
- 工具策略：不是 AI 工具导航，只有构成事件的重要工具或产品发布才进入雷达。
- 数据源策略：原始源为主，竞品源为辅。
- 内容原则：结构化生成，非结构化阅读；事实为主体，解读为辅助。
- 发布模式：Agent-first 自动发布，不设人工前置审核，管理页事后校正。
- 来源策略：轻量可见，详情可追溯，不做显眼来源状态块。
- 信息源策略：P1 先接简单、稳定、可回源的 10 个来源；X、Reddit、YouTube 等复杂高热度平台进入 P1.5 增强。
- 候选事件策略：P1 不单独建 CandidateEvent 表，候选判断字段放入 EvidenceCard。
- 热度策略：P1 坚持热度优先，同时用 impact_signal_weight 和 impact_score 承接 OpenAI、Anthropic、NVIDIA 等重大官方动态的重要度兜底。
- Agent 架构策略：工程化情报流水线 + LangGraph Agent Workflow，不采用单个万能 Agent 自由上网。
- 证据层策略：P1 不单独建 RawItem 表，EvidenceCard 同时保存来源信号和 AI 理解结果。
- 聚合策略：P1 先使用规则合并 + LLM 模糊判断，P1.5 再引入 embedding。
- 发布分层策略：收集到的信息、候选事件、发布事件、简报事件分层处理；公开展示事件必须有详情页，今日简报只从发布事件中精选。
- 首页卡片策略：P1 默认采用“标题 + 20-35 字中文辅助说明 + 主分类 + 轻量来源/时间 + 1 个语义信号标签 + 可选事件图”的平衡结构；完整来源、背景、后续关注和复杂热度指标进入详情页或后台。
- 后台监控策略：后台不仅是内容管理工具，也是 Agent 生产链路监控台，需记录采集、事件归并、Agent 运行、内容生成、发布和人工操作。
- 质量门禁策略：P1 不做真实性核验，Quality Gate 只检查生成质量、来源追溯、结构完整性和搬运风险。
- Agent 工具策略：P1 复杂节点允许受控 tool-calling，但工具必须白名单、限步数、限调用次数，Agent 不能直接发布或修改核心数据库状态。
- LLMClient 策略：迁入用户已有 Python LLMClient，只负责 chat、stream 和 provider/model 切换；结构化输出、工具绑定和业务 prompt 放在具体 Agent 节点中设计。
- 功能切片策略：模块是实现单元，功能切片是交付单元；后端 P1 第一切片为 HN AI 事件生产闭环。
- 管理策略：P1 提供维护者管理页，用于修改、隐藏、恢复、补来源、换图、调优先级和单条重新生成。
- 兜底策略：P1 只做最小失败保护；工程化候选池与兜底发布机制进入 P1.5。
- 采集架构策略：长期保留轻量 Source Registry + Adapter Manifest + Snapshot / Normalizer 的演进方向，但后端 P1 第一切片先冷启动，不先建设通用采集系统。
- 稳定源策略：P1 主路径为官方 RSS/API/稳定公开页面；RSSHub、Miniflux、Huginn、changedetection.io 等作为源码策略参考。
- 高热平台策略：Reddit 官方 API、YouTube Data API、GitHub Trending、Hugging Face Papers/Trending 进入 P1.5 热度增强；X 与 Facebook 非官方采集默认高风险，不进 P1。
- 合规边界策略：不鼓励绕过登录、破解反爬、账号池规模化采集或抓取私密内容；cookie/login/proxy 类方案不得进入默认生产链路。
- 技术架构策略：P1 采用 Next.js + Python Worker + PostgreSQL；暂不引入 FastAPI。
- 数据模型策略：P1 核心表为 sources、pipeline_runs、evidence_cards、event_clusters、event_cluster_cards、content_artifacts、quality_gate_results、published_events、briefs、brief_items、admin_actions。
- 后端开发策略：后端优先，但采用纵向功能切片开发，不按技术层长期堆模块。
- 当前实现边界：先不做前端页面和 UI，后端 P1 先通过本地命令跑通 AI 情报生产链路并写入 PostgreSQL。
- 后端目录策略：项目采用 `apps\web`、`apps\worker`、`packages`、`docs` 的 monorepo 结构。
- 后端依赖策略：Python Worker 使用 `pip + venv`，数据库使用本机 PostgreSQL。
- 后端数据库策略：Python 侧使用 SQLAlchemy + Alembic 管理 PostgreSQL schema，第一版迁移建立完整 P1 核心表。
- Agent 实现策略：P1 使用 LangGraph / LangChain，但不做自由失控型 Agent；复杂节点使用受控 tool-calling。
- LLMClient 策略：迁入用户已有 Python LLMClient，只负责 chat、stream 和 provider / model 切换，结构化输出与工具绑定放在具体 Agent 节点；默认 provider 为 `deepseek`，真实 API Key 只放本地环境变量。
- 后端首个功能切片：HN AI 事件生产闭环，历史目标是从 HN Algolia API 生成可发布事件、详情内容和今日简报并入库；后续需按 PRD v2.0 将“事件档案生产与发布”设为主闭环，晨报降为下游能力。
- HN 冷启动策略：最近 7 天、最多 100 条 story、固定 query profile、`hn_heat_score = points + num_comments * 2`。
- 原文补全策略：抓取 HN story 的 `original_url` 完整可读正文，保存到 `runtime/source-pages/`，数据库只保存摘要、hash、缓存路径和抓取状态。
- Agent 输出策略：所有 Agent 输出结构化 JSON，使用 Pydantic 校验，失败后最多修复 2 次。
- 发布验收策略：最多发布 10 条 PublishedEvent，最多生成 5 条 BriefItem，Quality Gate 不通过不发布，重复运行保持幂等。
- 运行报告策略：本地命令运行后打印终端报告，并写入 `runtime/pipeline-reports/*.txt`。
- 后端代理任务书策略：后端代理只负责工程底座、数据库、HN collector、原文缓存、Agent stub、pipeline、报告和测试；真实 Agent prompt 由 Agent 代理负责。
- 后端测试策略：后端第一轮必须写 pytest 测试，真实网络测试不禁止但必须用 `network` marker 标记；最终报告必须让用户看到测了什么、用什么测、结果是什么和哪些没测。
- 后端开发检查点策略：后端代理按阶段开发，每阶段输出检查点报告，最终测试通过后一次 commit/push。
- 后端 P1 工程实现状态：第一轮工程骨架已完成并合并进 `main`，包含 HN Algolia 采集、原文缓存、Agent stubs、pipeline 入库、运行报告、Alembic migration 和 pytest 测试。
- 后端 P1 当前边界：真实 LLM Agent、真实 prompt、repair prompt、tool-calling、前端页面、FastAPI、部署、定时任务和 X / Reddit / YouTube / Facebook 采集仍未进入本轮实现。
- P1 成功标准：用户能在网站首页看到 AI 事件，点进详情后能读懂事件；系统能从外部来源自动生产事件内容并发布；管理员能在后台查看、修正、隐藏和监控整个生产链路。
