# AI World Radar 产品需求文档

版本：v0.1  
更新时间：2026-06-02  
阶段：P1 MVP 产品需求确认版

## 1. 文档目的

本文档用于定义 AI World Radar P1 MVP 的产品需求边界、核心用户故事、页面形态、内容形态、Agent 自动生产链路、管理页能力、验收标准与后续迭代路线。

本文档只描述产品层面的需求，不进入具体 UI 视觉设计、技术架构、数据模型、Agent 编排方案和部署方案。后续阶段将基于本 PRD 继续拆解信息源策略、Agent 系统设计、技术架构和实现计划。

## 2. 产品定位

AI World Radar 是一个面向中文 AI 学习型用户的全球 AI 圈事件情报雷达。

P1 MVP 的目标不是做“最全 AI 新闻聚合站”，也不是做 AI 工具导航、GitHub Trending 中文版或通用 AI 搜索助手，而是把全球 AI 圈的碎片化信息整理成中文用户能快速理解、可以追溯来源、方便继续了解的 AI 事件情报。

P1 的一句话目标是：

> 用户每天打开 AI World Radar 后，能够在 5 分钟内了解最近全球 AI 圈最值得关注的事件，并知道这些事件为什么值得看、信息来自哪里、哪些内容仍然存在不确定性。

## 3. 目标用户

P1 主要服务中文 AI 学习型用户，包括但不限于：

- AI 初学者。
- 初级 Agent / LLM 应用开发者。
- 学生。
- 产品新人。
- 独立开发者。
- AI 爱好者。
- 想了解外网 AI 圈动态但不想长期刷外网信息源的人。

这类用户的共同特点不是身份，而是需求：

- 想了解全球 AI 圈正在发生什么。
- 想拓宽视野，知道外网 AI 圈有哪些大事、争论和新东西。
- 没有能力、时间或意愿长期刷 X、YouTube、GitHub、Hacker News、英文 Newsletter 和海外媒体。
- 希望中文内容足够快、足够克制、足够可追溯，而不是被标题党或洗稿内容淹没。

P1 不优先服务专业投研人员、资深研究员、机构分析师，也不以娱乐化 AI 新闻消费为主要方向。

## 4. P1 核心用户故事

作为一个中文 AI 学习型用户，我希望每天打开 AI World Radar 后，能在 5 分钟内看到全球 AI 圈最值得关注的事件，并知道这些事件的来源、基本背景、讨论焦点和为什么值得关注，从而不用自己长期刷外网信息源，也能保持对 AI 圈的基本认知。

## 5. 辅助用户故事

### 5.1 快速浏览首页

作为用户，我希望打开首页后，可以通过封面式事件卡快速扫过最近 AI 圈的重要事件，判断哪些值得点进去看。

### 5.2 阅读今日简报

作为用户，我希望只阅读今日简报，也能基本理解当天全球 AI 圈的重点事件，而不是只能看到标题列表。

### 5.3 理解单个事件

作为用户，我希望点击某个事件后，可以看到更完整的背景、发展脉络、外网讨论焦点、信息来源和后续关注点，而不是只看到一段摘要。

### 5.4 查看信息来源

作为用户，我希望知道事件信息来自哪里，可以点击来源继续查看原始页面，从而判断内容是否可信。

### 5.5 轻量分类浏览

作为用户，我希望可以按模型与产品、开源与工具、公司与人物、研究论文、政策与安全、观点与社区等方向粗略浏览，而不是只能看一个混合列表。

### 5.6 系统自动更新

作为用户，我希望网站可以自动更新内容，不依赖人工每天手动整理。

### 5.7 维护者人工兜底

作为维护者，我希望 Agent 自动发布后，仍然可以快速修改、隐藏、恢复、补来源、换图或重新生成某条事件，避免错误内容长期展示。

## 6. P1 用户路径

### 6.1 首页雷达主路径

打开首页 -> 浏览首页事件雷达 -> 查看事件卡 -> 点击事件详情 -> 阅读事件上下文与信息来源。

### 6.2 今日简报路径

打开首页 -> 进入今日简报 -> 阅读当天重点事件 -> 对感兴趣事件点击详情。

### 6.3 轻量分类路径

打开首页 -> 点击分类入口 -> 浏览该分类下的事件 -> 点击详情。

### 6.4 维护者路径

Agent 自动发布 -> 维护者进入管理页 -> 查看事件列表 -> 修改 / 隐藏 / 恢复 / 调整优先级 / 补来源 / 换图 / 重新生成 -> 前台同步更新。

## 7. P1 产品范围

### 7.1 P1 包含

- 匿名公开首页。
- 最近 24 小时滚动事件雷达。
- 封面式首页事件卡。
- 今日简报。
- 事件详情页。
- 图文混排能力。
- 信息来源展示。
- 轻量动态分类入口。
- Agent 自动内容生产与自动发布。
- 维护者管理页。
- 最小失败保护与人工兜底。

### 7.2 P1 不包含

- 普通用户登录。
- 收藏。
- 订阅。
- 提醒。
- 评论区。
- 社区互动。
- 个性化推荐。
- AI 追问助手。
- 复杂知识图谱。
- 完整新闻搜索引擎。
- 分钟级实时热榜。
- 多角色、多编辑审核流。
- 投资建议、技术选型建议或职业机会判断。
- 复杂事实核查系统。

## 8. 核心页面需求

### 8.1 首页事件雷达

首页是 P1 的主入口。首页默认展示最近 24 小时滚动窗口内的重点 AI 事件。

首页不按“必看 / 可看 / 观察”做硬分区，而是采用混合事件流。系统内部可以保留优先级判断，但前台不把页面切成多个强分区，避免用户阅读压力过大。

首页默认展示 Top 12 事件，并支持加载更多。Top 12 不是固定凑满数量；当有效内容不足时，允许展示更少事件，不强行生成低质量内容。

首页排序采用综合排名，主要考虑：

- 热度。
- 来源质量。
- 新鲜度。
- 影响范围。
- 与中文 AI 学习型用户的相关性。

首页提供轻量分类入口。分类入口不是固定频道承诺，而是动态浏览辅助；当前窗口内有对应内容时才展示，不展示空分类。

候选分类包括：

- 模型与产品。
- 开源与工具。
- 公司与人物。
- 研究论文。
- 政策与安全。
- 观点与社区。

### 8.2 首页事件卡

首页事件卡是封面式视觉入口，不承担完整解释任务。

事件卡应帮助用户快速判断：

- 这是什么事件。
- 大概属于什么方向。
- 是否值得点进去。

事件卡建议内容：

- 封面图或预览图。
- 标题。
- 可选辅助说明。
- 分类标签。
- 热度或优先级提示。
- 弱化展示的来源或时间提示。

事件卡长度标准：

- 标题：14-28 字。
- 辅助说明：0-35 字。

事件卡不应堆叠完整来源列表、长摘要、大段背景说明或复杂元信息。

### 8.3 今日简报

今日简报是由事件卡派生出的完整阅读版日报。它不是独立内容源，但需要具备独立阅读价值。

用户只阅读今日简报，也应能理解当天全球 AI 圈的主要事件。

今日简报使用中国自然日维度，而首页雷达使用最近 24 小时滚动窗口。

今日简报不采用“今日概览 / 快速扫读 / 更多动态 / 来源说明”这类过多分块。头部只保留必要信息：

- 标题。
- 日期。
- 最后更新时间。

每条简报应包含：

- 事件标题。
- 事件背景。
- 核心变化。
- 为什么值得关注。
- 外网讨论焦点。
- 不确定性或后续关注点。
- 进入事件详情页的入口。

今日简报单事件建议长度：180-320 字。

今日简报可以图文混排。图片是可选增强项，不强制每条都有。

### 8.4 事件详情页

事件详情页不是简报的加长版，也不是深度文章，而是事件上下文档案。

它用于承载简报没有提供的背景、来源、讨论分歧、事件发酵过程和后续关注点。

事件详情页采用“结构化生成，非结构化阅读”的原则：

- Agent 可以按结构化字段组织素材。
- 前台不把字段模板、元信息块或表格式结构直接展示给用户。
- 用户看到的正文应像编辑整理过的事件上下文说明，而不是 AI 字段报告。

详情页固定包含：

- 标题。
- 事件正文。
- 信息来源。

详情页可按事件情况条件性包含：

- 外网讨论分歧。
- 事件时间线。
- 后续关注。

其中，“后续关注”只在事件仍在发展、存在不确定性或需要继续观察时出现。后续关注只写可观察信号，不写主观预测。

详情页不展示大块元信息表，例如“优先级 / 来源状态 / 分类 / 发布时间 / 更新时间”。这些信息可以内部存在，也可以在 UI 中轻量辅助显示，但不作为详情页主体。

详情页正文建议长度：500-800 字。信息来源、图片说明不计入正文长度。

### 8.5 图文混排

P1 支持简报和详情页图文混排，但图片是理解增强项，不是装饰封面。

优先使用：

- 官方发布图。
- 产品截图。
- GitHub 仓库预览。
- 论文图。
- YouTube 缩略图。
- X / 社区热帖截图。
- 公司、人物或项目的公开图。

如果没有高质量、事件相关、来源清晰的图片，可以纯文本展示，不强行生成装饰图。

不建议 P1 使用 AI 生成封面图作为主要视觉来源，因为这会降低信息可信度，也容易让产品像营销站。

### 8.6 信息来源

P1 的来源展示原则是：轻量可见，详情可追溯，不做显眼来源状态块。

首页事件卡不放完整来源列表，只保留轻量来源提示，例如来源名称、平台名或时间提示。

今日简报中，关键事实可以自然内嵌来源链接，但不应频繁打断阅读。

事件详情页底部固定展示“信息来源”，列出 2-5 个可点击来源。来源展示应使用来源名称，不要求直接展示完整 URL。

来源可以包括：

- 官方博客。
- 官方公告。
- GitHub。
- Hugging Face。
- 论文页面。
- YouTube。
- X。
- Hacker News。
- 可信媒体报道。
- 竞品站点线索。

竞品网站可以作为线索来源、热度参考源、分类参考源和冷启动补充源，但不能作为正文事实的唯一依据，也不能直接搬运或洗稿竞品正文。

对未确认或来源不足的信息，不做醒目的“来源状态”模块；必要时在正文中自然提示，例如“目前主要来自社区讨论，尚未看到官方公告”。

## 9. Agent 自动生产链路

P1 系统以 Agent 自动生产为主。

Agent 需要完成从碎片信息到中文事件情报的完整生产链路，而不是只做摘要生成。

P1 Agent 生产链路包括：

1. 从多个来源发现候选信息。
2. 合并重复信息为同一个事件。
3. 判断事件热度、时效性、来源质量和重要性。
4. 生成首页事件卡内容。
5. 生成今日简报内容。
6. 生成事件详情正文。
7. 选择或补充相关图片。
8. 整理信息来源。
9. 自动发布到前台。
10. 暴露异常、失败或低置信提示给管理页。

P1 允许 Agent 自动发布内容，不设置人工前置审核。

人工维护不是默认审核流程，而是事后校正机制。

Agent 自动发布时必须遵守：

- 不能生成不存在的来源链接。
- 不能把未确认消息写成确定事实。
- 不能凭空编造数据、截图、人物言论或官方表态。
- 不能把每件事都拔高成行业趋势。
- 内容不足时宁可少展示，也不强行凑数。

## 10. 管理页需求

P1 提供面向维护者的管理页，不面向普通用户。

管理页目标不是成为完整 CMS，而是为 Agent 自动发布提供最小纠错能力。

管理页需要有基础访问保护。

P1 管理页应支持：

- 查看今日已发布事件。
- 查看 Agent 生成失败、低置信或异常提示。
- 修改事件标题。
- 修改首页卡片辅助说明。
- 修改今日简报内容。
- 修改事件详情正文。
- 修改或替换图片。
- 修改信息来源链接。
- 隐藏事件。
- 恢复事件。
- 手动调整优先级。
- 触发单条事件重新生成。

P1 管理页不做：

- 多角色权限。
- 复杂审核流。
- 团队协作。
- 评论管理。
- 运营数据分析后台。
- 完整富文本 CMS。

## 11. 内容质量标准

P1 内容应以高信息密度、低阅读负担、谨慎表达为原则。

内容质量标准包括：

- 标题不做震惊体，不使用“炸裂”“彻底变天”“史诗级”“全网沸腾”等夸张词。
- 摘要不只复述标题，必须交代一个新增信息点。
- 今日简报必须说明“发生了什么”和“为什么值得看”。
- 详情正文必须补充背景、讨论焦点或不确定性，不能只是把简报扩写。
- 对未确认消息，不能写成确定事实。
- 对社区热议事件，可以写热度，但要说明它主要来自社区讨论。
- 不把竞品网站正文洗稿成自己的正文。
- 不生成不存在的来源链接、数据、截图或人物言论。
- 不把每件事都拔高成行业趋势。
- 中文表达要像人工编辑整理，不要像模板化 AI 总结。

## 12. 最小失败保护

P1 不展开完整工程化候选池与兜底发布机制，但必须具备最小失败保护。

最小失败保护包括：

- Agent 没有产出有效候选时，不硬生成假内容。
- 生成失败时记录失败原因。
- 管理页能看到失败或低质量提示。
- 维护者可以手动触发重新生成。
- 当天内容不足时，允许首页展示更少事件，而不是强凑 12 条。

工程化候选池、规则爬虫、热度评分、失败重试、发布前异常拦截等能力进入 P1.5。

## 13. P1 验收标准

### 13.1 用户侧验收

- 用户打开首页后，可以看到最近 24 小时内的 AI 圈重点事件。
- 首页默认展示 12 条以内的封面式事件卡。
- 用户可以点击事件卡进入详情页。
- 用户可以阅读今日简报，并在不点详情的情况下理解当天重点事件。
- 用户可以通过轻量分类入口浏览不同类型事件。
- 普通用户无需登录即可浏览。

### 13.2 内容侧验收

- 每条事件至少包含标题、分类、时间、详情正文和信息来源。
- 封面或图片为可选项，不强制每条事件都有图。
- 今日简报由事件内容派生，不是独立随笔。
- 详情页正文不是字段堆叠，而是自然语言事件上下文。
- 信息来源必须可点击。
- 对来源不足或未确认内容，不写成确定事实。

### 13.3 Agent 侧验收

- Agent 可以自动发现候选信息。
- Agent 可以合并重复信息为事件。
- Agent 可以对事件排序。
- Agent 可以生成首页卡片、今日简报、详情正文和信息来源。
- Agent 可以自动发布内容。
- Agent 失败时应保留错误信息或低置信提示，供后台查看。

### 13.4 管理侧验收

- 维护者可以进入管理页查看已发布事件。
- 维护者可以编辑标题、摘要、简报、详情、图片和来源。
- 维护者可以隐藏或恢复事件。
- 维护者可以调整事件优先级。
- 维护者可以触发单条内容重新生成。

### 13.5 边界侧验收

- P1 不包含 AI 追问助手。
- P1 不包含收藏、订阅、评论、个性化推荐。
- P1 不承诺事实核查，只承诺来源展示和谨慎表达。
- P1 不直接搬运竞品网站正文。

## 14. 风险与约束

### 14.1 内容真实性风险

Agent 可能误判事件重要性，也可能把未确认消息写得过于确定。P1 不承诺事实核查，只要求来源可追溯、未确认内容谨慎表达、明显低可信内容可在管理页隐藏或修改。

### 14.2 信息源覆盖风险

P1 不能保证覆盖全球 AI 圈所有重要事件。产品目标不是“全网最全”，而是每天筛出一批对中文 AI 学习型用户有价值的重点事件。

### 14.3 版权与转载风险

P1 不直接搬运竞品网站正文，不把竞品文章洗稿成自己的正文。竞品网站可以作为线索来源，但正文应基于多个来源重新整理，并保留信息来源链接。

图片优先使用官方公开图、页面截图、平台预览图；没有合适图片时宁可不放。

### 14.4 Agent 稳定性风险

Agent 可能出现无候选、低质量生成、重复事件、排序异常等问题。P1 只做最小失败保护，工程化候选池放到 P1.5。

### 14.5 阅读体验风险

如果卡片、简报、详情都写得太满，用户会觉得像报告站。P1 要控制层级：卡片轻、简报中等、详情补上下文，不把结构化字段直接展示给用户。

### 14.6 成本风险

如果抓取、总结、配图、详情生成全部高频运行，成本会上升。P1 以每日和滚动更新为主，不做分钟级实时热榜。

### 14.7 定位漂移风险

产品容易滑向普通 AI 新闻聚合站、工具导航站或 AI 搜索助手。P1 只围绕一个核心目标：把全球 AI 圈碎片信息整理成中文用户能快速理解的 AI 情报雷达。

## 15. 迭代路线

### 15.1 P1：AI 情报雷达 MVP

目标：先做出一个用户每天能打开看的产品。

包含：

- 首页封面式事件卡。
- 最近 24 小时 AI 事件雷达。
- 今日简报。
- 事件详情页。
- 图文混排。
- 信息来源可追溯。
- Agent 自动生成与发布。
- 管理页人工纠错。
- 轻量分类入口。

### 15.2 P1.5：工程化候选池与兜底机制

目标：让产品从“能跑”变成“更稳定”。

包含：

- 固定数据源爬虫。
- 候选信息池。
- 规则热度评分。
- 去重与聚合规则。
- Agent 失败时的候选兜底。
- 内容质量检测。
- 发布前异常拦截。
- 失败重试机制。

### 15.3 P2：事件 AI 追问助手

目标：形成差异化，不只是新闻聚合。

包含：

- 每个事件详情页内置追问助手。
- 助手上下文注入当前事件、来源、背景、相关讨论。
- 用户可以问“这件事为什么重要”“和某某有什么关系”“适合我关注吗”等问题。
- 回答必须引用已有来源或说明不确定。
- 不做全站通用聊天机器人。

### 15.4 P3：个人化与持续关注

目标：让用户从“偶尔看”变成“持续用”。

包含：

- 关注分类。
- 收藏事件。
- 每日订阅。
- 事件后续更新提醒。
- 根据用户兴趣调整简报权重。

## 16. 当前阶段结论

P1 的核心不是“生成一份 AI 日报”，而是完成一个 Agent 驱动的信息生产与发布闭环：

> 多源信息发现 -> 事件聚合 -> 中文理解生成 -> 图文内容展示 -> 来源可追溯 -> 自动发布 -> 管理页人工兜底。

P1 做到这一步后，AI World Radar 就不再只是一个爬虫或新闻列表，而是一个具备真实业务价值、工程闭环和后续差异化空间的 Agent 产品。
