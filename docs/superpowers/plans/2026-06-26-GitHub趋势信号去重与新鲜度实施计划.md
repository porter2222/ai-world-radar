# GitHub 趋势信号去重与新鲜度 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 `github_repo_trends` 成为带 7 天冷却期的趋势信号，阻止同一 GitHub repo 在短时间内反复发布首页事件，同时保留 GitHub Release、HN、官方公告等强新鲜度信号的放行能力。

**Architecture:** 新增一个工程化的 `GitHubTrendFreshnessService`，在 `DailyPipelineService` 调用 LLM selector 之前过滤纯 GitHub repo trend 重复 group；被跳过的 `SourceSignal` 写入 `status = "skipped_duplicate_trend"` 和可解释 metadata。历史重复事件由独立 cleanup 脚本治理，默认 dry-run，只在用户授权 `--apply` 时把旧重复 `PublishedEvent.status` 改为 `hidden_duplicate`。

**Tech Stack:** Python 3、SQLAlchemy ORM、pytest、现有 Worker 服务层、现有 SQLite 单测夹具、PostgreSQL 生产运行环境。

---

## 0. 当前根因确认

已确认的代码事实：

- `apps/worker/worker/services/editorial_candidate_service.py` 可以在同一轮候选构造中按 `repo_full_name` 合并信号，但只处理传入的当前候选池，不查历史 `PublishedEvent`。
- `apps/worker/worker/services/daily_pipeline_service.py` 当前流程是：采集信号 -> 读取本轮新信号 -> 构造候选 group -> 调用 selector -> 对 selected group 运行事件生产 pipeline。
- `apps/worker/worker/services/event_service.py` 的发布幂等只按 `PublishedEvent.candidate_id` 判断，无法阻止同一 repo 被 LLM 生成不同 `candidate_key` 后再次发布。
- `apps/worker/worker/models/core.py` 里 `SourceSignal.status` 和 `PublishedEvent.status` 都是普通字符串字段，本次不需要数据库 migration。
- `apps/worker/worker/services/product_query_service.py` 首页只读取 `PublishedEvent.status == "published"`，因此历史重复事件标记成 `hidden_duplicate` 后会自然从首页消失。

根因结论：

```text
github_repo_trends 是趋势检测信号，不是天然新闻事件。
当前系统缺少跨轮次、按 GitHub repo 身份的冷却期判断。
```

```python
    def evaluate_group(
```

Rules:

- Only skip if current group is pure `github_repo_trends`。
- Always allow if current group contains a hard freshness source:

```python
{
    "github_releases",
    "hn_algolia",
    "openai_news",
    "anthropic_news",
    "nvidia_news",
    "deepmind_blog",
    "google_ai_blog",
    "huggingface_blog",

    "aws_machine_learning_blog",
```

- Query history through:

```text
PublishedEvent
-> EventCandidate
-> EventCandidateSignal
-> SourceSignal
-> Source
```

- A historical event counts as a blocking pure trend event only when:
  - `PublishedEvent.status == "published"`
  - `PublishedEvent.published_at >= now - 7 days`
  - linked signal repo matches current repo
  - all linked source keys are `github_repo_trends`

```python
def _apply_github_trend_freshness_gate(
    self,
    groups: list[EditorialCandidateGroup],
    *,
    now: datetime,
) -> tuple[list[EditorialCandidateGroup], dict[str, Any]]:
    service = GitHubTrendFreshnessService(self.session)
    allowed = []
    skipped = 0
    skipped_signals = 0
    examples = []
    for group in groups:
        decision = service.evaluate_group(group, now=now)
        if decision.action == "skip":
            skipped += 1
            skipped_signals += service.mark_skipped_signals(group.signal_ids, decision, skipped_at=now)
            examples.append({...})
            continue
        allowed.append(group)
    return allowed, {...}
```

## 9. Acceptance Criteria

- 同一 repo 的纯 `github_repo_trends` group，如果 7 天内已有同 repo 纯 trend 发布事件，则不会进入 selector，不会生成新的 `PublishedEvent`。
- 当前 group 含 `github_releases`、`hn_algolia` 或官方来源时，不受纯趋势冷却期限制，可以继续进入 selector。
- 被跳过信号写入：

```text
SourceSignal.status = skipped_duplicate_trend
SourceSignal.metadata_json.github_trend_freshness.decision = skip
```

- cleanup 脚本默认 dry-run；未获用户授权不得执行 `--apply`。
- `PublishedEvent.status = hidden_duplicate` 的历史重复事件不会出现在 `GET /events` 首页列表。
- 全量 worker pytest 通过。
- 验收文档写清楚“测了什么、拿什么测、结果是什么”。
