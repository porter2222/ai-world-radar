# P1-11 采集层 8 小时时效窗口设计

更新日期：2026-06-24

## 1. 背景

P1-10 已经解决首页读取问题：`GET /events` 默认只展示最近 48 小时内的 `PublishedEvent`，低量时向最近 7 天兜底；`GET /events/{slug}` 仍可访问历史事件。

真实运行观察发现，采集层仍会把较旧的来源信号写入 `source_signals`。例如 GitHub Releases、官方 RSS/HTML 和 GitHub repo trends 都可能把数周甚至数月前的 `published_at` 写入数据库。虽然后续 `EditorialCandidateService` 还有 48 小时候选窗口，但这不能阻止原始信号表持续混入明显过期数据。

因此 P1-11 的目标是把“每 8 小时采集一次”的产品策略落到采集入库边界：本轮采集只写入本轮窗口内发生或被本轮探测到的信号。

## 2. 目标

- `source_signals` 新增写入默认只接受最近 8 小时内的普通来源事件。
- 首页仍保持 P1-10 策略：读取最近 48 小时 published events，低量时 7 天兜底。
- 详情页仍按 slug 读取长期事件档案，不受采集窗口影响。
- 采集脚本需要在 stdout summary 中说明写入数量和跳过数量，方便验收。
- 真实外部来源、真实 PostgreSQL 和测试 fixture 都应走同一套窗口规则，只允许测试注入 `now` 保证稳定。

## 3. 非目标

- 不新增数据库表或 Alembic migration。
- 不做队列调度器、cron 管理、DailyEdition、历史归档页或前端结构调整。
- 不改变 `GET /events`、`GET /events/{slug}` 和 `/admin/*` 的接口契约。
- 不恢复旧 `EvidenceCard / EventCluster / ContentArtifact / QualityGateResult / Brief / BriefItem` 主链路。
- 不引入 X、Reddit、YouTube 等需要账号池、反爬或复杂配额策略的信息源。
- 不让 Agent 直接写库或决定发布，状态流转仍由 service / workflow 控制。

## 4. 当前代码事实

本设计基于当前 worktree 的代码读取结果，不基于历史记忆：

- `apps/worker/scripts/collect_source_signals.py` 负责调用各来源 collector，并把 mapper 生成的 `SourceSignalCreate` 交给 `SignalService.upsert_signal()`。
- `apps/worker/worker/services/signal_service.py` 当前只做幂等 upsert，不做时效过滤。
- `apps/worker/worker/sources/hn_source.py` 使用 `HNStory.created_at` 作为 `SourceSignal.published_at`。
- `apps/worker/worker/sources/github_source.py` 使用 GitHub release 的 `published_at`。
- `apps/worker/worker/sources/official_news_source.py` 使用 RSS/Atom/HTML entry 的 `published_at`。
- `apps/worker/worker/sources/github_trends_source.py` 当前使用 `repo.pushed_at or repo.updated_at` 作为趋势信号的 `published_at`，这会把旧仓库更新时间误当成本轮趋势事件时间。
- `apps/worker/worker/services/editorial_candidate_service.py` 当前在 selector 前还有默认 48 小时窗口，使用 `signal.published_at or signal.collected_at` 做硬过滤；P1-11 不移除这层保护。

## 5. 设计方案

### 5.1 统一采集窗口

采集脚本新增默认参数：

```text
--lookback-hours 8
```

采集脚本在本轮开始时计算：

```text
window_end = now
window_start = now - lookback_hours
```

规则：

- 普通来源信号必须有 `published_at`。
- `published_at < window_start` 的信号跳过，不写入 `source_signals`。
- `published_at > window_end + allowed_future_skew` 的信号跳过，避免来源时间异常污染数据库。
- `published_at is None` 的普通来源信号默认跳过。
- 测试可注入固定 `now`，真实运行使用 UTC 当前时间。

`allowed_future_skew` 第一版保持很小，建议 5 分钟，用于处理来源时间和本机时间的轻微偏差。

### 5.2 各来源规则

#### HN Algolia

- 当前 `fetch_hn_stories(days=...)` 使用天级 numeric filter。
- P1-11 应支持小时级窗口，推荐新增 `hours` 参数或在脚本层把 `--lookback-hours` 转换为 Algolia `created_at_i > window_start`。
- fixture 和真实 API 的最终入库都要再经过统一窗口过滤。

#### GitHub Releases

- 以 release `published_at` 为事件发生时间。
- 只写入最近 8 小时内发布的 release。
- 缺少 `published_at` 的 release 默认跳过。

#### 官方 RSS / Atom / HTML

- 以 entry `published_at` 为事件发生时间。
- 只写入最近 8 小时内发布的 entry。
- 月份级、无法解析或缺失日期的 entry 默认跳过。
- 当前真实采集已暴露长 URL 作为 `source_item_id` 会超过 128 字符，P1-11 实现时应保留稳定短 id 修复，不让真实官方源因长 URL 中断全源采集。

#### GitHub Repo Trends

GitHub repo trends 的语义不是“仓库创建或最后 push 发生了”，而是“本轮采集探测到该仓库满足热门项目条件”。

因此：

- `SourceSignal.published_at` 应使用本轮 detected time，也就是采集脚本传入的 `now`。
- `repo.pushed_at`、`repo.updated_at`、`repo.created_at` 只保留在 `metadata` 中。
- `source_hash` 继续包含 `snapshot_bucket`，确保同一仓库不同采集轮次可以形成不同快照。
- 后续 selector 可以把 star 总量、上一轮 star、star 增量、语言、topic 等作为热度依据，但不把旧 push 时间当作趋势事件时间。

## 6. 跳过统计与输出

采集 summary 至少需要能回答：

- 本轮写入或命中的 source 数量。
- 本轮 `source_signals` 当前总数或本轮涉及数量。
- 因过旧跳过多少条。
- 因缺少发布时间跳过多少条。
- 因未来时间异常跳过多少条。
- 如果开启 `--continue-on-source-error`，哪些 source 失败、失败原因是什么。

建议 summary 字段：

```json
{
  "status": "succeeded",
  "source_keys": ["hn_algolia"],
  "signals_count": 3,
  "window": {
    "lookback_hours": 8,
    "start": "2026-06-24T00:00:00+00:00",
    "end": "2026-06-24T08:00:00+00:00"
  },
  "skipped_signals": {
    "stale": 4,
    "missing_published_at": 1,
    "future": 0
  }
}
```

`signals_count` 保持与现有脚本兼容，不在 P1-11 强制改成 only-new count；如果后续要区分“本轮新增”和“幂等命中”，单独设计，不混入本次。

## 7. 测试策略

必须测试先行，且测试基于当前代码运行结果，不基于上下文推断。

### 7.1 单元 / 脚本级测试

- GitHub release 超过 8 小时不写入。
- GitHub release 在 8 小时内写入。
- 官方 RSS entry 超过 8 小时不写入。
- 官方 RSS / HTML 缺少 `published_at` 默认不写入。
- HN story 超过 8 小时不写入，8 小时内写入。
- GitHub repo trends 写入时 `published_at` 等于本轮 detected time，而不是旧 `pushed_at`。
- summary 返回 `skipped_signals.stale`、`skipped_signals.missing_published_at` 等可读统计。

### 7.2 回归测试

至少运行：

```powershell

```

### 7.3 真实 PostgreSQL 验收

真实验收必须使用当前代码和真实命令：

- 进程内临时加载本机真实数据库连接，不把 secret 写入文档。
- 跑 `collect_source_signals.py --source-group daily_all --lookback-hours 8`。
- 查询 PostgreSQL，确认本轮新写入普通来源信号不存在 `published_at < window_start`。
- GitHub repo trends 允许 `metadata.pushed_at` 很旧，但 `source_signals.published_at` 必须是本轮 detected time。
- 如继续跑真实 LLM pipeline，必须显式记录是否使用 `--agent-mode llm`、是否禁用 fallback、run_id、published_count 和 agent_runs_count。
