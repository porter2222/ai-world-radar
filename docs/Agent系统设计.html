P1 MVP Agent Design

AI World Radar Agent 系统设计

定义 P1 MVP 的 Agent 生产链路、模块边界、发布分层与质量门禁，确保产品不是普通爬虫，也不是简单的大模型总结器。

版本：v0.1 更新时间：2026-06-03 阶段：P1 MVP

1. 文档目的

本文档用于定义 AI World Radar P1 MVP 的 Agent 系统生产链路、模块边界、Agent 职责、发布分层、质量门禁和后续演进方向。

本文档只讨论 Agent 系统设计，不进入具体数据库字段、接口设计、UI 视觉设计和代码实现计划。后续阶段将基于本文档继续拆解技术架构、数据模型、接口设计和开发计划。

2. 设计背景

AI World Radar 的目标不是做普通 AI 新闻聚合站，也不是做一个把网页丢给大模型总结的工具，而是为中文 AI 学习型用户提供全球 AI 圈事件情报雷达。

自动发现全球 AI 圈碎片化信息，将其整理成中文用户能快速理解、可追溯来源、可点击详情的 AI 事件。

因此，Agent 系统必须解决多源识别、跨源合并、内容生成和自动发布质量控制等问题。

3. 架构调研结论

在设计前，已参考 GitHub 上多类相关项目的源码策略，包括采集与抽取类、深度研究与报告类、AI 新闻与日报类项目。

  • 采集与抽取类：Firecrawl、Crawl4AI、ScrapeGraphAI、RSSHub。
  • 深度研究与报告类：GPT Researcher、STORM、Open Deep Research。
  • AI 新闻与日报类：LearnPrompt/ai-news-radar、auto-news、Horizon、hn-buddy。

核心结论是：成熟产品不应让一个大 Agent 自由浏览全网、抓取内容、写稿和发布。AI World Radar P1 应采用“工程化情报流水线 + 局部 Agent 决策”的架构。

4. 核心设计原则

工程代码负责稳定性

信息源注册、定时采集、解析、去重、存储、发布和后台操作由工程代码负责。

Agent 负责语义判断

事件判断、聚合边界、排序解释、内容生成和质量检查由 Agent 负责。

先事件，后文章

系统的核心内容单位是事件，不是文章。同一事件可由多个来源共同支撑。

先证据，后写作

写作 Agent 只能基于 Evidence Card 和 Event Cluster 写作，不能自由搜索。

只要事件进入前台公开展示，就必须有详情页；今日简报只是从公开发布事件中精选出的重点事件。

5. P1 总体流程

Source Registry
-> Source Collector
-> Raw Item Store
-> Extract / Normalize
-> Evidence Card Agent
-> Candidate Router Agent
-> Event Cluster Agent
-> Event Ranking Agent
-> Content Planning Agent
-> Content Generation Agent
-> Quality Gate Agent
-> Publisher
-> Admin Console

前四个模块主要是工程模块，中间六个模块是 Agent 语义处理模块，最后两个模块是发布与管理模块。

6. 工程模块职责

6.1 Source Registry

负责管理信息源，包括来源名称、类型、抓取方式、频率、可信度权重、启用状态和最近抓取状态。

6.2 Source Collector

负责定时采集信息。P1 优先使用 RSS/API 和稳定公开页面，不默认接入复杂反爬平台。

6.3 Raw Item Store

保存原始采集结果，用于来源追溯、内容重跑、采集排查和生成内容核查。

6.4 Extract / Normalize

把不同来源统一成标题、链接、来源、发布时间、抓取时间、语言和正文片段等标准输入。P1 优先使用确定性代码，LLM 只做兜底。

7. Agent 模块职责

7.1 Evidence Card Agent

把标准化后的 Raw Item 压缩成内部证据单元，提取来源主张、主体、触发点、候选分类、证据句和不确定性。

7.2 Candidate Router Agent

将 Evidence Card 分为事实型候选事件、热议型候选事件、待核验线索和丢弃内容。

7.3 Event Cluster Agent

负责把多个 Evidence Card 合并成同一个事件。P1 使用规则合并 + LLM 模糊判断，暂不引入 embedding。

7.4 Event Ranking Agent

根据热度、来源可信度、事件重要性、中文用户价值、时间新鲜度和多源交叉情况给事件排序。

7.5 Content Planning Agent

决定事件是否公开展示、展示在哪个区域、是否进入今日简报、是否只留后台观察或不发布。

7.6 Content Generation Agent

对公开事件生成事件卡片和详情页，对简报事件额外生成今日简报条目。该 Agent 不能自由搜索，只能基于证据和发布计划写作。

7.7 Quality Gate Agent

检查幻觉、夸大、事实错配、热议与事实混淆、搬运风险、日期错误、标题党和中文表达质量。

8. 发布与管理模块

8.1 Publisher

负责把通过质量门禁的内容发布到首页事件列表、今日简报、事件详情页、分类页和数据 JSON。

8.2 Admin Console

提供维护者人工兜底能力，包括隐藏、置顶、降权、重写、重新生成、合并、拆分、降级为待核验和恢复发布。

9. 内容发布规则

收集到的信息 ≠ 候选事件 ≠ 发布事件 ≠ 简报事件
  • 收集到的信息只是原始输入。
  • 候选事件进入系统内部判断，不一定展示。
  • 发布事件进入前台展示，必须有卡片和详情页。
  • 简报事件从发布事件中精选，额外进入今日简报。

这能避免把所有低价值信息写进简报，也能保证用户点击任何前台事件时都能看到详情页。

10. 演进路线

10.1 P1

工程代码采集稳定来源，Evidence Card 作为核心证据层，规则 + LLM 做事件聚合，不引入 embedding。

10.2 P1.5

接入 X、Reddit、YouTube 等高热平台，引入 embedding 做语义召回，增强来源健康、失败兜底和热度计算。

10.3 P2

考虑事件级 AI 追问助手、个性化排序、收藏订阅、趋势专题、知识图谱和更复杂的人机协同审核。

11. P1 验收标准

  • 系统能自动从多个来源采集 AI 圈信息。
  • 系统能将原始信息整理成 Evidence Card。
  • 系统能路由事实事件、热议事件、待核验线索和丢弃内容。
  • 系统能将多个来源聚合为同一个事件。
  • 系统能对事件进行排序和发布规划。
  • 公开展示事件都有详情页。
  • 今日简报只包含精选事件，不是全量事件合集。
  • 生成内容能通过质量门禁。
  • 维护者可以通过管理页事后校正。

12. 当前阶段结论

AI World Radar P1 的 Agent 系统不采用“万能 Agent 自由上网”的方案，而采用“工程化情报流水线 + 局部 Agent 决策”的方案。

工程代码负责采集、存储、调度和发布；Agent 负责理解、判断、合并、排序、写作和质检；所有公开事件都有详情页；今日简报只从公开事件中精选重点内容。
