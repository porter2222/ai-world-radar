Agent System Design

AI World Radar Agent 系统设计

v0.2 明确 P1 采用工程化 LangGraph Agent Workflow：工程代码控制主流程，Agent 负责语义判断、内容生产和质量门禁，复杂节点允许受控工具调用。

1. 文档目的

本文档用于定义 AI World Radar P1 的 Agent 系统设计 v0.2。它描述 Agent 在后端情报生产链路中的职责、边界、工具调用方式、LangGraph 工作流设计，以及与工程代码、LLMClient、数据库发布链路之间的关系。

核心问题：P1 的 Agent 到底是什么，哪些事情由 Agent 做，哪些事情必须由工程代码控制。

本文档不讨论 UI、具体前端页面、具体数据库字段细节、部署方案和 P2 AI 追问助手。

2. v0.2 修订背景

v0.1 文档形成于早期 Agent 架构讨论阶段。后续产品和技术设计继续推进，已经确认了新的 P1 口径。

  • P1 不单独建 RawItem 表。
  • P1 不单独建 CandidateEvent 表。
  • P1 不做真实性核验。
  • P1 使用 LangGraph / LangChain。
  • P1 迁入用户已有 Python LLMClient。
  • P1 复杂节点允许受控 tool-calling。
  • P1 后端第一功能切片是 HN AI 事件生产闭环。

3. Agent 系统的核心定位

AI World Radar 的 Agent 不是自由浏览互联网的万能 Agent，而是工程化情报流水线中的多阶段 Agent Workflow。

工程代码负责：
采集、解析、存储、调度、发布、失败控制。

Agent 负责：
理解、聚合、排序、写作、简报、质量门禁。

AI World Radar 的目标不是做普通 AI 新闻聚合站，也不是做一个把网页丢给大模型总结的工具，而是为中文 AI 学习型用户提供全球 AI 圈事件情报雷达。

4. P1 Agent 边界

P1 的 Agent 可以调用工具，但必须受控。

允许不允许
在指定节点调用白名单工具自由浏览全网
使用来源内容和数据库已有证据自由决定发布
生成结构化结果直接修改核心数据库状态
参与聚合、生成和质检无限循环调用工具
Agent 可以提出判断和生成内容，但主流程、数据库写入和发布状态由工程代码控制。

5. P1 不做什么

  • 不做自由自治 Agent。
  • 不做复杂研究 Agent。
  • 不做 AI 追问助手。
  • 不做真实性核验。
  • 不做用户个性化推荐。
  • 不做 embedding 聚合。
  • 不做 X、Reddit、YouTube 复杂高热平台采集。
  • 不做 RawItem 独立表。
  • 不做 CandidateEvent 独立表。

工程代码控制主流程

7. 工程代码负责的部分

  • 信息源配置。
  • HN / GitHub Changelog 采集。
  • URL 归一化。
  • 基础字段解析。
  • PostgreSQL 读写。
  • PipelineRun 记录。
  • 发布状态写入。
  • 失败状态记录。
  • Agent 最大步数限制。
  • 工具白名单控制。
  • 运行脚本入口。

8. Agent 负责的部分

  • 判断内容是否与 AI 圈相关。
  • 提取 EvidenceCard。
  • 判断多个 EvidenceCard 是否属于同一事件。
  • 解释事件热度和中文用户价值。
  • 生成事件卡片内容。
  • 生成事件详情内容。
  • 生成今日简报内容。
  • 检查内容是否跑题、空泛、夸大、缺来源。

Agent 的结果必须经过结构化解析、质量门禁和工程代码发布。

9. LLMClient 的职责边界

用户已有 Python LLMClient 会迁入项目，但它不是 Agent 系统本身。

负责不负责
provider / model 切换业务 prompt
api_key / base_url 配置EvidenceCard schema
chat / stream_chat工具绑定
基础 OpenAI-compatible 调用Agent 循环、质量门禁、数据库写入
LLMClient 是模型调用底座，不是业务 Agent。

10. LangGraph 工作流设计

collect_sources
-> build_evidence_cards
-> cluster_events
-> rank_events
-> generate_event_content
-> generate_brief
-> quality_gate
-> publish_results

第一版工作流不追求复杂分支，先追求状态清晰、输入输出明确、失败可记录、节点可单独测试，并且可通过本地命令手动触发。

11. Agent 节点设计

EvidenceAgent
EventClusterAgent
RankingAgent
DetailWriterAgent
BriefWriterAgent
QualityGateAgent

EvidenceAgent 从 SourceItem 生成 EvidenceCard，判断 AI 相关性并提取主体、触发点、分类、热度线索和中文用户价值。

EventClusterAgent 判断多个 EvidenceCard 是否属于同一事件，输出合并原因和事件聚合结果。

RankingAgent 根据热度、重要度、中文用户价值给事件排序。

DetailWriterAgent 负责生成事件详情正文，可调用受控工具补充上下文。

BriefWriterAgent 负责从已发布事件中生成今日简报。

QualityGateAgent 检查生成内容质量，输出发布建议。

12. 受控 Tool-Calling 设计

fetch_url_content(url)
fetch_hn_comments(item_id)
extract_page_metadata(url)
search_existing_evidence(query)
load_cluster_evidence(cluster_id)
  • 每个 Agent 最多 3-5 步。
  • 每个 Agent 最多 2-3 次工具调用。
  • 工具只能读数据，不能直接发布。
  • 工具不能绕过来源策略自由爬全网。
  • 工具调用结果必须进入 Agent 输入上下文。

13. EvidenceCard 设计口径

P1 不再把 RawItem 作为正式 P1 表。EvidenceCard 是来源信号 + AI 理解结果的统一内部证据单元。

来源信息：
original_title
original_url
source_id
published_at
raw_summary / raw_excerpt
raw_heat_metrics

AI 理解结果：
claim_summary
normalized_title
subjects
event_trigger
event_type
category
heat_clues
impact_clues
audience_value_reason
candidate_score
merge_key_hint

14. EventCluster 设计口径

EventCluster 是多个 EvidenceCard 聚合后的内部事件，不等于 PublishedEvent。

EventCluster：内部事件
PublishedEvent：公开展示事件

第一版聚合策略是规则合并 + Agent 辅助判断，不使用 embedding。

15. 内容生成 Agent 设计

内容生成不再是简单摘要，而是事件内容生产。DetailWriterAgent 输入 EventCluster、关联 EvidenceCards、RankingReason、来源链接和可选工具补充结果。

  • 不能直接搬运原文。
  • 不能把热议写成事实。
  • 不能夸大影响。
  • 中文表达要自然。
  • 用户阅读时不应看到大量结构化字段堆叠。

16. Brief 生成 Agent 设计

BriefWriterAgent 负责从已发布事件中生成今日简报内容。

  • 简报不是全量事件合集。
  • 简报只选 Top 3-5 个重点事件。
  • 简报应让用户只看它也能知道今日重点。
  • 每条 brief item 必须能跳到事件详情。

17. Quality Gate Agent 设计

P1 Quality Gate 不做真实性核验，只检查生成质量和工程发布安全。

  • 内容是否为空。
  • 结构是否可解析。
  • 标题和正文是否明显不一致。
  • 是否缺少来源链接。
  • 是否直接搬运原文。
  • 是否过度夸张。
  • event_card 和 event_detail 是否成对存在。
  • brief_item 是否关联 published_event。
publish
regenerate
manual_review
discard

18. 发布与入库边界

Agent 不能直接发布。Agent 输出的是结构化建议和内容草稿，最终由 PublishService 写入数据库。

Agent 生成内容
QualityGate 给建议
PublishService 决定写入发布表

19. 后端 P1 第一功能切片：HN AI 事件生产闭环

Hacker News API
-> SourceItem
-> EvidenceAgent
-> EventClusterAgent
-> RankingAgent
-> DetailWriterAgent
-> BriefWriterAgent
-> QualityGateAgent
-> PublishService
-> PostgreSQL
-> PipelineRun 报告

完成后应能通过本地命令触发：

python -m app.scripts.run_pipeline --source hn --limit 20
模块是实现单元，功能切片是交付单元。
